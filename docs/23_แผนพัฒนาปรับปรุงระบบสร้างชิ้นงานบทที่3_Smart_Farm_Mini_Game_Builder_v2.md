# แผนการพัฒนาปรับปรุงระบบสร้างชิ้นงานบทที่ 3  
# Smart Farm Mini Game Builder v2

**บทเรียน:** บทที่ 3 ผู้จัดการฟาร์มอัจฉริยะ  
**ระบบ:** ระบบสร้างชิ้นงานบทที่ 3 / Smart Farm Mini Game Builder  
**เป้าหมาย:** ปรับระบบสร้างชิ้นงานให้เป็นด่านเกมที่เล่นได้จริง สมเหตุสมผลทางตรรกะ สอดคล้องกับตัวเกมบทที่ 3 และเหมาะกับนักเรียน ป.4

---

## 1. ภาพรวมปัญหาที่พบ

ระบบสร้างชิ้นงานบทที่ 3 มีทิศทางที่ดีแล้ว เพราะมี Builder เฉพาะบทที่ 3 รองรับการเลือกชนิดเกม วัตถุ ตัวหลอก กฎ และการทดลองเล่นก่อนส่ง แต่ยังมีจุดที่ควรปรับปรุงเพื่อให้สมบูรณ์ขึ้น ได้แก่

1. ระบบฟอร์มเก่าของบทที่ 3 ยังอาจค้างอยู่ ทำให้ข้อมูลชิ้นงานปนกันได้  
2. หน้าเล่นผลงานของเพื่อนยังเป็นการรัน simulation มากกว่าให้ผู้เล่นสร้างกฎเอง  
3. Builder ของเกม If ยังควรแยก “วัตถุเข้าเงื่อนไข” กับ “ตัวหลอก/วัตถุปล่อยผ่าน” ให้ชัดเจน  
4. ปุ่มสร้างกฎให้อัตโนมัติอาจลดคุณค่าทางการเรียนรู้  
5. โครงสร้างข้อมูล `matchesCondition` เหมาะกับ If แต่ไม่เหมาะกับ If / Else และ If / Else If / Else  
6. ปลายทางของผลผลิตจากสัตว์บางรายการยังไม่สมเหตุสมผล  
7. ระบบ validation ยังตรวจแค่ความครบถ้วน แต่ยังไม่ตรวจคุณภาพของด่านมากพอ  
8. ตำแหน่ง animation ยังเสี่ยงคลาดเคลื่อนเมื่อจอเปลี่ยนขนาด  
9. Server-side validation ยังไม่เข้มพอ  
10. หน้าครูตรวจงานควรทดลองเล่นด่านของนักเรียนได้จริง

## 2. เป้าหมายการพัฒนา

การปรับปรุงครั้งนี้ต้องทำให้ระบบสร้างชิ้นงานบทที่ 3 เป็นระบบสร้างด่านเกมที่สมบูรณ์ขึ้น โดยมีเป้าหมายดังนี้

- เด็กสร้างด่านเกมสายพานตรรกะที่เล่นได้จริง
- ผลงานของเด็กสามารถให้เพื่อนเล่นในโหมด Challenge ได้
- ระบบ If ต้องเป็น Single-Action If อย่างถูกต้อง
- ระบบ If / Else และ If / Else If / Else ต้องแยก branch ชัดเจน
- มีตัวหลอกที่มีคุณภาพ ไม่ใช่แค่มีจำนวนครบ
- ข้อมูล `work_data` ต้องตรวจสอบและแปลงเป็น `levelConfig` ได้
- ครูสามารถทดลองเล่น ตรวจเฉลย และประเมินคุณภาพด่านได้
- ระบบต้องไม่รับข้อมูลผิดรูปแบบจาก route เก่าหรือ request ตรง

## 3. หลักการตรรกะที่ต้องล็อก

### 3.1 เกม If: Single-Action If

เกม If ต้องมี “คำสั่งพิเศษ” เพียงทางเดียว

```text
ถ้า [เข้าเงื่อนไข] → [ทำคำสั่งพิเศษ]
ถ้าไม่เข้าเงื่อนไข → ปล่อยผ่านอัตโนมัติ
```

ตัวอย่าง

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้างน้ำ]
วัตถุอื่น → ปล่อยผ่านอัตโนมัติ
```

ข้อกำหนดสำคัญ:

- ไม่มี Else slot
- ไม่มีปลายทางปกติให้เด็กเลือก
- `pass_through` เป็นพฤติกรรมของระบบ ไม่ใช่บล็อกที่เด็กต้องลาก
- ต้องมีวัตถุเข้าเงื่อนไขอย่างน้อย 1 ชิ้น
- ต้องมีวัตถุไม่เข้าเงื่อนไขอย่างน้อย 2 ชิ้น
- ต้องมีตัวหลอกอย่างน้อย 1 ชิ้น

### 3.2 เกม If / Else

```text
ถ้า [เข้าเงื่อนไข] → [ปลายทาง A]
นอกเหนือจากนี้ → [ปลายทาง B]
```

เกมนี้ต้องมี 2 branch ชัดเจน คือ If และ Else

### 3.3 เกม If / Else If / Else

```text
ถ้า [เงื่อนไข A] → [ปลายทาง A]
นอกเหนือจากนี้ถ้า [เงื่อนไข B] → [ปลายทาง B]
นอกเหนือจากนี้ → [ปลายทาง C]
```

เกมนี้ต้องแสดงการตรวจจากบนลงล่าง และมีปลายทางอย่างน้อย 3 ทาง

## 5. งานปรับปรุงที่ 2: เพิ่ม Challenge Mode ให้ผลงานของเพื่อน

### ปัญหา

หน้าเล่นผลงานของเพื่อนยังมีแนวโน้มเป็นการรันกฎที่ผู้สร้างตั้งไว้แล้ว ผู้เล่นจึงไม่ได้คิดกฎเอง

### แนวทางพัฒนา

เพิ่ม 2 โหมดในหน้าเล่นผลงาน

#### Preview Mode

ใช้สำหรับผู้สร้างและครู

- แสดงกฎของผู้สร้าง
- รันด่านตามกฎของผู้สร้าง
- ใช้ตรวจว่าด่านทำงานถูกหรือไม่

#### Challenge Mode

ใช้สำหรับเพื่อนเล่น

- ซ่อนกฎของผู้สร้างก่อนเล่น
- แสดงภารกิจ วัตถุ ปลายทาง และคลังบล็อก
- ให้ผู้เล่นลากบล็อกสร้างกฎเอง
- รันระบบจากกฎของผู้เล่น
- หลังเล่นจบจึงแสดงเฉลยของผู้สร้าง

### Flow

```text
เปิดด่านของเพื่อน
→ อ่านภารกิจ
→ ดูวัตถุและปลายทาง
→ สร้างกฎด้วย Drag & Drop
→ เริ่มสายพาน
→ ระบบประเมินจากกฎของผู้เล่น
→ สรุปผล
→ แสดงเฉลยหลังเล่นจบ
```

### Acceptance Criteria

- [ ] เพื่อนต้องสร้างกฎเองก่อนเล่น
- [ ] กฎของผู้สร้างถูกซ่อนก่อนเล่น
- [ ] หลังเล่นดูเฉลยได้
- [ ] ระบบรันจากกฎของผู้เล่น ไม่ใช่กฎของผู้สร้าง
- [ ] ครูสามารถสลับดู Preview Mode และ Challenge Mode ได้

## 6. งานปรับปรุงที่ 3: ปรับ Builder ของเกม If

### ปัญหา

ถ้าเด็กเลือกวัตถุที่เป็น pass-through เป็นวัตถุหลักของเกม If อาจทำให้ด่านไม่มีเงื่อนไขพิเศษชัดเจน

### แนวทางพัฒนา

เมื่อเลือกเกม If ให้ Builder บังคับลำดับการสร้างดังนี้

```text
เลือกเงื่อนไขพิเศษ
→ เลือกคำสั่งพิเศษ
→ เลือกวัตถุที่เข้าเงื่อนไข
→ เลือกตัวหลอก/วัตถุที่ไม่เข้าเงื่อนไข
```

UI ควรแยกวัตถุเป็น 2 กลุ่ม

#### กลุ่มที่ 1: วัตถุเป้าหมาย

เช่น

- แครอทเปื้อนโคลน
- ผักกาดมีหนอน
- มันฝรั่งมีหน่องอก

#### กลุ่มที่ 2: ตัวหลอก / วัตถุปล่อยผ่าน

เช่น

- แครอทสะอาด
- แครอทมีตำหนิ
- ผักกาดใบแหว่งแต่ไม่มีหนอน
- มันฝรั่งเปื้อนดินแต่ไม่มีหน่อ

ข้อความที่ควรแสดง:

```text
เกม If มีหน้าที่จับเฉพาะกรณีพิเศษ วัตถุอื่นจะปล่อยผ่านอัตโนมัติ
```

### Acceptance Criteria

- [ ] โหมด If ไม่มี Else
- [ ] โหมด If ต้องเลือกเงื่อนไขพิเศษก่อน
- [ ] โหมด If แยกวัตถุเข้าเงื่อนไขกับตัวหลอก
- [ ] ส่งงานไม่ได้ถ้าไม่มีวัตถุเข้าเงื่อนไข
- [ ] ส่งงานไม่ได้ถ้าไม่มีตัวหลอก

## 7. งานปรับปรุงที่ 4: ปรับปุ่มสร้างกฎอัตโนมัติ

### ปัญหา

ปุ่ม “สร้างกฎให้อัตโนมัติ” อาจทำให้เด็กไม่ฝึกคิดกฎเอง

### แนวทางพัฒนา

เปลี่ยนชื่อปุ่มเป็น

```text
ใส่ตัวอย่างเริ่มต้น
```

หรือ

```text
ดูตัวอย่างกฎ
```

และบันทึกข้อมูลการใช้ตัวช่วยลง `work_data`

```json
{
  "builder_assistance": {
    "used_auto_fill": true,
    "auto_fill_count": 1
  }
}
```

ถ้าใช้ตัวช่วย ครูควรเห็นข้อมูลนี้ในหน้าตรวจงาน

อาจเพิ่มกติกาเสริมว่า ถ้าเด็กใช้ตัวอย่างเริ่มต้น ต้องแก้ไขกฎด้วยตนเองอย่างน้อย 1 จุดก่อนส่ง

### Acceptance Criteria

- [ ] ไม่มีคำว่า “สร้างกฎให้อัตโนมัติ”
- [ ] ระบบบันทึกการใช้ตัวช่วย
- [ ] ครูเห็นว่าเด็กใช้ตัวช่วยหรือไม่
- [ ] สามารถกำหนดได้ว่าการใช้ตัวช่วยมีผลต่อการประเมินหรือไม่

## 8. งานปรับปรุงที่ 5: ปรับ schema ของวัตถุ

### ปัญหา

`matchesCondition` ใช้ได้ดีกับเกม If แต่ไม่เหมาะกับ If / Else และ If / Else If / Else เพราะ Else ก็เป็น branch ที่ถูกต้อง ไม่ใช่แค่ “ไม่เข้าเงื่อนไข”

### แนวทางพัฒนา

เพิ่ม field ใหม่ชื่อ `expectedRuleBranch`

ค่าที่เป็นไปได้

```text
if
else_if
else
pass_through
```

ตัวอย่างวัตถุในเกม If

```json
{
  "id": "scarred_carrot",
  "label": "แครอทมีตำหนิ",
  "expectedRuleBranch": "pass_through",
  "correctResult": "pass_through",
  "isDecoy": true
}
```

ตัวอย่างวัตถุในเกม If / Else

```json
{
  "id": "small_orange",
  "label": "ส้มลูกเล็ก",
  "expectedRuleBranch": "else",
  "correctResult": "processing_box"
}
```

ตัวอย่างวัตถุในเกม If / Else If / Else

```json
{
  "id": "egg_small_good",
  "label": "ไข่ใบเล็กไม่ร้าว",
  "expectedRuleBranch": "else_if",
  "correctResult": "standard_egg_tray"
}
```

### แนวทางใช้ `matchesCondition`

- ใช้เฉพาะเกม If ได้
- หรือเลิกใช้ แล้วใช้ `expectedRuleBranch` กับ `correctResult` แทนทั้งหมด

### Acceptance Criteria

- [ ] วัตถุทุกชิ้นมี `expectedRuleBranch`
- [ ] วัตถุทุกชิ้นมี `correctResult`
- [ ] ระบบทดสอบใช้ `correctResult` เป็นหลัก
- [ ] UI แสดง branch ที่วัตถุควรเข้าได้ถูกต้อง

## 9. งานปรับปรุงที่ 6: ปรับปลายทางของผลผลิตจากสัตว์

### ปัญหา

บางปลายทางของผลผลิตจากสัตว์ยังดูไม่สอดคล้อง เช่น ขนแกะไปถาดมาตรฐาน หรือนมคุณภาพดีไปห้องเย็นโดยไม่ชัดเจน

### แนวทางใหม่

แยกปลายทางตามชนิดผลผลิต

### 9.1 ไข่ไก่

| วัตถุ | ปลายทาง |
|---|---|
| ไข่ใบใหญ่ไม่ร้าว | ถาดไข่พรีเมียม |
| ไข่ใบเล็กไม่ร้าว | ถาดไข่มาตรฐาน |
| ไข่ร้าว | ถังคัดทิ้ง |

Actions:

```text
premium_egg_tray
standard_egg_tray
reject_bin
```

### 9.2 ขนแกะ

| วัตถุ | ปลายทาง |
|---|---|
| ขนแกะสะอาดและนุ่ม | ม้วนไหมพรมพรีเมียม |
| ขนแกะมีเศษหญ้า | เครื่องทำความสะอาดขน |
| ขนแกะเปียกหรือสกปรกมาก | ถังรีไซเคิล |

Actions:

```text
premium_wool_roll
wool_cleaning_machine
wool_recycle_bin
```

### 9.3 น้ำนม

| วัตถุ | ปลายทาง |
|---|---|
| นมสดเย็นคุณภาพดี | บรรจุขวดพร้อมส่ง |
| นมอุณหภูมิสูงเล็กน้อย | ห้องทำความเย็น |
| นมเสียหรืออุณหภูมิสูงมาก | ถังคัดทิ้ง |

Actions:

```text
milk_bottling
cool_room
reject_bin
```

### Acceptance Criteria

- [ ] ไม่มีขนแกะเข้าถาดไข่
- [ ] ไม่มีนมดีเข้าถังคัดทิ้งโดยไม่มีเหตุผล
- [ ] ทุก action อ่านแล้วเข้าใจว่าปลายทางคืออะไร
- [ ] ปลายทางมีไอคอนหรือภาพที่สอดคล้องกับวัตถุ

## 10. งานปรับปรุงที่ 7: เพิ่ม validation เชิงคุณภาพ

### ปัญหา

ระบบตรวจครบถ้วนแล้ว แต่ยังไม่พอที่จะบอกว่าด่าน “ดี” หรือ “ท้าทาย” หรือไม่

### Validation ที่ควรเพิ่ม

#### 10.1 Branch Balance

ตรวจว่ามีวัตถุครบในแต่ละ branch

```text
If:
- เข้าเงื่อนไข ≥ 1
- pass-through ≥ 2

If / Else:
- If ≥ 2
- Else ≥ 2

If / Else If / Else:
- If ≥ 2
- Else If ≥ 2
- Else ≥ 2
```

#### 10.2 Decoy Quality

ตรวจว่าตัวหลอกมีคุณสมบัติใกล้เคียงกับวัตถุเป้าหมาย เช่น

```text
ไข่ใบใหญ่ไม่ร้าว → ไข่ใบใหญ่แต่ร้าว
กล้วยสุก → กล้วยเหลืองอมเขียว
แครอทเปื้อนโคลน → แครอทมีตำหนิแต่ไม่เปื้อนโคลน
```

ถ้าตัวหลอกไม่ใกล้เคียง ให้เตือนว่า

```text
ตัวหลอกนี้อาจง่ายเกินไป เพราะไม่คล้ายกับวัตถุเป้าหมาย
```

#### 10.3 Destination Coverage

ตรวจว่าปลายทางทุกจุดมีวัตถุที่ควรไปถึงจริง

#### 10.4 Item Diversity

ตรวจว่าวัตถุไม่ซ้ำชนิดเดียวมากเกินไป

### qualityCheck ที่ควรบันทึก

```json
{
  "qualityCheck": {
```

## 11. งานปรับปรุงที่ 8: แก้ตำแหน่ง animation

### แนวทางพัฒนา

ให้คำนวณตำแหน่งจริงจาก element ด้วย `getBoundingClientRect()`

ตัวอย่าง

```javascript
function getBoardPoint(selector, boardEl) {
  const boardRect = boardEl.getBoundingClientRect();
  const targetRect = document.querySelector(selector).getBoundingClientRect();

  return {
    x: targetRect.left + targetRect.width / 2 - boardRect.left,
    y: targetRect.top + targetRect.height / 2 - boardRect.top
  };
}
```

ตำแหน่งที่ต้องคำนวณจริง

- จุดเริ่มต้นสายพาน

## 12. งานปรับปรุงที่ 9: เพิ่ม Server-side Validation

### ปัญหา

ถ้าตรวจเฉพาะฝั่ง client ผู้ใช้สามารถส่ง request ตรงและบันทึกงานผิดรูปแบบได้

### สิ่งที่ server ต้องตรวจ

สำหรับ `game_id = 3`

```text
project_type ต้องเป็น smart_farm_mini_game
logic_type ต้องเป็น if, if_else หรือ if_else_if_else
items ต้องครบขั้นต่ำ
rules ต้องครบตาม logic_type
testResult.tested ต้องเป็น true
title และ mission ต้องไม่ว่าง
```

สำหรับ `logic_type = if`

```text
mode ต้องเป็น single_action_if
ต้องไม่มี else rule
default_behavior ต้องเป็น pass_through
ต้องมีวัตถุเข้าเงื่อนไข
ต้องมีวัตถุ pass-through
ต้องมีตัวหลอก
```

ตัวอย่าง pseudo-code

```php
if ($game_id === 3) {
    $data = json_decode($work_data, true);

    if (($data['project_type'] ?? '') !== 'smart_farm_mini_game') {
        throw new Exception('บทที่ 3 ต้องส่งงานในรูปแบบ Smart Farm Mini Game');
    }

    if (empty($data['testResult']['tested'])) {
        throw new Exception('กรุณาทดลองเล่นด่านก่อนส่งงาน');
    }

    if (($data['logic_type'] ?? '') === 'if') {
        if (($data['mode'] ?? '') !== 'single_action_if') {
            throw new Exception('เกม If ต้องใช้โหมด Single-Action If');
        }

        foreach (($data['rules'] ?? []) as $rule) {
            if (($rule['type'] ?? '') === 'else') {
                throw new Exception('เกม If ไม่ต้องมี Else');
            }
        }
    }
}
```

### Acceptance Criteria

- [ ] Server ปฏิเสธ structured_reflection ของบทที่ 3
- [ ] Server บังคับให้ test ก่อนส่ง
- [ ] Server ตรวจ logic_type
- [ ] Server ตรวจ Single-Action If
- [ ] Error message อ่านเข้าใจง่าย

## 19. สรุป

ระบบสร้างชิ้นงานบทที่ 3 มีพื้นฐานดีแล้ว แต่ยังต้องปรับให้สมบูรณ์ใน 3 ด้านหลัก

1. **ด้านตรรกะ**  
   เกม If ต้องเป็น Single-Action If และ pass-through อัตโนมัติ ส่วน If / Else และ If / Else If / Else ต้องแยก branch ชัดเจน

2. **ด้านประสบการณ์เล่นจริง**  
   ผลงานของนักเรียนต้องกลายเป็น Challenge ที่เพื่อนเล่นได้จริง ไม่ใช่เพียง simulation ของกฎที่ผู้สร้างตั้งไว้

3. **ด้านคุณภาพและความปลอดภัยของข้อมูล**  
   ต้องมี validation ทั้งฝั่ง client และ server รวมถึง quality check สำหรับความท้าทายของด่าน

เมื่อพัฒนาตามแผนนี้ ระบบ Smart Farm Mini Game Builder จะยกระดับจากระบบส่งงานเป็นพื้นที่ให้นักเรียนออกแบบเกมตรรกะฟาร์มของตนเองได้อย่างแท้จริง
