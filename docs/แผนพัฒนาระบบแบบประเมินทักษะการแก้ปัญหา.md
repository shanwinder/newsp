# แผนพัฒนาระบบแบบประเมินทักษะการแก้ปัญหา

> โครงการ: แบบฝึกทักษะออนไลน์ เรื่อง การแก้ปัญหาอย่างเป็นขั้นตอน  
> Repository: `shanwinder/newsp`  
> สถานะเอกสาร: แผนสำหรับพัฒนาและตรวจสอบก่อนนำขึ้นใช้งานจริง  
> ขอบเขตผู้ใช้หลัก: ครูผู้สอนและผู้ดูแลระบบ  
> กลุ่มเป้าหมายของการประเมิน: นักเรียนชั้นประถมศึกษาปีที่ 4

---

## 1. ที่มาและเหตุผล

ระบบปัจจุบันมีโมดูลแบบทดสอบก่อนเรียน–หลังเรียนสำหรับให้นักเรียนตอบข้อสอบปรนัย และมีโมดูลตรวจชิ้นงานสำหรับให้ครูตรวจผลงานพร้อมบันทึกข้อเสนอแนะ อย่างไรก็ตาม แบบประเมินทักษะการแก้ปัญหาเป็นเครื่องมือสังเกตพฤติกรรมที่ครูต้องเป็นผู้ให้คะแนนตามรูบริก จึงไม่ควรนำไปปะปนกับระบบข้อสอบปรนัยเดิม

ระบบใหม่ต้องเปลี่ยนแบบประเมินบนกระดาษให้เป็นแบบฟอร์มดิจิทัล โดยยังรักษาโครงสร้างทางวิชาการของเครื่องมือไว้ครบถ้วน ได้แก่

- ประเมินทักษะ 4 ด้าน
- ด้านละ 3 รายการ รวม 12 รายการ
- ใช้มาตรประมาณค่า 4 ระดับ ตั้งแต่ 1–4 คะแนน
- ประเมินหลังจบบทเรียนจำนวน 4 ครั้ง
- ครูเป็นผู้สังเกตและตัดสินคะแนน
- ระบบคำนวณค่าเฉลี่ยรายด้าน ค่าเฉลี่ยภาพรวม และระดับทักษะ
- สามารถส่งออกข้อมูลดิบเพื่อวิเคราะห์ผลการวิจัย

ฟีเจอร์ใหม่นี้ควรใช้บริบทโรงเรียน ห้องเรียน ครูผู้สอน และรอบการเรียนเดิมของระบบ เพื่อไม่ให้เกิดข้อมูลซ้ำซ้อน และควรเชื่อมโยงกับหน้าตรวจชิ้นงานเพื่อให้ครูดูหลักฐานก่อนให้คะแนนได้สะดวก

---

## 2. วัตถุประสงค์ของการพัฒนา

1. สร้างระบบบันทึกผลการประเมินทักษะการแก้ปัญหาแบบดิจิทัลสำหรับครูผู้สอน
2. รองรับการประเมินนักเรียนรายบุคคลหลังจบบทเรียนทั้ง 4 บท
3. คำนวณและแปลผลคะแนนรายด้านและภาพรวมโดยอัตโนมัติ
4. แสดงรายงานรายบุคคล รายบทเรียน รายด้าน และภาพรวมชั้นเรียน
5. ส่งออกข้อมูลดิบและข้อมูลสรุปในรูปแบบ CSV เพื่อใช้วิเคราะห์ผลการวิจัย
6. รักษาความถูกต้องของเครื่องมือด้วยระบบเวอร์ชันและการล็อกผลที่ยืนยันแล้ว
7. เชื่อมโยงข้อมูลชิ้นงาน คะแนนเกม และข้อสังเกตของครูเพื่อใช้เป็นหลักฐานประกอบการประเมิน โดยไม่ให้ระบบตัดสินคะแนนแทนครู

## 3. ขอบเขตระบบ

### 3.1 สิ่งที่อยู่ในขอบเขต

- แบบประเมินมาตรประมาณค่า 4 ระดับ จำนวน 12 รายการ
- การเลือกห้องเรียน บทเรียน และนักเรียน
- การบันทึกแบบร่างและการยืนยันผล
- ช่องบันทึกหลักฐานหรือข้อสังเกตรายข้อ
- ช่องบันทึกข้อสังเกตภาพรวม
- การคำนวณคะแนนและแปลผล
- รายงานค่าเฉลี่ยและส่วนเบี่ยงเบนมาตรฐาน
- รายงานความครบถ้วนของการประเมิน
- การส่งออก CSV
- การเชื่อมโยงจาก Dashboard และห้องตรวจผลงาน
- การควบคุมสิทธิ์ตามโรงเรียน ห้องเรียน ครู และรอบการเรียน
- การเก็บเวอร์ชันของเครื่องมือที่ใช้ประเมิน

### 3.2 สิ่งที่ไม่อยู่ในขอบเขตระยะแรก

- การให้ AI ตัดสินคะแนนแทนครู
- การให้คะแนนอัตโนมัติจากคะแนนเกมเพียงอย่างเดียว
- การให้นักเรียนประเมินตนเอง
- การให้เพื่อนประเมินเพื่อน
- การเปรียบเทียบก่อนเรียน–หลังเรียนด้วยเครื่องมือนี้
- การแก้ไขข้อความแบบประเมินผ่านหน้าเว็บโดยครูทั่วไป
- การรวมคะแนนนี้เข้ากับคะแนนดาวหรืออันดับของนักเรียนโดยอัตโนมัติ

## 4. หลักการออกแบบระบบ

### 4.1 ครูเป็นผู้ประเมิน

คะแนนจากเกม จำนวนครั้งที่ลอง คำตอบ หรือสถานะชิ้นงานใช้เป็นหลักฐานประกอบเท่านั้น การตัดสินคะแนน 1–4 ต้องมาจากการสังเกตของครู เพราะหลายรายการวัดการอธิบายเหตุผล การวางแผน ความเป็นระบบ และการตรวจสอบข้อผิดพลาด

### 4.2 แยกจากระบบข้อสอบเดิม

ระบบข้อสอบเดิมจัดเก็บคำตอบแบบ A–D และตรวจความถูกต้อง จึงควรสร้างโมดูลใหม่โดยไม่ดัดแปลงตาราง `assessment_*` ให้รองรับรูบริก เพราะจะทำให้โครงสร้างข้อมูลและความหมายของคะแนนสับสน

### 4.3 ใช้บริบทเดิมของระบบ

ทุกระเบียนต้องผูกกับข้อมูลต่อไปนี้

- `school_id`
- `classroom_id`
- `teacher_id`
- `learning_session_id`
- `user_id`
- `lesson_id`

### 4.4 แบบประเมินเป็น Single Source of Truth

ข้อความรายการประเมิน โครงสร้างด้าน เกณฑ์แปลผล และคำอธิบายคะแนนควรเก็บในไฟล์ตั้งค่ากลาง เช่น `config/problem_solving_rubric.php` เพื่อให้ทุกหน้าและทุก API ใช้ข้อมูลชุดเดียวกัน

### 4.5 รองรับเวอร์ชันเครื่องมือ

ผลประเมินทุกชุดต้องบันทึก `rubric_version` เพื่อให้ข้อมูลเดิมยังตีความได้ถูกต้องหากมีการปรับแก้เครื่องมือภายหลัง เช่น หลังได้รับข้อเสนอแนะจากผู้เชี่ยวชาญ

### 4.6 ผลที่ยืนยันแล้วต้องถูกล็อก

เมื่อครูกดยืนยันผลเป็น `final` ระบบต้องไม่อนุญาตให้แก้ไขโดยตรง หากจำเป็นต้องแก้ไขให้ใช้กระบวนการเปิดล็อกพร้อมบันทึกผู้ดำเนินการและเวลา

## 5. ความสัมพันธ์กับโครงสร้างระบบปัจจุบัน

### 5.1 ไฟล์และโมดูลที่นำแนวทางมาใช้ร่วมกัน

- `config/lessons.php` ใช้เป็นแหล่งชื่อและหัวข้อบทเรียนทั้ง 4 บท
- `includes/context.php` ใช้ตรวจสอบบริบทโรงเรียน ห้องเรียน ครู และรอบการเรียน
- `pages/dashboard.php` ใช้เป็นจุดเข้าฟีเจอร์สำหรับครู
- `pages/review_work.php` ใช้แนวทาง UI สำหรับเลือกบทเรียน เลือกนักเรียน ดูชิ้นงาน และบันทึกข้อมูล
- `api/get_works_list.php` ใช้เป็นข้อมูลประกอบการเชื่อมโยงผลงานกับนักเรียน
- ฟังก์ชันคำนวณค่าเฉลี่ยและส่วนเบี่ยงเบนมาตรฐานในระบบแบบทดสอบสามารถนำตรรกะมาปรับใช้ใน helper ใหม่
- ระบบโหลด CSS ผ่าน `includes/app_head.php` และ `$page_styles` ต้องใช้แนวทางเดียวกัน

### 5.2 สิ่งที่ต้องแยกจากระบบเดิม

- ไม่ใช้ `assessment_questions`
- ไม่ใช้ `assessment_answers`
- ไม่ใช้ `assessment_attempts`
- ไม่คำนวณ `is_correct`
- ไม่แสดงเป็น Pre-test หรือ Post-test
- ไม่ใช้คะแนนเต็ม 20 หรือการคำนวณ E1/E2 ในโมดูลนี้

## 6. โครงสร้างแบบประเมิน

### 6.1 ด้านที่ 1 การระบุและทำความเข้าใจปัญหา

| รหัส | รายการประเมิน |
|---|---|
| `understand_goal` | บอกเป้าหมายหรือสิ่งที่ต้องการแก้ไขจากสถานการณ์ได้ถูกต้อง |
| `understand_conditions` | ระบุข้อมูล เงื่อนไข หรือกติกาที่เกี่ยวข้องกับปัญหาได้ |
| `understand_relevance` | แยกข้อมูลที่จำเป็นต่อการแก้ปัญหาออกจากข้อมูลที่ไม่เกี่ยวข้องได้ |

### 6.2 ด้านที่ 2 การวางแผนแก้ปัญหา

| รหัส | รายการประเมิน |
|---|---|
| `plan_approach` | เสนอวิธีหรือแนวทางในการแก้ปัญหาก่อนลงมือปฏิบัติได้ |
| `plan_reason` | เลือกวิธีแก้ปัญหาที่เหมาะสมและอธิบายเหตุผลในการเลือกได้ |
| `plan_sequence` | เรียงลำดับขั้นตอนการแก้ปัญหาได้อย่างถูกต้องและต่อเนื่อง |

### 6.3 ด้านที่ 3 การดำเนินการตามแผน

| รหัส | รายการประเมิน |
|---|---|
| `execute_steps` | ลงมือแก้ปัญหาตามลำดับขั้นตอนที่วางแผนไว้ได้ |
| `execute_conditions` | ใช้ข้อมูล กติกา และเงื่อนไขในการตัดสินใจระหว่างแก้ปัญหาได้อย่างเหมาะสม |
| `execute_systematic` | ดำเนินกิจกรรมอย่างเป็นระบบ โดยไม่เลือกคำตอบหรือวิธีการอย่างคาดเดา |

### 6.4 ด้านที่ 4 การตรวจสอบผลและปรับปรุงวิธีแก้ปัญหา

| รหัส | รายการประเมิน |
|---|---|
| `review_result` | ตรวจสอบว่าผลลัพธ์เป็นไปตามเป้าหมายหรือเงื่อนไขของปัญหาหรือไม่ |
| `review_error` | ระบุขั้นตอนหรือสาเหตุที่ทำให้เกิดข้อผิดพลาดได้ |
| `review_improve` | ปรับปรุงวิธีแก้ปัญหาและทดลองใหม่จนได้ผลลัพธ์ที่ถูกต้อง |

## 7. เกณฑ์การให้คะแนน

### 7.1 คะแนนระดับ 4

ผู้เรียนแสดงพฤติกรรมได้ถูกต้องครบถ้วน เป็นระบบ และทำได้ด้วยตนเอง สามารถอธิบายเหตุผล ตรวจสอบ หรือปรับปรุงวิธีได้อย่างชัดเจนตามลักษณะของด้านที่ประเมิน

### 7.2 คะแนนระดับ 3

ผู้เรียนแสดงพฤติกรรมได้ถูกต้องเป็นส่วนใหญ่ มีข้อบกพร่องเล็กน้อย หรือได้รับคำถามกระตุ้นและคำชี้แนะเพียงเล็กน้อย

### 7.3 คะแนนระดับ 2

ผู้เรียนแสดงพฤติกรรมได้เพียงบางส่วน ลำดับหรือเหตุผลยังไม่ชัดเจน และต้องได้รับคำถามกระตุ้นหรือความช่วยเหลือหลายครั้ง

### 7.4 คะแนนระดับ 1

ผู้เรียนยังไม่สามารถแสดงพฤติกรรมตามรายการประเมินได้ แม้ได้รับคำแนะนำ หรือแสดงความเข้าใจและการดำเนินการคลาดเคลื่อนอย่างชัดเจน

### 7.5 เกณฑ์แปลผลค่าเฉลี่ย

| ค่าเฉลี่ย | ระดับทักษะ |
|---:|---|
| 3.26–4.00 | ดีมาก |
| 2.51–3.25 | ดี |
| 1.76–2.50 | พอใช้ |
| 1.00–1.75 | ควรปรับปรุง |

ยังไม่กำหนดสถานะผ่านหรือไม่ผ่านจนกว่าจะมีเกณฑ์ทางการในเอกสารวิธีดำเนินการวิจัย

## 8. รอบการประเมิน

ระบบต้องรองรับการประเมินหลังจบบทเรียน 4 ครั้ง โดยใช้ข้อมูลจาก `config/lessons.php`

| ครั้งที่ | บทเรียน | หัวข้อ |
|---:|---|---|
| 1 | บทที่ 1 | การใช้เหตุผลเชิงตรรกะ |
| 2 | บทที่ 2 | การเรียงลำดับขั้นตอน |
| 3 | บทที่ 3 | การใช้เงื่อนไข IF / ELSE |
| 4 | บทที่ 4 | การตรวจสอบและแก้ไขข้อผิดพลาด |

หนึ่งผลประเมินหมายถึง นักเรียน 1 คน × บทเรียน 1 บท × รอบการเรียน 1 รอบ × เครื่องมือ 1 เวอร์ชัน

## 9. ประสบการณ์ใช้งานของครู

### 9.1 เส้นทางหลัก

1. ครูเข้าสู่ Dashboard
2. เลือกห้องเรียน
3. กดปุ่ม **ประเมินทักษะ**
4. เลือกบทเรียนที่ต้องการประเมิน
5. เลือกนักเรียนจากรายชื่อ
6. ดูผลงาน คะแนนเกม หรือข้อมูลประกอบ
7. ให้คะแนนทั้ง 12 รายการ
8. บันทึกหลักฐานหรือข้อสังเกต
9. บันทึกเป็นแบบร่าง หรือยืนยันผล
10. เลื่อนไปยังนักเรียนคนถัดไป
11. เปิดหน้ารายงานเพื่อตรวจสอบความครบถ้วนและผลรวม

### 9.2 สถานะการประเมิน

- `not_started` ยังไม่เริ่มประเมิน
- `draft` บันทึกบางส่วนหรือครบแล้วแต่ยังไม่ยืนยัน
- `final` ยืนยันผลแล้ว
- `unlocked` เคยยืนยันแล้วและถูกเปิดให้แก้ไขโดยผู้มีสิทธิ์

- ชื่อห้องเรียน

- ชื่อบทเรียน
- จำนวนประเมินแล้ว เช่น 8/16 คน

## 12. โครงสร้างฐานข้อมูล

### 12.1 ตารางหัวข้อการประเมิน

```sql
CREATE TABLE problem_solving_evaluations (
    id INT NOT NULL AUTO_INCREMENT,
    rubric_version VARCHAR(20) NOT NULL,
    user_id INT NOT NULL,
    lesson_id TINYINT UNSIGNED NOT NULL,

    school_id INT NOT NULL,
    classroom_id INT NOT NULL,
    teacher_id INT NOT NULL,
    learning_session_id INT NOT NULL,
    evaluator_id INT NOT NULL,

    status ENUM('draft', 'final') NOT NULL DEFAULT 'draft',
    overall_note TEXT DEFAULT NULL,

    evaluated_at DATETIME DEFAULT NULL,
    finalized_at DATETIME DEFAULT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
        ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (id),

    UNIQUE KEY uniq_problem_solving_evaluation (
        rubric_version,
        user_id,
        lesson_id,
        learning_session_id
    ),

    KEY idx_problem_solving_context (
        school_id,
        classroom_id,
        teacher_id,
        learning_session_id
    ),

    KEY idx_problem_solving_student (
        user_id,
        lesson_id
    ),

    KEY idx_problem_solving_evaluator (evaluator_id)
) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci;
```

### 12.2 ตารางคะแนนรายข้อ

```sql
CREATE TABLE problem_solving_evaluation_scores (
    id INT NOT NULL AUTO_INCREMENT,
    evaluation_id INT NOT NULL,
    item_key VARCHAR(60) NOT NULL,
    domain_key VARCHAR(30) NOT NULL,
    score TINYINT UNSIGNED NOT NULL,
    evidence_note TEXT DEFAULT NULL,
    created_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP
        ON UPDATE CURRENT_TIMESTAMP,

    PRIMARY KEY (id),

    UNIQUE KEY uniq_problem_solving_item (
        evaluation_id,
        item_key
    ),

    KEY idx_problem_solving_domain (
        evaluation_id,
        domain_key
    ),

    CONSTRAINT fk_problem_solving_evaluation
        FOREIGN KEY (evaluation_id)
        REFERENCES problem_solving_evaluations(id)
        ON DELETE CASCADE,

    CONSTRAINT chk_problem_solving_score
        CHECK (score BETWEEN 1 AND 4)
) ENGINE=InnoDB
  DEFAULT CHARSET=utf8mb4
  COLLATE=utf8mb4_unicode_ci;
```

## 15. การคำนวณคะแนน

### 15.1 คะแนนรายด้าน

แต่ละด้านมี 3 รายการ

```text
ค่าเฉลี่ยรายด้าน = ผลรวมคะแนนในด้าน ÷ จำนวนรายการในด้าน
```

### 15.2 คะแนนภาพรวม

```text
คะแนนรวมต่ำสุด = 12
คะแนนรวมสูงสุด = 48
ค่าเฉลี่ยภาพรวม = ผลรวมคะแนนทั้ง 12 รายการ ÷ 12
```

ระบบต้องคำนวณจากจำนวนรายการใน config ไม่ hard-code จำนวน 3 และ 12 ใน helper

### 15.3 ส่วนเบี่ยงเบนมาตรฐาน

รายงานระดับชั้นเรียนใช้ Sample Standard Deviation

```text
SD = sqrt(sum((x - mean)^2) / (n - 1))
```

ถ้ามีข้อมูลน้อยกว่า 2 คน ให้แสดง 0.00 หรือ `-` ตามข้อกำหนด UI ที่เลือกใช้ทั้งระบบให้สม่ำเสมอ

### 15.4 การปัดทศนิยม

- เก็บค่าคำนวณด้วยความละเอียดเต็มระหว่างประมวลผล
- แสดงค่าเฉลี่ยและ S.D. 2 ตำแหน่ง
- การแปลผลใช้ค่าจริงก่อนปัดทศนิยม เพื่อป้องกันค่าชายขอบคลาดเคลื่อน

## 20. ความปลอดภัยและความถูกต้องของข้อมูล

### 20.1 การตรวจสิทธิ์

ทุกหน้าและ API ต้องตรวจว่า

- ผู้ใช้เป็นครูหรือผู้ดูแลระบบ
- บัญชียังใช้งานได้
- ห้องเรียนอยู่ภายใต้สิทธิ์ของผู้ใช้
- นักเรียนอยู่ในโรงเรียน ห้องเรียน และครูตาม context
- รอบการเรียนตรงกับ context ที่ใช้งาน

### 20.2 CSRF

คำขอเขียนข้อมูลทุกจุดต้องใช้ CSRF token

### 20.3 SQL Injection

ใช้ Prepared Statement ทุก Query ที่มีข้อมูลจาก Request

### 20.4 XSS

- PHP Output ใช้ `htmlspecialchars`
- JavaScript ที่สร้าง HTML จากข้อมูลผู้ใช้ต้อง escape ข้อความ
- ข้อสังเกตและหลักฐานไม่อนุญาต HTML

### 20.5 Transaction

การบันทึกหัวข้อและคะแนนรายข้อต้องอยู่ใน transaction เดียวกัน หากรายการใดล้มเหลวต้อง rollback ทั้งชุด

### 20.6 การป้องกันข้อมูลข้ามห้อง

ห้ามใช้ `evaluation_id` เพียงอย่างเดียวในการแก้ไข ต้องตรวจ context ร่วมทุกครั้ง

### 20.7 การล็อกผล

- `final` แก้ไขไม่ได้
- เปิดล็อกต้องมีเหตุผล
- บันทึกผู้เปิดล็อกและเวลา

## 21. Autosave

ระยะแรกแนะนำให้ใช้ปุ่ม **บันทึกฉบับร่าง** ที่ชัดเจนก่อน เพื่อหลีกเลี่ยงข้อมูลซ้ำและปัญหาเครือข่าย

เมื่อระบบพื้นฐานเสถียรแล้วจึงเพิ่ม Autosave โดยมีข้อกำหนด

- Debounce 800–1,200 มิลลิวินาที
- แสดงสถานะ `กำลังบันทึก`, `บันทึกแล้ว`, `บันทึกไม่สำเร็จ`
- ห้าม Autosave เปลี่ยนสถานะเป็น `final`
- เมื่อออกจากหน้าและมีข้อมูลยังไม่บันทึก ให้แจ้งเตือน
- ป้องกันการเขียนทับด้วย `updated_at` หรือ Revision Token

## 22. การจัดการ Concurrent Edit

แม้ปกติจะมีครูคนเดียวต่อห้อง แต่ควรป้องกันการเปิดผลเดียวกันหลายแท็บ

แนวทางขั้นต่ำ

1. API ส่ง `updated_at` กลับมา
2. Request บันทึกต้องส่ง `last_known_updated_at`
3. หากข้อมูลในฐานเปลี่ยนหลังเวลานั้น ให้ตอบ `409 Conflict`
4. UI แจ้งให้โหลดข้อมูลล่าสุดก่อนบันทึกใหม่

## 23. Responsive และ Accessibility

### 23.1 Mobile/Tablet

- ปุ่มคะแนนต้องมีพื้นที่กดอย่างน้อย 44×44 px
- รายชื่อนักเรียนเปลี่ยนจาก Sidebar เป็น Offcanvas บนหน้าจอแคบ
- การ์ดด้านเรียงหนึ่งคอลัมน์
- ปุ่มบันทึกติดด้านล่างแบบ Sticky โดยไม่บังเนื้อหา

### 23.2 Keyboard

- ใช้ Tab เลื่อนไปรายการประเมินได้
- ปุ่มคะแนนใช้ Radio Group ที่เข้าถึงได้
- แสดง Focus State ชัดเจน

### 23.3 Screen Reader

- แต่ละรายการมี `fieldset` และ `legend`
- ปุ่มคะแนนมี `aria-label` ที่บอกข้อและระดับคะแนน
- ข้อความแจ้งบันทึกสำเร็จใช้ `aria-live`

### 23.4 สี

ห้ามใช้สีเพียงอย่างเดียวสื่อสถานะ ต้องมีข้อความหรือไอคอนร่วมด้วย

## 25. แผนการทดสอบ

### 25.1 Unit Test

- การคำนวณค่าเฉลี่ยรายด้าน
- การคำนวณค่าเฉลี่ยภาพรวม
- การแปลผลชายขอบ 3.26, 2.51, 1.76 และ 1.00
- การคำนวณ S.D.
- การตรวจคะแนนนอกช่วง 1–4
- การตรวจจำนวนรายการตาม config

### 25.2 Permission Test

- ครู A เปิดข้อมูลห้องของครู B ไม่ได้
- ครู A บันทึกผลนักเรียนห้องอื่นไม่ได้
- นักเรียนเรียก API ไม่ได้
- Visitor เรียก API ไม่ได้
- Admin เปิดข้อมูลได้ตามสิทธิ์ที่กำหนด

### 25.3 Data Integrity Test

- นักเรียนหนึ่งคนมีผลได้หนึ่งชุดต่อบทเรียนต่อรอบต่อเวอร์ชัน
- บันทึกซ้ำเป็นการ Update ไม่สร้างแถวซ้ำ
- Final ที่คะแนนไม่ครบต้องถูกปฏิเสธ
- Final แก้ไขไม่ได้
- เปิดล็อกต้องมี Audit Log
- ลบหัวข้อแล้วคะแนนรายข้อถูกลบตาม Foreign Key

### 25.4 UI Test

- ให้คะแนนครบ 12 ข้อ
- สลับนักเรียนโดยข้อมูลไม่ค้าง
- บันทึก Draft และกลับมาแก้ไขได้
- Final แล้ว UI เป็น Read-only
- Badge สถานะอัปเดตทันที
- ปุ่มก่อนหน้าและถัดไปทำงานถูกต้อง
- ช่องหมายเหตุรองรับภาษาไทยและข้อความยาว

### 25.5 Report Test

- ไม่นับ Draft ในผลสรุป
- Mean ตรงกับข้อมูลดิบ
- S.D. ตรงกับสูตร
- รายงานแต่ละบทกรองถูกต้อง
- CSV เปิดใน Excel แล้วภาษาไทยไม่เพี้ยน
- ข้อมูล Raw และ Summary ตรวจย้อนกลับกันได้

### 25.6 Regression Test

ตรวจว่าฟีเจอร์ใหม่ไม่กระทบ

- ระบบเข้าสู่ระบบ
- Dashboard
- ระบบข้อสอบก่อน–หลัง
- ระบบแบบสอบถาม
- ห้องตรวจผลงาน
- การเล่นเกมและบันทึกคะแนน
- การโหลด CSS และ JavaScript กลาง

## 26. Acceptance Criteria

ฟีเจอร์ถือว่าพร้อมใช้งานเมื่อผ่านเกณฑ์ต่อไปนี้ทั้งหมด

1. ครูสามารถเลือกบทเรียนและนักเรียนได้ถูกต้องตามห้องเรียน
2. ระบบแสดงรายการประเมินครบ 12 ข้อตรงตามเวอร์ชันเครื่องมือ
3. ทุกข้อเลือกคะแนนได้เฉพาะ 1–4
4. ครูบันทึก Draft ได้แม้คะแนนยังไม่ครบ
5. ระบบไม่อนุญาตให้ Final จนกว่าจะมีคะแนนครบทุกข้อ
6. ระบบคำนวณรายด้านและภาพรวมได้ถูกต้อง
7. ระบบแปลผลตามช่วงคะแนนที่กำหนด
8. ผล Final ถูกล็อกและไม่แก้ไขโดยตรง
9. รายงานไม่นับ Draft ในสถิติหลัก
10. รายงานแสดง Mean และ S.D. ถูกต้อง
11. Export CSV มีข้อมูลเพียงพอต่อการวิเคราะห์งานวิจัย
12. ผู้ใช้ไม่สามารถเข้าถึงข้อมูลข้ามห้องหรือข้ามครู
13. หน้าประเมินใช้งานได้ที่ 390×844 และ 1366×768 โดยไม่มี Horizontal Overflow
14. ไม่มี Console Error, PHP Warning หรือ Database Error ในกระบวนการหลัก
15. ระบบข้อสอบ แบบสอบถาม เกม และห้องตรวจผลงานเดิมยังทำงานตามปกติ

## 27. ความเสี่ยงและแนวทางลดความเสี่ยง

| ความเสี่ยง | ผลกระทบ | แนวทางป้องกัน |
|---|---|---|
| ครูตีความระดับคะแนนไม่ตรงกัน | คะแนนขาดความสม่ำเสมอ | แสดงคำอธิบายรูบริกชัดเจน และจัดทำคู่มือพร้อมตัวอย่างพฤติกรรม |
| ระบบให้คะแนนอัตโนมัติมากเกินไป | เครื่องมือไม่ตรงตามนิยาม | ใช้ข้อมูลเกมเป็นหลักฐานเท่านั้น ครูเป็นผู้ตัดสิน |
| แก้ข้อความเครื่องมือหลังเก็บข้อมูล | ข้อมูลต่างเวอร์ชันปะปน | บันทึก `rubric_version` ทุกผลประเมิน |
| ครูกด Final ผิดคน | ข้อมูลคลาดเคลื่อน | แสดงชื่อและรหัสนักเรียนเด่นชัด พร้อม Confirmation ก่อน Final |
| ข้อมูลหายระหว่างบันทึก | ต้องประเมินใหม่ | ใช้ transaction, Draft และสถานะบันทึกที่ชัดเจน |
| เปิดหลายแท็บแล้วเขียนทับ | คะแนนล่าสุดสูญหาย | ใช้ `updated_at` ตรวจ Conflict |
| ผลรายงานรวม Draft | ค่าเฉลี่ยผิด | ค่าเริ่มต้นรายงานใช้เฉพาะ `final` |
| ชิ้นงานกลุ่มทำให้เข้าใจว่าเป็นคะแนนกลุ่ม | ประเมินรายบุคคลผิดหลัก | บังคับเลือกสมาชิกและสร้างผลรายคน |
| Migration กระทบฐานจริง | ระบบหยุดทำงาน | ทดสอบบนฐานสำรองและทำ Backup ก่อน Deploy |

- คู่มือครูอธิบายการใช้คะแนนและหลักฐานประกอบชัดเจน

- มีการสำรองข้อมูลและแผนย้อนกลับกรณี Deploy ล้มเหลว

## 30. งานต่อยอดภายหลัง

หลังระบบหลักมีความเสถียร สามารถพัฒนาต่อได้ดังนี้

- Autosave แบบมี Conflict Detection
- แดชบอร์ด Heatmap รายข้อ
- ระบบแนบภาพหรือไฟล์หลักฐาน
- ระบบพิมพ์แบบประเมินรายบุคคลเป็น PDF
- เปรียบเทียบความสอดคล้องของผู้ประเมินหลายคน
- สร้างตัวอย่างพฤติกรรมแบบ Anchor สำหรับแต่ละระดับคะแนน
- เชื่อมข้อมูลการใช้ Hint จำนวนครั้งที่ทดลอง และลำดับการแก้ปัญหาเป็น Evidence Timeline
- ระบบแจ้งเตือนนักเรียนที่ยังประเมินไม่ครบ
- ระบบ Import คะแนนจากแบบประเมินกระดาษ

ฟีเจอร์ต่อยอดต้องไม่เปลี่ยนหลักสำคัญว่า **ผู้ประเมินเป็นผู้ตัดสินคะแนนตามรูบริก ส่วนข้อมูลจากระบบเป็นหลักฐานประกอบการตัดสินใจ**
