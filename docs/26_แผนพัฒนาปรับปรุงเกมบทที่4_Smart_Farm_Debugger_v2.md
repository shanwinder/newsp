# แผนการปรับปรุงพัฒนาเกมบทที่ 4  
## Farm Debug Quest: ภารกิจตรวจสอบและแก้ไขข้อผิดพลาดในฟาร์มอัจฉริยะ

> เอกสารฉบับนี้จัดทำจากคอนเซ็ปเกมเว็บ **Farm Debug Quest** โดยนำมาวิเคราะห์ร่วมกับสภาพจริงของระบบแบบฝึกทักษะออนไลน์ใน GitHub repository `shanwinder/newsp` โดยโฟกัสเฉพาะส่วนของ **บทที่ 4: การตรวจสอบและแก้ไขข้อผิดพลาด** เพื่อใช้เป็นแผนการปรับปรุงพัฒนาระบบต่อไป

---

## 1. หลักคิดในการปรับปรุง

การปรับปรุงบทที่ 4 จะไม่เริ่มจากการรื้อระบบเดิมทั้งหมด แต่จะใช้โครงสร้างเดิมของระบบแบบฝึกทักษะออนไลน์เป็นฐาน ได้แก่ ระบบเลือกบทเรียน ระบบเลือกด่าน ระบบบันทึกคะแนน ระบบแสดงผลงานนักเรียน และ engine เดิมของ `SmartFarmDebuggerLite` แล้วปรับให้บทที่ 4 มีความเป็น “เกม Debugging” มากขึ้นตามคอนเซ็ป **Farm Debug Quest**

แนวคิดหลักคือ เด็กต้องไม่ได้เพียงอ่านโจทย์และเลือกคำตอบ แต่ต้องได้เห็นระบบฟาร์มทำงานผิดพลาดก่อน จากนั้นจึงสังเกตอาการ หาจุดผิด เลือกวิธีซ่อม และทดสอบซ้ำ จึงจะเข้าใจว่าการตรวจสอบและแก้ไขข้อผิดพลาดเป็นกระบวนการ ไม่ใช่การเดาคำตอบ

ดังนั้น บทที่ 4 ควรยึดแกนการเรียนรู้ดังนี้

```text
ดูอาการเสีย → รันทดสอบ → เห็นผลผิด → หาจุดผิด → ซ่อม → ลองใหม่ → เห็นผลที่ถูกต้อง
```

## 2. สภาพปัจจุบันของระบบบทที่ 4

จากโครงสร้างในระบบปัจจุบัน บทที่ 4 มีองค์ประกอบสำคัญอยู่แล้ว ได้แก่

1. มี `game_id = 4` สำหรับบทเรียน “ช่างซ่อมระบบฟาร์มอัจฉริยะ”
2. มี stage 10, 11 และ 12 เป็นด่านของบทที่ 4
3. หน้า `play_game.php` ตรวจจับว่า stage 10–12 เป็น debugger stage
4. มีไฟล์ engine กลาง `smart_farm_debugger_lite.js`
5. มีไฟล์ CSS เฉพาะบทที่ 4 คือ `smart_farm_debugger_lite.css`
6. มีไฟล์เนื้อหา stage 10, 11, 12 แยกเป็น configuration
7. มีระบบบันทึกคะแนนผ่าน `window.sendResult()`
8. มีระบบสร้างชิ้นงานบทที่ 4 เบื้องต้นผ่าน `create_project_debug.php`
9. มีระบบตรวจสอบข้อมูลส่งงานบทที่ 4 ใน `api/save_work.php`
10. มีลานโชว์ผลงานที่สามารถสรุปข้อมูลโจทย์ Debug Challenge ได้บางส่วน

กล่าวคือ ระบบมี “โครงสร้าง” แล้ว แต่ยังต้องปรับ “ประสบการณ์การเล่น” ให้เป็นเกมมากขึ้น และต้องเชื่อมระบบสร้างโจทย์กับระบบเล่นโจทย์ของเพื่อนให้สมบูรณ์

## 3. ปัญหาหลักที่ควรแก้

### 3.1 เกมยังเป็นการเลือกคำตอบมากกว่าเกมจำลองสถานการณ์

แม้ flow ปัจจุบันจะมีขั้นตอนดูอาการ หาจุดผิด ซ่อม และทดสอบ แต่ภาพในเกมยังค่อนข้างนิ่ง ผู้เล่นยังไม่ได้เห็นระบบทำงานผิดแบบชัดเจน เช่น หุ่นยนต์เดินผิดทาง น้ำท่วมโรงเรือน หรือผลไม้เข้าลังผิด

ควรเพิ่มการจำลองสถานการณ์แบบสั้น ๆ ให้เด็กเห็นความผิดพลาดก่อนตอบ เช่น

- แครอทเปื้อนโคลนวิ่งผ่านเครื่องล้างไปเฉย ๆ
- ฝนตกแต่สปริงเกอร์ยังเปิดจนน้ำท่วม
- เครื่องคัดผลไม้หยุดก่อนเก็บครบจำนวน
- หุ่นยนต์เดินผิดทางหรือรดน้ำผิดแปลง

### 3.2 บทที่ 4 ยังไม่ใช้ Phaser อย่างเต็มที่

หน้าเล่นเกมโหลด Phaser อยู่แล้ว แต่ engine ของบทที่ 4 ยังใช้ DOM/HTML/CSS เป็นหลัก การทำงานนี้ดีในแง่ความง่าย แต่ยังไม่ดึงพลังของ Phaser มาใช้สร้างภาพเคลื่อนไหวและความรู้สึกแบบเกม

แนวทางที่เหมาะสมคือไม่จำเป็นต้องแปลงทั้งหมดเป็น Phaser ทันที แต่ควรใช้แนวทาง **Hybrid** คือ

- ใช้ DOM/CSS สำหรับแผงคำสั่ง ปุ่ม ตัวเลือก และ feedback
- ใช้ Phaser สำหรับฉากฟาร์ม การเคลื่อนที่ของวัตถุ แอนิเมชันผิด/ถูก และเอฟเฟกต์ผ่านด่าน

### 3.3 เนื้อหา stage 10–12 ยังควรจัดใหม่ให้สัมพันธ์กับ Farm Debug Quest

คอนเซ็ป Farm Debug Quest แบ่งเป็น 3 เกมหลัก ได้แก่

1. แปลงผักหลงทาง
2. โรงเรือนอัจฉริยะรวน
3. เครื่องคัดผลไม้เจอบั๊ก

ในระบบจริง บทที่ 4 มี stage 10–12 อยู่แล้ว จึงควร map ใหม่เป็น

- **Stage 10: แปลงผักหลงทาง**  
  เน้นบั๊กจากลำดับคำสั่ง ทิศทาง คำสั่งหาย หรือคำสั่งเกิน

- **Stage 11: โรงเรือนอัจฉริยะรวน**  
  เน้นบั๊กจากเงื่อนไข ตัวเลข ลำดับเงื่อนไข และกรณีอื่น

- **Stage 12: เครื่องคัดผลไม้เจอบั๊ก**  
  เน้นบั๊กจากการทำซ้ำ จำนวนรอบผิด การคัดแยกผิด และการตรวจระบบหลายจุด

โครงสร้างนี้ทำให้บทที่ 4 ไม่ซ้ำกับบทที่ 3 และทำให้เด็กเห็นว่า Debugging ใช้ได้กับหลายสถานการณ์ ไม่ใช่เฉพาะสายพานหรือเงื่อนไข If อย่างเดียว

### 3.4 ระบบสร้างโจทย์บทที่ 4 ยังไม่เป็นเกมสำหรับเด็ก

ฟอร์มสร้างชิ้นงานบทที่ 4 ยังเป็นแนวกรอกข้อความ เช่น ชื่อโจทย์ อาการเสีย จุดผิด วิธีซ่อม และผลการทดลองเล่น ซึ่งใช้ได้ในระดับครูหรือเอกสาร แต่ยังไม่สนุกพอสำหรับเด็ก ป.4

ควรเปลี่ยนเป็น Builder แบบเลือกภาพและตัวเลือกง่าย ๆ เช่น

- เลือกฉาก: แปลงผัก / โรงเรือน / เครื่องคัดผลไม้
- เลือกอาการเสียจากภาพ
- เลือกจุดผิด
- เลือกวิธีซ่อม
- กดลองเล่น
- กดส่งให้เพื่อนลองซ่อม

### 3.5 ลานโชว์ผลงานยังควรเพิ่มระบบเล่นโจทย์ Debug ของเพื่อน

ปัจจุบันลานโชว์ผลงานสามารถแสดงสรุปโจทย์ debug ได้ แต่ยังควรเพิ่มระบบให้เพื่อนกดเล่นโจทย์บทที่ 4 ได้จริง โดยสร้างหน้าเล่นเฉพาะ เช่น `play_debug_work.php` หรือปรับ `play_smart_farm_work.php` ให้รองรับ `game_id = 4` และ `project_type = smart_farm_debug_lite_challenge`

## 4. เป้าหมายการพัฒนาใหม่

### 4.1 เป้าหมายด้านการเรียนรู้

หลังเล่นบทที่ 4 นักเรียนควรสามารถ

1. สังเกตผลลัพธ์ที่ผิดปกติจากระบบได้
2. บอกได้ว่าข้อผิดพลาดอยู่ที่ลำดับคำสั่ง เงื่อนไข คำสั่ง จำนวนรอบ หรือคำสั่งที่หายไป
3. เลือกวิธีแก้ไขข้อผิดพลาดอย่างง่ายได้
4. ทดสอบผลหลังแก้ไขได้
5. เข้าใจว่าการแก้บั๊กต้องทดลอง ตรวจสอบ และปรับปรุงซ้ำ

### 4.2 เป้าหมายด้านเกม

บทที่ 4 ต้องมีลักษณะเป็นเกมมากขึ้น โดยมี

1. ฉากฟาร์มที่มีชีวิตชีวา
2. วัตถุเคลื่อนไหวและแสดงผลผิด/ถูก
3. ภารกิจชัดเจนในแต่ละด่าน
4. Feedback ทันที
5. ระบบดาวที่เข้าใจง่าย
6. คำใบ้ที่ช่วยให้คิดต่อ ไม่เฉลยทันที
7. ความยากเพิ่มทีละน้อย

### 4.3 เป้าหมายด้านเทคนิค

1. ใช้โครงสร้างเดิมของระบบให้มากที่สุด
2. ปรับ `smart_farm_debugger_lite.js` ให้รองรับ simulation มากขึ้น
3. ใช้ Phaser ในส่วนฉากและ animation
4. ปรับ stage config ให้เป็นมาตรฐานเดียวกัน
5. เพิ่มระบบเล่นโจทย์ debug ของเพื่อน
6. เพิ่ม validation ของงานบทที่ 4
7. เพิ่ม log เพื่อให้ครูดูได้ว่าเด็กพลาดเรื่องใด

## 5. แผนปรับปรุง Stage 10–12

## Stage 10: แปลงผักหลงทาง

### แนวคิด

ผู้เล่นช่วยหุ่นยนต์ฟาร์มบอทแก้ระบบรดน้ำและระบบเดินทางในแปลงผัก โดยเน้นบั๊กจากลำดับคำสั่งและการทำงานผิดขั้นตอน

### จุดประสงค์

เด็กเข้าใจว่า “คำสั่งเรียงผิดเพียงจุดเดียว ทำให้ผลลัพธ์เปลี่ยนได้”

### ด่านย่อย

#### ด่าน 10-1: รดน้ำผิดที่

- **อาการเสีย:** หุ่นยนต์รดน้ำก่อนเดินถึงแปลงผัก
- **จุดผิด:** ลำดับคำสั่ง
- **วิธีซ่อม:** ย้ายคำสั่ง “รดน้ำ” ไปหลังคำสั่ง “เดินถึงแปลงผัก”
- **ภาพที่ควรเห็น:** หุ่นยนต์รดน้ำลงพื้นว่าง แล้วพืชยังแห้งอยู่
- **ผลเมื่อแก้ถูก:** หุ่นยนต์เดินถึงแปลงผัก แล้วรดน้ำถูกจุด พืชโตขึ้น

#### ด่าน 10-2: เลี้ยวผิดทาง

- **อาการเสีย:** หุ่นยนต์เดินไปชนรั้วแทนที่จะไปแปลงผัก
- **จุดผิด:** คำสั่งทิศทาง
- **วิธีซ่อม:** เปลี่ยน “เลี้ยวซ้าย” เป็น “เลี้ยวขวา” หรือกลับกัน
- **ภาพที่ควรเห็น:** หุ่นยนต์ชนรั้ว มีเอฟเฟกต์สั่นเบา ๆ
- **ผลเมื่อแก้ถูก:** หุ่นยนต์เดินผ่านทางถูกต้อง

#### ด่าน 10-3: รดน้ำไม่ครบ

- **อาการเสีย:** มีแปลงผัก 3 แปลง แต่หุ่นยนต์รดน้ำแค่ 2 แปลง
- **จุดผิด:** คำสั่งหาย หรือจำนวนรอบไม่ครบ
- **วิธีซ่อม:** เพิ่มคำสั่งรดน้ำอีก 1 ครั้ง หรือเปลี่ยนจำนวนรอบเป็น 3
- **ภาพที่ควรเห็น:** แปลงสุดท้ายยังเหี่ยว
- **ผลเมื่อแก้ถูก:** ทั้ง 3 แปลงเขียวครบ

### สิ่งที่ต้องแก้ในระบบ

- เปลี่ยน config ของ `stage10.js` ให้เป็นภารกิจหุ่นยนต์/แปลงผัก
- เพิ่มชนิดบั๊ก `order`, `direction`, `missing_step`
- เพิ่ม animation หุ่นยนต์เดินบน grid
- เพิ่ม object เช่น robot, crop1, crop2, crop3, fence, waterDrop
- เพิ่มปุ่ม “ดูอาการซ้ำ” เพื่อ replay การเดินผิด

## Stage 11: โรงเรือนอัจฉริยะรวน

### แนวคิด

ผู้เล่นซ่อมระบบโรงเรือนที่ใช้เซนเซอร์ เช่น ฝนตก ดินแห้ง อุณหภูมิสูง และอากาศปกติ โดยเน้นบั๊กจากเงื่อนไข ลำดับเงื่อนไข ตัวเลข และกรณีอื่น

### จุดประสงค์

เด็กเข้าใจว่าเงื่อนไขต้องตรงกับสถานการณ์ และบางครั้งต้องตรวจลำดับหรือกรณีที่ระบบยังไม่ได้กำหนด

### ด่านย่อย

#### ด่าน 11-1: ฝนตกแต่สปริงเกอร์ยังเปิด

- **อาการเสีย:** ฝนตกแล้ว แต่สปริงเกอร์ยังรดน้ำ ทำให้ผักน้ำท่วม
- **จุดผิด:** ลำดับเงื่อนไข
- **วิธีซ่อม:** ให้ระบบตรวจ “ฝนตก” ก่อน “ดินแห้ง”
- **ภาพที่ควรเห็น:** ฝนตกพร้อมสปริงเกอร์เปิด น้ำท่วมแปลง
- **ผลเมื่อแก้ถูก:** ฝนตกแล้วสปริงเกอร์ปิด

#### ด่าน 11-2: โรงเรือนร้อนแต่พัดลมไม่เปิด

- **อาการเสีย:** อุณหภูมิ 38°C แต่พัดลมไม่เปิด
- **จุดผิด:** ตัวเลขในเงื่อนไขสูงเกินไป
- **วิธีซ่อม:** เปลี่ยนจาก “สูงกว่า 50°C” เป็น “สูงกว่า 35°C”
- **ภาพที่ควรเห็น:** ต้นไม้เหี่ยว เหงื่อขึ้น พัดลมนิ่ง
- **ผลเมื่อแก้ถูก:** พัดลมหมุน ต้นไม้กลับมาสดชื่น

#### ด่าน 11-3: อากาศปกติแต่ระบบค้าง

- **อาการเสีย:** อุณหภูมิ 25°C แต่ระบบไม่รู้จะทำอะไร
- **จุดผิด:** ขาดคำสั่งกรณีอื่น
- **วิธีซ่อม:** เพิ่ม “นอกเหนือจากนี้ → ปิดระบบ”
- **ภาพที่ควรเห็น:** แผงควบคุมขึ้นเครื่องหมายคำถาม
- **ผลเมื่อแก้ถูก:** ระบบปิดอุปกรณ์และแสดงไฟเขียว

### สิ่งที่ต้องแก้ในระบบ

- ใช้เนื้อหาของ `stage11.js` เดิมต่อได้ แต่เพิ่ม animation ให้ชัดขึ้น
- แก้ engine ให้แสดง correct block แบบหลายบรรทัดได้ เพราะบางด่านเป็น if / else if / else
- ใช้ `questionText` จาก config จริง ไม่ใช้คำถามกลางอย่างเดียว
- เพิ่มภาพโรงเรือน พัดลม สปริงเกอร์ ฝน เทอร์โมมิเตอร์ และแผงควบคุม

## Stage 12: เครื่องคัดผลไม้เจอบั๊ก

### แนวคิด

ผู้เล่นซ่อมเครื่องคัดผลไม้ก่อนส่งตลาด โดยเน้นบั๊กจากการทำซ้ำ จำนวนรอบผิด คัดผิดประเภท และตรวจระบบหลายจุด

### จุดประสงค์

เด็กเข้าใจว่าการแก้บั๊กในระบบใหญ่ต้องตรวจทีละจุด แก้ทีละส่วน และทดสอบซ้ำ

### ด่านย่อย

#### ด่าน 12-1: เก็บมะม่วงไม่ครบ

- **อาการเสีย:** ใบสั่งต้องการมะม่วง 3 ลูก แต่เครื่องหยิบแค่ 2 ลูก
- **จุดผิด:** จำนวนรอบทำซ้ำ
- **วิธีซ่อม:** เปลี่ยน “ทำซ้ำ 2 ครั้ง” เป็น “ทำซ้ำ 3 ครั้ง”
- **ภาพที่ควรเห็น:** ลังมีมะม่วงไม่ครบ
- **ผลเมื่อแก้ถูก:** ลังมีมะม่วงครบ 3 ลูก

#### ด่าน 12-2: คัดผลไม้ผิดสี

- **อาการเสีย:** เครื่องส่งมะเขือเทศสีเขียวเข้าลังเดียวกับสีแดง
- **จุดผิด:** เงื่อนไขกว้างเกินไป
- **วิธีซ่อม:** เปลี่ยนเงื่อนไขเป็น “ถ้าสีแดง → ใส่ลัง”
- **ภาพที่ควรเห็น:** ผลไม้ผิดสีตกลงลัง
- **ผลเมื่อแก้ถูก:** สีแดงเข้าลัง สีเขียวถูกปล่อยผ่าน

#### ด่าน 12-3: ส่งผลผลิตก่อนระบบพร้อม

- **อาการเสีย:** ฟาร์มมีบั๊ก 3 จุด ได้แก่ จำนวนผลไม้ไม่ครบ คัดผิดสี และเครื่องส่งของยังไม่เปิด
- **จุดผิด:** หลายจุด
- **วิธีซ่อม:** แตะจุดเสียทีละจุด แล้วแก้ให้ครบ
- **ภาพที่ควรเห็น:** แผนที่ฟาร์มมีไฟแดงหลายจุด
- **ผลเมื่อแก้ถูก:** ทุกจุดเป็นไฟเขียว รถส่งผลผลิตออกจากฟาร์ม

### สิ่งที่ต้องแก้ในระบบ

- ปรับ `stage12.js` จากศูนย์ซ่อมฟาร์มฉุกเฉินให้เป็นศูนย์คัดผลไม้และส่งผลผลิต
- เพิ่มชนิดบั๊ก `repeat_count`, `broad_condition`, `multi_bug`
- เพิ่มระบบ map point ที่มีจุดเสียหลายจุด แต่ต้องนับคะแนนจากการแตะจุดหลอกด้วย
- เพิ่มวัตถุผลไม้ ลัง สายพาน รถส่งของ และไฟสถานะ
- เพิ่มระบบ “ซ่อมครบทุกจุดจึงจบเกม”

## 6. แผนปรับปรุง Engine

### 6.1 ปรับ state machine

จากเดิมที่มี `observe`, `identify`, `fix`, `test` ควรเพิ่มความหมายให้ชัดขึ้นเป็น

1. `observe` = ดูภารกิจ
2. `runBroken` = ดูระบบทำงานผิด
3. `identify` = เลือกจุดผิด
4. `fix` = เลือกวิธีซ่อม
5. `runFixed` = ทดสอบหลังซ่อม
6. `summary` = สรุปผล

แต่เพื่อไม่ให้แก้โค้ดมากเกินไป สามารถเก็บชื่อ phase เดิมไว้ แล้วเพิ่ม function ภายใน เช่น

- `playBrokenSimulation()`
- `playFixedSimulation()`
- `resetSimulation()`
- `renderSceneByPhase()`

### 6.2 เพิ่มระบบ simulation ต่อด่าน

ใน config แต่ละด่านควรมีข้อมูลเพิ่ม เช่น

```javascript
simulation: {
  type: 'robot_path',
  broken: {
    steps: ['move', 'water_wrong_place']
  },
  fixed: {
    steps: ['move', 'arrive_crop', 'water_crop']
  }
}
```

ตัวอย่างชนิด simulation ที่ควรรองรับ

- `robot_path` สำหรับหุ่นยนต์เดินผิดทาง
- `greenhouse_sensor` สำหรับโรงเรือนรวน
- `conveyor_sorting` สำหรับเครื่องคัดผลไม้
- `farm_map_repair` สำหรับซ่อมหลายจุด

### 6.3 แก้การแสดง correct block

ปัจจุบันบางด่านมี `correctBlock.additionalBlocks` หรือ `missingBlock` ซึ่งไม่ใช่ rule row เดี่ยว จึงควรเพิ่ม function เช่น

- `renderRuleList(blocks)`
- `renderCorrectBlock(lv)`
- `renderBuggyBlockSet(lv)`

เพื่อให้ด่านที่มีหลายเงื่อนไขแสดงผลถูกต้อง เช่น

```text
ถ้าฝนตก → ปิดสปริงเกอร์
แต่ถ้าดินแห้ง → เปิดสปริงเกอร์
```

### 6.4 ใช้ asset จริงแทน emoji

ปัจจุบัน config มี `asset.path` และ `fallbackIcon` แล้ว แต่ engine ควรเลือกใช้ภาพจริงก่อน หากโหลดไม่ได้จึง fallback เป็น emoji

แนวทางที่ควรทำ

1. ถ้ามี `obj.asset.path` ให้ render `<img>`
2. ถ้าโหลดภาพไม่สำเร็จ ให้แสดง `fallbackIcon`
3. ห้ามให้ logic ผูกกับ emoji
4. กำหนดขนาดภาพจาก `asset.width` และ `asset.height`
5. รองรับ sprite sheet ในอนาคต

### 6.5 เพิ่ม Phaser layer

ควรเพิ่ม container เฉพาะสำหรับ Phaser เช่น

```html
<div class="debug-farm-area">
  <div id="debug-phaser-scene"></div>
</div>
```

จากนั้นให้ Phaser จัดการ

- การเดินของหุ่นยนต์
- น้ำไหล / สปริงเกอร์
- พัดลมหมุน
- ผลไม้เคลื่อนบนสายพาน
- เอฟเฟกต์ผิด เช่น สั่น แดง น้ำท่วม
- เอฟเฟกต์ถูก เช่น ไฟเขียว ดาว ประกาย

DOM ยังใช้กับส่วนคำสั่งและปุ่มได้เหมือนเดิม เพื่อให้พัฒนาง่ายและอ่านง่าย

## 8. แผนปรับปรุงระบบสร้างชิ้นงานบทที่ 4

### 8.1 เป้าหมาย

ให้นักเรียนสร้างโจทย์บั๊กของตนเองได้โดยไม่ต้องเขียนกฎซับซ้อน

### 8.2 Flow ใหม่

```text
เลือกฉาก
→ เลือกอาการเสีย
→ เลือกจุดผิด
→ เลือกวิธีซ่อม
→ ทดลองเล่น
→ ส่งให้เพื่อนลองซ่อม
```

### 8.3 ฟอร์มที่ควรมี

#### ช่องที่ 1: เลือกฉาก

- แปลงผัก
- โรงเรือน
- เครื่องคัดผลไม้

#### ช่องที่ 2: เลือกอาการเสีย

ตัวอย่าง

- แครอทเลอะไม่ถูกล้าง
- ฝนตกแต่สปริงเกอร์ยังเปิด
- ผลไม้เข้าลังผิด
- เครื่องหยิบผลไม้ไม่ครบ

#### ช่องที่ 3: เลือกจุดผิด

- ลำดับคำสั่ง
- เงื่อนไข
- คำสั่ง
- ตัวเลข
- จำนวนรอบ
- ขาดคำสั่ง

#### ช่องที่ 4: เลือกวิธีซ่อม

ให้เป็นตัวเลือกตามอาการเสียที่เลือก ไม่ควรให้เด็กพิมพ์เองทั้งหมด

#### ช่องที่ 5: ทดลองเล่น

ต้องให้เด็กกดทดลองก่อนส่งจริง เพื่อยืนยันว่าโจทย์เล่นได้

### 8.4 work_data ที่ควรเก็บ

  "game_id": 4,
  "builder_version": "2.0",

  "theme": "vegetable_repair",

## 10. แผนปรับปรุงฐานข้อมูลและชื่อด่าน

### 10.1 ปรับชื่อ stage ในฐานข้อมูล

ควรปรับให้ชื่อในฐานข้อมูลตรงกับเกมจริง

```text
Stage 10: แปลงผักหลงทาง
Stage 11: โรงเรือนอัจฉริยะรวน
Stage 12: เครื่องคัดผลไม้เจอบั๊ก
```

### 10.2 ปรับคำอธิบาย stage

#### Stage 10

ฝึกตรวจหาข้อผิดพลาดจากลำดับคำสั่ง คำสั่งผิด และคำสั่งที่หายไป ผ่านภารกิจหุ่นยนต์รดน้ำในแปลงผัก

#### Stage 11

ฝึกตรวจหาข้อผิดพลาดจากเงื่อนไข ลำดับเงื่อนไข ตัวเลข และกรณีอื่น ผ่านระบบโรงเรือนอัจฉริยะ

#### Stage 12

ฝึกตรวจหาข้อผิดพลาดจากการทำซ้ำ การคัดแยกผิด และการซ่อมหลายจุด ผ่านเครื่องคัดผลไม้ก่อนส่งตลาด

### 10.3 เพิ่มข้อมูล log

ควรบันทึกข้อมูลเพิ่มเติมใน `game_logs` หรือสร้างรายละเอียดใน `detail` เช่น

```json
{
  "game_id": 4,
  "stage_id": 10,
  "level_id": "10-1",
  "bug_type": "order",
  "phase": "fix",
  "selected": "water_first",
  "expected": "water_after_arrive",
  "result": "wrong"
}
```

ข้อมูลนี้จะช่วยให้ครูรู้ว่าเด็กพลาดเรื่องใดมากที่สุด

## 12. Acceptance Criteria

### ด้านการเรียนรู้

- นักเรียนสามารถอธิบายได้ว่าอาการเสียเกิดจากจุดผิดใด
- นักเรียนสามารถเลือกวิธีซ่อมได้ถูกต้อง
- นักเรียนเข้าใจว่าต้องทดสอบหลังแก้ไข
- นักเรียนเรียนรู้จากการลองผิดลองถูกโดยไม่รู้สึกว่าถูกลงโทษ

### ด้านเกม

- แต่ละด่านมีภาพแสดงอาการเสียชัดเจน
- มี animation ก่อนและหลังแก้ไข
- มี feedback ทันที
- มีคำใบ้ 3 ระดับ
- มีดาว 0–3 ดาว
- เล่นจบได้ทุกด่าน

### ด้าน UI

- ปุ่มใหญ่ แตะง่าย
- ภาษาไทยง่าย
- ไม่มีศัพท์เทคนิคยากบนหน้าจอเด็ก
- วัตถุไม่ซ้อนกัน
- เล่นได้ทั้งคอมพิวเตอร์และมือถือ

### ด้านเทคนิค

- `stage10.js`, `stage11.js`, `stage12.js` โหลดได้ถูกต้อง
- `smart_farm_debugger_lite.js` รองรับ config ใหม่
- ส่งคะแนนผ่าน `window.sendResult()` ได้
- Builder บันทึก `work_data` ได้ถูกต้อง
- ลานโชว์ผลงานเปิดเล่นโจทย์ debug ได้
- ไม่มี error ใน console ระหว่างเล่น

## 15. สรุป

แนวทางที่เหมาะสมที่สุดคือใช้โครงสร้างเดิมของระบบบทที่ 4 เป็นฐาน แล้วปรับให้สอดคล้องกับคอนเซ็ป Farm Debug Quest โดยเน้นให้เด็กเห็นอาการเสียจริงก่อนซ่อม ใช้ภาพและ animation ช่วยอธิบาย ใช้ภาษาไทยง่าย ลดศัพท์เทคนิค และเชื่อมระบบสร้างโจทย์กับการเล่นโจทย์ของเพื่อนให้สมบูรณ์

เมื่อปรับครบ บทที่ 4 จะไม่ใช่เพียงแบบฝึกเลือกคำตอบ แต่จะกลายเป็นเกม Debugging ที่มีวงจรการเรียนรู้ครบถ้วน ได้แก่ สังเกต ทดลอง หาจุดผิด ซ่อม ทดสอบซ้ำ และสะท้อนผล ซึ่งตรงกับเป้าหมายของรายวิชาวิทยาการคำนวณ ป.4 เรื่องการตรวจสอบและแก้ไขข้อผิดพลาดอย่างเหมาะสม
