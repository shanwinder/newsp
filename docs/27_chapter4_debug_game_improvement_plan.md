# แผนการพัฒนาปรับปรุงเกมบทที่ 4  
## เรื่อง การตรวจสอบและแก้ไขข้อผิดพลาด  
### แนวทางใหม่: เกมตารางเงื่อนไขธีมเกษตร สำหรับวิทยาการคำนวณ ป.4

เอกสารนี้จัดทำเพื่อใช้เป็นแผนพัฒนาปรับปรุงเกมบทที่ 4 ของระบบแบบฝึกทักษะออนไลน์ โดยปรับจากแนวคิดเดิมที่เป็นกรอบเกมฉากใหญ่ ไปสู่รูปแบบ **เกมตารางเงื่อนไข** ที่เหมาะกับเนื้อหาเรื่อง `IF`, `IF / ELSE` และ `IF / ELSE IF / ELSE` มากกว่า

เป้าหมายคือให้เด็กเห็นความสัมพันธ์ระหว่าง **สถานการณ์ในฟาร์ม → เงื่อนไขที่ผิด → การแก้ไข → ผลลัพธ์หลังทดสอบ** อย่างชัดเจน ไม่เกิดปัญหาวัตถุหรือบล็อกคำสั่งทับซ้อนกัน และยังคงความเป็นเกมผ่านภาพ สี สถานะ ดาว คำใบ้ และ animation เบา ๆ

# 1. สรุปปัญหาที่พบจากการพัฒนารอบล่าสุด

## 1.1 รูปแบบกรอบเกมเดิมยังไม่เหมาะกับเนื้อหา Debugging

การใช้กรอบเกมแบบฉากใหญ่ทำให้ผู้เรียนต้องมองหลายอย่างพร้อมกัน ทั้งวัตถุในฉาก ภารกิจ บล็อกคำสั่ง ตัวเลือก และ feedback ส่งผลให้เนื้อหาหลักเรื่องการตรวจสอบและแก้ไขข้อผิดพลาดไม่ชัดเจนพอ

เนื้อหาบทที่ 4 ไม่ได้เน้นการเคลื่อนที่หรือการผจญภัยเป็นหลัก แต่เน้นให้ผู้เรียนเข้าใจโครงสร้างเงื่อนไข เช่น

```text
ถ้า เงื่อนไขเป็นจริง → ทำคำสั่ง
มิฉะนั้น → ทำคำสั่งอีกแบบ
```

ดังนั้นการออกแบบเป็น “ตารางเงื่อนไข” หรือ “ตารางฟาร์ม” จะเหมาะสมกว่า เพราะเด็กสามารถเห็นสถานการณ์ เงื่อนไข และผลลัพธ์ได้พร้อมกัน

## 1.2 วัตถุในฉากและบล็อกที่มีปัญหาเกิดการทับซ้อนกัน

วัตถุบางอย่างในกรอบเกม เช่น icon, object, label, block หรือข้อความ feedback มีโอกาสทับซ้อนกัน โดยเฉพาะเมื่อหน้าจอมีขนาดเล็ก หรือเมื่อข้อความภาษาไทยยาวกว่าพื้นที่ที่ออกแบบไว้

ปัญหานี้ทำให้เด็กอ่านโจทย์ยาก ไม่รู้ว่าต้องคลิกหรือแก้ตรงไหน ภาพในเกมดูรก และการใช้งานบนมือถือมีปัญหามากขึ้น แนวทางแก้คือทุกองค์ประกอบต้องอยู่ใน grid หรือ card ที่มีพื้นที่ชัดเจน ไม่ปล่อยให้วัตถุลอยอิสระทับกัน

## 1.3 UI ควรจัดจอเกมและส่วนแก้ปัญหาให้อยู่ข้างกัน

บทที่ 4 ควรจัดหน้าจอคล้ายบทที่ 2 โดยแบ่งพื้นที่เป็น 2 ส่วน คือด้านซ้ายเป็นจอเกมหรือตารางสถานการณ์ และด้านขวาเป็นแผงตรวจสอบและแก้ไขข้อผิดพลาด รูปแบบนี้ช่วยให้ผู้เรียนดูสถานการณ์ทางซ้าย แล้วตรวจเงื่อนไขหรือเลือกวิธีแก้ทางขวาได้ทันที

## 1.4 เนื้อหาควรจัดใหม่เป็น 3 ด่านหลักตามโครงสร้างเงื่อนไข

| Stage | เนื้อหา | โครงสร้าง | ระดับ |
|---|---|---|---|
| Stage 10 | IF | ถ้า...แล้ว... | พื้นฐาน |
| Stage 11 | IF / ELSE | ถ้า...แล้ว... มิฉะนั้น... | กลาง |
| Stage 12 | IF / ELSE IF / ELSE | ถ้า... มิฉะนั้นถ้า... มิฉะนั้น... | สูง |

แต่ละ stage มี 3 ด่านย่อย รวมทั้งหมด 9 ด่านย่อย โดยใช้ธีมเกษตรทั้งหมดและไต่ระดับความยากอย่างค่อยเป็นค่อยไป

# 2. เป้าหมายของการปรับปรุง

## 2.1 เป้าหมายด้านผู้เรียน

หลังเล่นบทที่ 4 นักเรียนควรสามารถ

1. อ่านสถานการณ์ง่าย ๆ ในฟาร์มได้
2. บอกได้ว่าระบบทำงานผิดเพราะเงื่อนไขใด
3. เลือกเงื่อนไขหรือคำสั่งที่ถูกต้องได้
4. เข้าใจความแตกต่างระหว่าง `IF`, `IF / ELSE` และ `IF / ELSE IF / ELSE`
5. ทดลองแก้ไขแล้วสังเกตผลลัพธ์ที่เปลี่ยนไปได้
6. เข้าใจว่าการตรวจสอบและแก้ไขข้อผิดพลาดต้องดูผลลัพธ์ ทดสอบ และปรับปรุง

## 2.2 เป้าหมายด้านเกม

เกมบทที่ 4 ควรมีลักษณะดังนี้

1. เล่นง่าย ไม่ซับซ้อนเกินวัย ป.4
2. เห็นปัญหาชัดเจนในตารางเกม
3. เห็นเงื่อนไขที่มีปัญหาในแผงด้านขวา
4. เลือกวิธีแก้ไขจากตัวเลือกไม่เกิน 3 ข้อ
5. มีปุ่ม “ทดสอบระบบ” เพื่อดูผลหลังแก้ไข
6. มี feedback ทันที
7. มี animation เบา ๆ เช่น สีแดง สีเขียว กระพริบ หรือวัตถุเปลี่ยนสถานะ
8. ไม่มีวัตถุทับซ้อนกัน
9. เล่นได้ดีทั้งคอมพิวเตอร์และมือถือ

## 2.3 เป้าหมายด้านเทคนิค

1. ใช้โครงสร้างเดิมของระบบให้มากที่สุด
2. ปรับ `SmartFarmDebuggerLite` ให้รองรับเกมแบบตารางเงื่อนไข
3. ปรับ `stage10.js`, `stage11.js`, `stage12.js` ให้มีด่านย่อยอย่างละ 3 ด่าน
4. ปรับ CSS ให้ใช้ grid/flex layout แทนการวางวัตถุแบบลอยอิสระ
5. รักษาการส่งคะแนนผ่าน `window.sendResult()` ให้ทำงานเหมือนเดิม
6. ลดการพึ่งพา Phaser สำหรับฉากหลัก หาก DOM Grid ให้ผลชัดกว่า
7. ใช้ Phaser เฉพาะจุดที่จำเป็น เช่น animation หรือ effect สั้น ๆ เท่านั้น
8. ปรับ responsive design ให้เหมาะกับมือถือ

# 3. แนวทาง UI ใหม่

## 3.1 โครงสร้าง Layout หลัก

หน้าจอเกมบทที่ 4 ควรใช้ 2 คอลัมน์

```text
┌──────────────────────────────┬──────────────────────────────┐
│         จอเกมตารางฟาร์ม          │       แผงตรวจสอบและแก้ไข       │
│                              │                              │
│  แสดงสถานการณ์ในรูปแบบตาราง     │  ภารกิจ                       │
│  แสดงผลก่อนแก้และหลังแก้         │  อาการผิดปกติ                  │
│  ใช้สี/animation บอกสถานะ        │  เงื่อนไขที่มีบั๊ก               │
│                              │  ตัวเลือกแก้ไข                  │
│                              │  ปุ่มทดสอบ / คำใบ้ / ถัดไป      │
└──────────────────────────────┴──────────────────────────────┘
```

บนจอคอมพิวเตอร์ใช้ 2 คอลัมน์ ส่วนบนมือถือให้เรียงเป็น 1 คอลัมน์ โดยให้จอเกมอยู่ด้านบนและแผงแก้ไขอยู่ด้านล่าง

## 3.2 จอเกมตารางฟาร์มด้านซ้าย

จอเกมควรเป็น grid ขนาด 3x3 หรือ 2x3 ตามความเหมาะสมของด่าน แต่ละ cell ควรมี icon หรือภาพขนาดเท่ากัน label ภาษาไทยสั้น ๆ และ status เช่น ปกติ / ผิดปกติ / แก้แล้ว

ตัวอย่าง grid 3x3

```text
┌───────────┬───────────┬───────────┐
│ สภาพอากาศ │ แปลงผัก    │ สปริงเกอร์ │
├───────────┼───────────┼───────────┤
│ ดิน       │ ถังน้ำ     │ พัดลม      │
├───────────┼───────────┼───────────┤
│ ผลลัพธ์    │ ไฟสถานะ    │ คะแนน      │
└───────────┴───────────┴───────────┘
```

## 3.3 แผงตรวจสอบและแก้ไขด้านขวา

แผงด้านขวาควรมีลำดับชัดเจน ได้แก่ ภารกิจ อาการเสีย เงื่อนไขที่มีบั๊ก คำถามนำคิด ตัวเลือกการแก้ไข ปุ่มทดสอบระบบ Feedback และปุ่มด่านถัดไป

ตัวอย่าง

```text
ภารกิจ:
ช่วยระบบรดน้ำแปลงผักให้ทำงานถูกต้อง

อาการเสีย:
ดินแห้ง แต่ระบบไม่เปิดน้ำ

เงื่อนไขที่มีบั๊ก:
ถ้า ดินชื้น → เปิดน้ำ

เลือกวิธีแก้:
A. ถ้า ดินแห้ง → เปิดน้ำ
B. ถ้า ฝนตก → เปิดน้ำ
C. ถ้า ดินชื้น → ปิดน้ำ

[ทดสอบระบบ]
```

## 3.4 กฎการออกแบบเพื่อป้องกันวัตถุซ้อนกัน

1. ทุกวัตถุในจอเกมต้องอยู่ใน `.farm-cell`
2. ห้ามใช้ absolute positioning สำหรับ object หลัก หากไม่จำเป็น
3. บล็อกเงื่อนไขต้องอยู่ใน panel ขวาเท่านั้น
4. ข้อความ feedback ต้องอยู่ในกล่อง feedback ไม่ลอยทับฉาก
5. การ highlight จุดที่ผิดให้ใช้ border, glow หรือ pulse effect รอบ cell
6. จำกัดความยาวข้อความใน cell ให้สั้นที่สุด
7. ปุ่มต้องมีระยะห่างเพียงพอสำหรับจอสัมผัส
8. ในมือถือให้ลดขนาด icon และเรียง cell ให้ไม่ล้นจอ
9. ใช้ `overflow-wrap`, `min-width: 0`, `max-width: 100%` กับข้อความภาษาไทย
10. ทดสอบที่ความกว้าง 360px, 768px, 1024px และ 1366px

# 5. Stage 10: IF  
## ภารกิจ IF ฟาร์มพื้นฐาน

Stage 10 เป็นด่านพื้นฐานที่สุด ใช้สอนโครงสร้าง

```text
ถ้า เงื่อนไขเป็นจริง → ทำคำสั่ง
```

ผู้เรียนต้องเข้าใจว่า ระบบจะทำงานเมื่อเงื่อนไขตรงกับสถานการณ์เท่านั้น

## 5.1 ด่าน 10-1: ดินแห้งต้องรดน้ำ

### สถานการณ์

ดินในแปลงผักแห้ง ผักเริ่มเหี่ยว แต่ระบบไม่เปิดน้ำ เพราะตั้งเงื่อนไขผิด

### เงื่อนไขที่มีบั๊ก

```text
ถ้า ดินชื้น → เปิดน้ำ
```

### เงื่อนไขที่ถูกต้อง

```text
ถ้า ดินแห้ง → เปิดน้ำ
```

### ตัวเลือก

| ตัวเลือก | ข้อความ | สถานะ |
|---|---|---|
| A | ถ้า ดินแห้ง → เปิดน้ำ | ถูก |
| B | ถ้า ฝนตก → เปิดน้ำ | ผิด |
| C | ถ้า ดินชื้น → ปิดน้ำ | ผิด |

### จอเกมก่อนแก้

| Cell | สถานะ |
|---|---|
| ดิน | ดินแห้ง |
| ผัก | ผักเหี่ยว |
| น้ำ | สปริงเกอร์ปิด |

### จอเกมหลังแก้ถูก

| Cell | สถานะ |
|---|---|
| ดิน | ดินแห้ง |
| ผัก | ผักสดชื่น |
| น้ำ | สปริงเกอร์เปิด |

### Feedback

- ถูก: `ถูกต้อง! เมื่อดินแห้ง ระบบต้องเปิดน้ำ`
- ผิด: `ลองดูอีกครั้ง ตอนนี้ปัญหาคือดินแห้ง ไม่ใช่ฝนตกหรือดินชื้น`

### ความรู้ที่ได้

เงื่อนไข IF ต้องตรงกับสถานการณ์จริง

## 5.2 ด่าน 10-2: แดดแรงต้องเปิดสแลน

### สถานการณ์

แดดแรงมาก ผักเริ่มเหี่ยว แต่ระบบไม่เปิดสแลนบังแดด

### เงื่อนไขที่มีบั๊ก

```text
ถ้า ฝนตก → เปิดสแลน
```

### เงื่อนไขที่ถูกต้อง

```text
ถ้า แดดแรง → เปิดสแลน
```

### ตัวเลือก

| ตัวเลือก | ข้อความ | สถานะ |
|---|---|---|
| A | ถ้า แดดแรง → เปิดสแลน | ถูก |
| B | ถ้า ฝนตก → เปิดสแลน | ผิด |
| C | ถ้า กลางคืน → เปิดสแลน | ผิด |

### จอเกมก่อนแก้

| Cell | สถานะ |
|---|---|
| อากาศ | แดดแรง |
| ผัก | ผักเหี่ยว |
| สแลน | ยังไม่กาง |

### จอเกมหลังแก้ถูก

| Cell | สถานะ |
|---|---|
| อากาศ | แดดแรง |
| ผัก | ผักเริ่มฟื้น |

# 6. Stage 11: IF / ELSE

# 7. Stage 12: IF / ELSE IF / ELSE  
## ภารกิจศูนย์ควบคุมฟาร์มหลายเงื่อนไข

- cell เปลี่ยนเป็นสีเขียว

# 16. สรุปแนวทางใหม่

การพัฒนาเกมบทที่ 4 ควรเปลี่ยนจาก “เกมฉากใหญ่” เป็น “เกมตารางเงื่อนไขธีมเกษตร” เพราะเนื้อหา Debugging โดยเฉพาะ IF, IF / ELSE และ IF / ELSE IF / ELSE ต้องการให้ผู้เรียนเห็นเงื่อนไขและผลลัพธ์อย่างชัดเจน

รูปแบบใหม่จะช่วยแก้ปัญหาเดิมได้หลายด้าน ได้แก่

- ลดปัญหาวัตถุและบล็อกคำสั่งทับซ้อนกัน
- ทำให้ผู้เรียนเห็นเงื่อนไขที่ผิดชัดเจน
- ทำให้จอเกมกับส่วนแก้ไขสัมพันธ์กัน
- เหมาะกับเด็ก ป.4 มากกว่า
- ยังมีความเป็นเกมผ่านสี สถานะ animation feedback ดาว และคำใบ้
- ต่อยอดระบบเดิมได้โดยไม่ต้องรื้อใหม่ทั้งหมด

เมื่อพัฒนาตามแผนนี้ บทที่ 4 จะเป็นเกมที่เข้าใจง่าย เล่นสนุก และตรงกับเป้าหมายการเรียนรู้เรื่องการตรวจสอบและแก้ไขข้อผิดพลาดมากขึ้น
