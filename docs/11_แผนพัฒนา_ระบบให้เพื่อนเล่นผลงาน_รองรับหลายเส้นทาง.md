# แผนการพัฒนาอย่างละเอียด: ระบบให้เพื่อนเล่นผลงาน และการรองรับคำตอบที่ถูกต้องได้หลายเส้นทาง

> เอกสารนี้เป็นแผนพัฒนาต่อจากระบบสร้างชิ้นงานบทที่ 2 “เส้นทางเดินรถไถ”  
> เป้าหมายคือเพิ่มระบบให้เพื่อนสามารถ “ลองเล่นโจทย์ที่นักเรียนออกแบบ” ได้จริง และปรับความหมายของ “คำสั่งเฉลย” ให้รองรับกรณีที่คำตอบที่ถูกต้องอาจมีได้มากกว่า 1 เส้นทาง  
> แนวทางสำคัญคือ **ตรวจความสำเร็จจากผลลัพธ์ของภารกิจ ไม่ใช่ตรวจว่าคำสั่งตรงกับคำสั่งของเจ้าของผลงานหรือไม่**

---

## 1. ที่มาและเหตุผล

ระบบสร้างชิ้นงานบทที่ 2 ในปัจจุบันให้นักเรียนออกแบบโจทย์เส้นทางรถไถ โดยกำหนดข้อมูลสำคัญ เช่น

- ประเภทภารกิจ
- จุดเริ่มต้นของรถไถ
- เป้าหมายหรือโรงนา
- สิ่งกีดขวาง
- ผลผลิต
- คำสั่งที่ใช้ทดสอบโจทย์
- สถานะ `validated`

โครงสร้างนี้พร้อมต่อยอดเป็นระบบให้เพื่อนเล่นผลงานได้แล้ว เพราะ `work_data` มีข้อมูลเพียงพอสำหรับสร้างโจทย์กลับมาเป็นเกมอีกครั้ง

อย่างไรก็ตาม มีประเด็นสำคัญที่ต้องแก้ก่อนพัฒนาระบบเล่นผลงาน คือคำว่า “คำสั่งเฉลย” อาจทำให้เข้าใจผิดว่าโจทย์หนึ่งมีคำตอบถูกต้องเพียงชุดเดียว ทั้งที่ในเกมเดินรถไถบนตาราง คำตอบที่ถูกต้องอาจมีได้หลายเส้นทาง

ตัวอย่างเช่น

```text
รถไถอาจเดินอ้อมกองฟางทางซ้ายหรือทางขวาก็ได้
ถ้าทั้งสองเส้นทางไม่ชนสิ่งกีดขวางและไปถึงเป้าหมายเหมือนกัน ทั้งสองคำตอบควรถูกต้อง
```

ดังนั้นระบบใหม่ต้องใช้หลักการว่า

```text
คำสั่งของเจ้าของผลงาน = ตัวอย่างคำตอบที่ผ่านการทดสอบ
คำสั่งของเพื่อน = คำตอบใหม่ที่ต้องตรวจด้วยกติกาภารกิจ
```

## 2. หลักการสำคัญของระบบใหม่

### 2.1 ไม่ตรวจว่าคำสั่งเหมือนกัน

ห้ามตรวจคำตอบของเพื่อนด้วยวิธีนี้

```javascript
playerCommands === ownerCommands
```

เพราะจะทำให้คำตอบอื่นที่ถูกต้องถูกตัดสินว่าผิด

### 2.2 ต้องตรวจจากผลลัพธ์ของภารกิจ

ระบบต้องนำคำสั่งของผู้เล่นไปจำลองเดินบนแผนที่จริง แล้วตรวจว่าเป็นไปตามเงื่อนไขหรือไม่

```text
1. รถไถไม่ออกนอกแผนที่
2. รถไถไม่ชนสิ่งกีดขวาง
3. ถ้าเป็นภารกิจไปถึงจุดหมาย ต้องไปถึง target
4. ถ้าเป็นภารกิจหลบสิ่งกีดขวาง ต้องไปถึง target โดยไม่ชน obstacle
5. ถ้าเป็นภารกิจเก็บเกี่ยว ต้องเก็บ crops ครบก่อนถึง barn
```

### 2.3 เปลี่ยนความหมายของ `commands`

ใน `work_data` ปัจจุบันมี field

```json
"commands": ["RIGHT", "RIGHT", "UP"]
```

ให้ตีความใหม่ว่า

```text
commands = คำสั่งตัวอย่างของผู้ออกแบบ
```

ไม่ใช่

```text
commands = คำตอบเดียวที่ถูกต้อง
```

### 2.4 ยังไม่จำเป็นต้องแก้ database

รอบแรกยังใช้ field `commands` เดิมได้ เพื่อไม่ให้ต้องแก้ฐานข้อมูลและระบบบันทึกงานหลายจุด

แต่ควรเปลี่ยนข้อความใน UI จาก “คำสั่งเฉลย” เป็น

```text
คำสั่งตัวอย่าง
```

หรือ

```text
หนึ่งวิธีที่ทำภารกิจสำเร็จ
```

## 3. เป้าหมายการพัฒนา

### 3.1 เป้าหมายหลัก

เพิ่มระบบให้เพื่อนสามารถเล่นโจทย์จากผลงานของนักเรียนได้ โดยซ่อนคำสั่งตัวอย่างไว้ก่อน และให้ผู้เล่นสร้างคำสั่งของตนเองเพื่อแก้โจทย์

### 3.2 เป้าหมายย่อย

1. เพิ่มปุ่ม “ลองเล่นโจทย์นี้” ในหน้า showcase
2. สร้างหน้าใหม่สำหรับเล่นผลงาน เช่น `pages/play_student_work.php`
3. โหลด `work_data` จาก `student_works`
4. ตรวจสิทธิ์ว่าเป็นงานที่ผู้เล่นสามารถเข้าถึงได้
5. แสดงโจทย์จากชิ้นงานเป็นเกมรถไถ
6. ซ่อนคำสั่งตัวอย่างของเจ้าของผลงานในตอนเริ่ม
7. ให้เพื่อนสร้างคำสั่งของตนเอง
8. ตรวจคำตอบด้วย simulation ตามกติกาภารกิจ
9. ยอมรับคำตอบที่ถูกต้องหลายเส้นทาง
10. หลังเล่นผ่าน อาจให้ดู “วิธีตัวอย่างของผู้ออกแบบ”
11. รอบแรกยังไม่ต้องบันทึกคะแนนลงฐานข้อมูล

## 4. ขอบเขตงานรอบแรก: MVP

### 4.1 สิ่งที่ต้องทำใน MVP

- เพิ่มปุ่ม “ลองเล่นโจทย์นี้” ใน `showcase.php`
- สร้างหน้า `play_student_work.php`
- ใช้ `work_data` ของผลงานที่ validated แล้ว
- สร้างเกมให้เพื่อนเล่นจากข้อมูลใน `work_data`
- ให้เพื่อนสร้างคำสั่งเอง
- ตรวจผลจากการจำลองภารกิจจริง
- แสดงผลว่าผ่านหรือยังไม่ผ่าน
- มีปุ่มดูวิธีตัวอย่างหลังจากเล่นผ่าน หรือหลังจากผู้เล่นกดยอมดู

### 4.2 สิ่งที่ยังไม่ทำใน MVP

- ยังไม่ต้องบันทึกผลการเล่นลงฐานข้อมูล
- ยังไม่ต้องมี leaderboard
- ยังไม่ต้องมีระบบจัดอันดับเวลา
- ยังไม่ต้องนับจำนวนคนเล่นสำเร็จ
- ยังไม่ต้องเก็บสถิติรายคน
- ยังไม่ต้องให้คะแนนเจ้าของผลงานจากจำนวนคนเล่น
- ยังไม่ต้องสร้าง table ใหม่
- ยังไม่ต้องแก้ database

## 6. โครงสร้างข้อมูล `work_data`

### 6.1 ใช้โครงเดิมต่อได้

ตัวอย่าง `work_data` ปัจจุบัน

```json
{
  "project_type": "tractor_route",
  "mission_type": "obstacle",
  "grid": {
    "cols": 6,
    "rows": 5
  },
  "start": {
    "col": 0,
    "row": 4
  },
  "target": {
    "col": 5,
    "row": 0
  },
  "barn": null,
  "obstacles": [
    { "col": 2, "row": 3, "type": "hay" }
  ],
  "crops": [],
  "commands": ["RIGHT", "UP", "RIGHT", "UP", "RIGHT", "RIGHT"],
  "validated": true
}
```

ให้ตีความว่า

```text
commands = sample solution
validated = เจ้าของผลงานพิสูจน์แล้วว่าโจทย์มีอย่างน้อย 1 วิธีผ่าน
```

### 6.2 เพิ่ม field ใหม่ในอนาคตได้ แต่ยังไม่จำเป็น

ถ้าต้องการให้ชัดขึ้นในอนาคต อาจเพิ่ม field เช่น

```json
{
  "sample_commands": ["RIGHT", "UP", "RIGHT"],
  "allow_multiple_solutions": true,
  "solution_type": "sample"
}
```

แต่ MVP ยังไม่ต้องเพิ่ม เพราะจะทำให้ต้องแก้ระบบบันทึกและ preview หลายจุด

### 6.3 แนวทาง backward compatibility

ถ้าอนาคตเพิ่ม `sample_commands` ให้รองรับทั้งแบบเก่าและใหม่

```javascript
const sampleCommands = data.sample_commands || data.commands || [];
```

## 8. สร้างหน้า `play_student_work.php`

### 8.1 ชื่อไฟล์ที่แนะนำ

```text
pages/play_student_work.php
```

### 8.2 หน้าที่ของหน้านี้

หน้า `play_student_work.php` ทำหน้าที่โหลดผลงานจาก `student_works` แล้วสร้างเกมรถไถจาก `work_data`

### 8.3 Flow การทำงาน

```text
1. รับ work_id จาก query string
2. ตรวจ session ผู้ใช้
3. ดึงข้อมูลผลงานจาก student_works
4. ตรวจว่า work นี้เป็น project_type = tractor_route
5. ตรวจว่า validated = true
6. ตรวจสิทธิ์ตาม school_id / classroom_id / learning_session_id
7. แสดงข้อมูลเจ้าของผลงาน
8. แสดงภารกิจ
9. แสดงแผนที่
10. ให้ผู้เล่นสร้างคำสั่งเอง
11. กดรันเพื่อทดสอบ
12. ถ้าผ่าน แสดงผลสำเร็จ
13. เปิดปุ่ม “ดูวิธีตัวอย่างของผู้ออกแบบ”
```

## 9. การตรวจสิทธิ์ฝั่ง Server

### 9.1 เหตุผล

ห้ามให้ผู้ใช้เปิด

```text
play_student_work.php?work_id=123
```

แล้วดูงานที่อยู่นอกโรงเรียน ห้องเรียน หรือรอบการเรียนของตนเอง

### 9.2 เงื่อนไขขั้นต่ำ

ควรตรวจอย่างน้อย

```text
1. ผู้ใช้ login แล้ว
2. work.status IN ('submitted', 'reviewed')
3. work.game_id = 2
4. work_data.project_type = 'tractor_route'
5. work_data.validated = true
6. school_id ตรงกับผู้ใช้
7. ถ้ามี classroom_id / learning_session_id ต้องอยู่ในบริบทที่เกี่ยวข้อง
```

### 9.3 ตัวอย่าง query แนวคิด

```php
$work_id = (int)($_GET['work_id'] ?? 0);

$stmt = $conn->prepare("
    SELECT sw.*, u.fullname, u.nickname
    FROM student_works sw
    JOIN users u ON sw.student_id = u.id
    WHERE sw.id = ?
      AND sw.game_id = 2
      AND sw.status IN ('submitted', 'reviewed')
      AND sw.school_id = ?
    LIMIT 1
");
```

หมายเหตุ: ปรับ field ตาม schema จริงของระบบ

### 9.4 ตรวจ work_data หลัง query

```php
$data = json_decode($work['work_data'], true);

if (!$data || ($data['project_type'] ?? '') !== 'tractor_route') {
    die('ไม่พบโจทย์เส้นทางรถไถ');
}

if (empty($data['validated'])) {
    die('โจทย์นี้ยังไม่พร้อมให้เล่น');
}
```

### 12.2 รูปแบบผลลัพธ์

```javascript
{
    success: true,
    reason: 'success',
    message: 'สำเร็จ! คุณแก้โจทย์นี้ได้แล้ว',
    path: [
        { col: 0, row: 4, step: 0 },
        { col: 1, row: 4, step: 1, dir: 'RIGHT' }
    ],
    collected: ['1,2', '4,1']
}
```

ถ้าไม่ผ่าน

```javascript
{
    success: false,
    reason: 'obstacle',
    message: 'รถไถชนกองฟาง ลองวางเส้นทางใหม่',
    failedAt: { col: 2, row: 3 },
    path: [...]
}
```

### 12.3 เหตุผลที่ควรรองรับ

| reason | ความหมาย |
|---|---|
| success | ผ่านภารกิจ |
| no_commands | ยังไม่มีคำสั่ง |
| out_of_bounds | รถไถออกนอกแผนที่ |
| obstacle | ชนสิ่งกีดขวาง |
| not_reached_target | คำสั่งหมดแต่ยังไม่ถึงเป้าหมาย |
| crops_missing | ถึงโรงนาแล้วแต่เก็บผลผลิตไม่ครบ |
| not_reached_barn | เก็บครบแล้วแต่ยังไม่ถึงโรงนา |
| invalid_data | ข้อมูลโจทย์ไม่สมบูรณ์ |

### 12.4 ตัวอย่าง validator

```javascript
function validatePlayerCommands(data, playerCommands) {
    const cols = data.grid?.cols || 6;
    const rows = data.grid?.rows || 5;

    if (!Array.isArray(playerCommands) || playerCommands.length === 0) {
        return {
            success: false,
            reason: 'no_commands',
            message: 'เพิ่มคำสั่งก่อนกดรันนะ'
        };
    }

    const dirs = {
        UP: { dc: 0, dr: -1 },
        DOWN: { dc: 0, dr: 1 },
        LEFT: { dc: -1, dr: 0 },
        RIGHT: { dc: 1, dr: 0 }
    };

    let pos = { col: data.start.col, row: data.start.row };
    const path = [{ ...pos, step: 0 }];
    const collected = new Set();

    const obstacles = new Set((data.obstacles || []).map(o => `${o.col},${o.row}`));
    const crops = data.crops || [];

    for (let i = 0; i < playerCommands.length; i++) {
        const cmd = playerCommands[i];
        const dir = dirs[cmd];

        if (!dir) continue;

        pos = {
            col: pos.col + dir.dc,
            row: pos.row + dir.dr
        };

        path.push({ ...pos, step: i + 1, dir: cmd });

        if (pos.col < 0 || pos.col >= cols || pos.row < 0 || pos.row >= rows) {
            return {
                success: false,
                reason: 'out_of_bounds',
                message: 'รถไถออกนอกแปลง ลองตรวจทิศทางอีกครั้ง',
                failedAt: pos,
                path,
                collected: Array.from(collected)
            };
        }

        if (obstacles.has(`${pos.col},${pos.row}`)) {
            return {
                success: false,
                reason: 'obstacle',
                message: 'รถไถชนสิ่งกีดขวาง ลองวางเส้นทางใหม่',
                failedAt: pos,
                path,
                collected: Array.from(collected)
            };
        }

        crops.forEach((crop, index) => {
            if (crop.col === pos.col && crop.row === pos.row) {
                collected.add(String(index));
            }
        });
    }

    if (data.mission_type === 'harvest') {
        const barn = data.barn;
        const totalCrops = crops.length;

        if (collected.size < totalCrops) {
            return {
                success: false,
                reason: 'crops_missing',
                message: 'ยังเก็บผลผลิตไม่ครบ ลองวางเส้นทางใหม่',
                path,
                collected: Array.from(collected)
            };
        }

        if (!barn || pos.col !== barn.col || pos.row !== barn.row) {
            return {
                success: false,
                reason: 'not_reached_barn',
                message: 'เก็บผลผลิตแล้ว ต้องกลับไปที่โรงนาให้สำเร็จ',
                path,
                collected: Array.from(collected)
            };
        }

        return {
            success: true,
            reason: 'success',
            message: 'สำเร็จ! คุณเก็บผลผลิตครบและกลับถึงโรงนาแล้ว',
            path,
            collected: Array.from(collected)
        };
    }

    const target = data.target;
    if (!target || pos.col !== target.col || pos.row !== target.row) {
        return {
            success: false,
            reason: 'not_reached_target',
            message: 'ยังไม่ถึงเป้าหมาย ลองปรับลำดับคำสั่งอีกครั้ง',
            path,
            collected: Array.from(collected)
        };
    }

    return {
        success: true,
        reason: 'success',
        message: 'สำเร็จ! คุณแก้โจทย์นี้ได้แล้ว',
        path,
        collected: Array.from(collected)
    };
}
```

### 12.5 จุดสำคัญ

Validator นี้ไม่เคยเปรียบเทียบกับ `data.commands`


เพราะ `data.commands` เป็นแค่ตัวอย่างคำตอบ

## 23. ตัวอย่างกรณีทดสอบคำตอบหลายเส้นทาง

### กรณีที่ 1: ไปถึงเป้าหมายคนละเส้นทาง

```text
เจ้าของผลงาน:
RIGHT, RIGHT, UP, UP

เพื่อน:
UP, UP, RIGHT, RIGHT

ถ้าทั้งสองถึง target และไม่ผิดกติกา ต้องถือว่าถูกทั้งคู่
```

### กรณีที่ 2: หลบสิ่งกีดขวางคนละด้าน

```text
เจ้าของผลงานเดินอ้อมซ้าย
เพื่อนเดินอ้อมขวา

ถ้าไม่ชน obstacle และถึง target ต้องถือว่าถูก
```

### กรณีที่ 3: เก็บผลผลิตคนละลำดับ

```text
เจ้าของผลงานเก็บ crop A ก่อน crop B
เพื่อนเก็บ crop B ก่อน crop A

ถ้าเก็บครบและกลับ barn ต้องถือว่าถูก
```

### กรณีที่ 4: ใช้คำสั่งมากกว่าแต่สำเร็จ

```text
เพื่อนเดินอ้อมไกลกว่า แต่ไม่ชนและถึงเป้าหมาย

ถือว่าถูก แต่ feedback อาจบอกว่าใช้คำสั่งมากกว่าวิธีตัวอย่าง
```

## 24. คำสั่งสำหรับ Codex

```text
พัฒนาระบบให้เพื่อนเล่นผลงานของบทที่ 2 โดยรองรับคำตอบที่ถูกต้องได้หลายเส้นทาง

1. เปลี่ยนถ้อยคำในระบบจาก “คำสั่งเฉลย” เป็น “คำสั่งตัวอย่าง” หรือ “วิธีตัวอย่างของผู้ออกแบบ”
2. ห้ามตรวจคำตอบของผู้เล่นด้วยการเทียบกับ commands ของเจ้าของผลงาน
3. ให้ commands ใน work_data เป็น sample solution เท่านั้น
4. เพิ่มปุ่ม “ลองเล่นโจทย์นี้” ใน pages/showcase.php สำหรับงาน tractor_route ที่ validated = true
5. สร้างหน้า pages/play_student_work.php
6. play_student_work.php ต้องรับ work_id และตรวจสิทธิ์ฝั่ง server
7. ต้องตรวจว่า work.game_id = 2, project_type = tractor_route, validated = true, status อยู่ใน submitted/reviewed
8. หน้าเล่นผลงานต้องซ่อน commands ของเจ้าของผลงานตอนเริ่ม
9. ผู้เล่นต้องสร้างคำสั่งของตนเองด้วยปุ่มลูกศร
10. ระบบต้องจำลองคำสั่งของผู้เล่นบนแผนที่จริง
11. ตรวจผลลัพธ์ตาม mission_type:
    - target: ถึง target
    - obstacle: ถึง target โดยไม่ชน obstacle
    - harvest: เก็บ crops ครบแล้วถึง barn
12. ถ้าผู้เล่นใช้เส้นทางอื่นที่สำเร็จ ต้องถือว่าถูก
13. หลังเล่นผ่าน ให้แสดงจำนวนคำสั่งของผู้เล่นเทียบกับวิธีตัวอย่างของผู้ออกแบบ
14. เพิ่มปุ่ม “ดูวิธีตัวอย่างของผู้ออกแบบ”
15. ยังไม่ต้องบันทึกผลการเล่นลงฐานข้อมูลในรอบนี้
16. ห้ามแก้ database ในรอบนี้
17. ห้ามแก้บทที่ 3 และ 4
18. ทดสอบ target, obstacle, harvest และกรณีคำตอบหลายเส้นทาง
```

## 27. ความเสี่ยงและวิธีลดความเสี่ยง

| ความเสี่ยง | วิธีลดความเสี่ยง |

| คำตอบที่ถูกแต่ยาวมากอาจผ่าน | MVP ถือว่าผ่านก่อน ภายหลังค่อยเพิ่ม challenge เรื่องจำนวนคำสั่ง |

## 28. สรุป

ระบบให้เพื่อนเล่นผลงานควรพัฒนาบนหลักการว่า

```text
ผลงานของนักเรียน = โจทย์ที่นักเรียนออกแบบ
commands ของเจ้าของผลงาน = วิธีตัวอย่างหนึ่งวิธี
คำตอบของเพื่อน = ชุดคำสั่งใหม่ที่ต้องตรวจด้วยกติกาภารกิจ
```

ดังนั้นการตรวจคำตอบต้องใช้การจำลองเส้นทางจริง ไม่ใช่การเทียบกับคำสั่งตัวอย่าง

แนวทางนี้จะทำให้ระบบสร้างชิ้นงานบทที่ 2 มีคุณภาพสูงขึ้นมาก เพราะนักเรียนไม่ได้เพียงส่งผลงานให้ครูดู แต่สามารถสร้างโจทย์ให้เพื่อนเล่น ทดลองแก้ และเปรียบเทียบวิธีคิดได้จริง ซึ่งสอดคล้องกับการเรียนรู้แบบใช้เกมเป็นฐานและการแก้ปัญหาอย่างเป็นขั้นตอน
