# 24_แผนพัฒนาเกมบทที่ 4  
# Smart Farm Debugger: ภารกิจจับบั๊กกู้ฟาร์ม  
## ฉบับขัดเกลาเพื่อพัฒนาจริง

**บทเรียน:** บทที่ 4 การตรวจหาและแก้ไขข้อผิดพลาด  
**ชื่อบทเรียนในระบบ:** บทที่ 4: ช่างซ่อมระบบฟาร์มอัจฉริยะ  
**ชื่อเกมรวม:** Smart Farm Debugger: ภารกิจจับบั๊กกู้ฟาร์ม  
**แนวเกม:** Bug Hunt + System Repair + Visual Block Debugging + Farm Simulation  
**กลุ่มเป้าหมาย:** นักเรียนชั้นประถมศึกษาปีที่ 4  
**สถานะเอกสาร:** ใช้เป็นแผนพัฒนาเกมและสเปกสำหรับสั่งพัฒนา / ตรวจงาน  
**แนวทางสำคัญ:** ต่อจากบทที่ 3 โดยให้ผู้เรียนเปลี่ยนบทบาทจาก “ผู้สร้างกฎ” เป็น “ผู้ซ่อมกฎที่ผิดพลาด”

## 1. สรุปแนวคิดหลัก

บทที่ 4 ควรต่อยอดจากบทที่ 3 อย่างเป็นธรรมชาติ บทที่ 3 ผู้เรียนได้สร้างกฎควบคุมระบบฟาร์มด้วย If / Else / Else If แล้ว ส่วนบทที่ 4 ผู้เรียนจะได้รับระบบฟาร์มที่ “ทำงานผิดพลาด” และต้องทำหน้าที่เป็นช่างซ่อมระบบ

แกนหลักของเกมคือ

```text
ระบบฟาร์มทำงานผิดพลาด
→ ผู้เรียนสังเกตอาการ
→ วิเคราะห์ว่าสาเหตุอาจอยู่ที่บล็อกใด
→ ระบุบั๊ก
→ แก้เงื่อนไขหรือคำสั่ง
→ ทดสอบระบบอีกครั้ง
→ ระบบกลับมาทำงานถูกต้อง
```

หัวใจของบทนี้ไม่ใช่การเดาคำตอบ แต่คือการฝึกกระบวนการ Debugging ที่เป็นขั้นตอน ได้แก่

1. ดูผลลัพธ์ที่ผิด
2. ตั้งสมมติฐานว่าสาเหตุคืออะไร
3. ตรวจบล็อกที่น่าสงสัย
4. แก้เฉพาะจุดที่เป็นสาเหตุ
5. ทดสอบใหม่
6. อธิบายเหตุผลได้ว่าแก้ถูกเพราะอะไร

## 2. เหตุผลที่บทที่ 4 ควรเป็นเกม Debugging

บทที่ 3 สอนให้เด็ก “สร้างตรรกะ” ส่วนบทที่ 4 ควรสอนให้เด็ก “ตรวจตรรกะ” เพราะในการคิดเชิงคำนวณ การเขียนคำสั่งให้ถูกตั้งแต่แรกไม่ใช่เรื่องเดียวที่สำคัญ แต่การหาจุดผิด แก้ไข และทดสอบซ้ำก็สำคัญไม่แพ้กัน

บทนี้จึงควรให้ผู้เรียนเข้าใจว่า

```text
ถ้าระบบทำงานผิด ไม่ได้แปลว่าระบบพังทั้งหมด
แต่อาจมีบล็อกเล็ก ๆ บางจุดที่ต้องแก้ให้ถูก
```

เกมควรทำให้การ Debug ดูสนุก จับต้องได้ และไม่กดดันเกินไป โดยใช้ “แมลงบั๊ก” เป็นตัวละครแทนข้อผิดพลาด เช่น บั๊กซ่อนอยู่ในเงื่อนไข บั๊กแอบสลับคำสั่ง หรือบั๊กทำให้ลำดับเงื่อนไขเพี้ยน

## 3. ความเชื่อมโยงกับบทที่ 3

บทที่ 4 ควรใช้โลกเดียวกับบทที่ 3 เพื่อให้ผู้เรียนรู้สึกต่อเนื่อง แต่ต้องเปลี่ยนรูปแบบการเล่นให้ชัดเจน

| บท | บทบาทผู้เรียน | สิ่งที่ผู้เรียนทำ |
|---|---|---|
| บทที่ 3 | ผู้จัดการฟาร์มอัจฉริยะ | สร้างกฎควบคุมระบบฟาร์ม |
| บทที่ 4 | ช่างซ่อมระบบฟาร์ม | ตรวจหาบั๊ก แก้กฎ และทดสอบระบบ |

ตัวอย่างความต่อเนื่อง:

- บทที่ 3: สร้างกฎคัดแครอทเปื้อนโคลนเข้าเครื่องล้าง  
- บทที่ 4: ระบบส่งแครอทสะอาดเข้าเครื่องล้างผิด ต้องหาว่าเงื่อนไขผิดตรงไหน  

- บทที่ 3: สร้างกฎคัดไข่ไก่หลายระดับ  
- บทที่ 4: ไข่ร้าวหลุดเข้าถาดพรีเมียม ต้องแก้เงื่อนไขที่กว้างเกินไป  

- บทที่ 3: เรียนว่า If ไม่มี Else และ pass-through คือพฤติกรรมปกติ  
- บทที่ 4: ต้องรู้ว่าในด่าน If บางวัตถุผ่านไปเฉย ๆ คือ “ถูก” ไม่ใช่บั๊ก

## 4. เป้าหมายการเรียนรู้

เมื่อจบบทที่ 4 ผู้เรียนควรสามารถ

1. อธิบายได้ว่า Debugging คือการตรวจหาและแก้ไขข้อผิดพลาด
2. สังเกตผลลัพธ์ที่ผิดจากระบบฟาร์มได้
3. เชื่อมโยงอาการผิดกับสาเหตุที่เป็นไปได้
4. แยกประเภทบั๊กพื้นฐานได้
5. ระบุบล็อกที่เป็นสาเหตุของบั๊กได้
6. แก้บล็อกเงื่อนไขหรือคำสั่งได้ถูกต้อง
7. ทดสอบระบบหลังแก้ไขได้
8. อธิบายเหตุผลของการแก้ไขได้
9. เข้าใจว่า Debugging ต้องอาศัยหลักฐานจากผลลัพธ์ ไม่ใช่การเดาสุ่ม
10. สร้างโจทย์บั๊กของตนเองให้เพื่อนแก้ได้

## 5. หลักการออกแบบเกม

### 5.1 Run Before Fix

ทุกด่านต้องเริ่มจากการให้ระบบทำงานผิดให้ผู้เรียนดู 1 รอบก่อน ไม่ควรให้เด็กเห็นโค้ดแล้วแก้ทันที เพราะเป้าหมายคือฝึกการสังเกตอาการ

ตัวอย่าง:

```text
ระบบคัดไข่เริ่มทำงาน
ไข่ใบใหญ่แต่ร้าวถูกส่งเข้าถาดพรีเมียม
ถาดพรีเมียมสั่นและขึ้นเครื่องหมายเตือน
ระบบหยุดและถามว่า “เกิดอะไรขึ้น?”
```

### 5.2 Observe → Diagnose → Fix → Test

ทุกด่านควรบังคับ flow นี้

```text
Observe: ดูอาการ
Diagnose: เลือกประเภทบั๊กหรือจุดที่ผิด
Fix: แก้บล็อก
Test: ทดสอบใหม่
Explain: สรุปเหตุผล
```

### 5.3 ไม่ควรให้ลองมั่วได้ไม่จำกัด

ถ้าผู้เรียนลองผิดลองถูกได้ไม่จำกัด เกมจะกลายเป็นการเดา ดังนั้นควรมีจำนวนครั้งในการตรวจหรือแก้จำกัด เช่น

- Stage 10: แก้ได้ 3 ครั้ง
- Stage 11: แก้ได้ 3 ครั้ง
- Stage 12: แก้ได้ 4 ครั้ง แต่มีหลายระบบ

### 5.4 Feedback ต้องอธิบายเหตุผล

Feedback ต้องไม่ใช่แค่ “ถูก/ผิด” แต่ต้องอธิบายว่า

```text
เกิดอะไรขึ้น
เพราะอะไร
ต้องแก้ตรงไหน
```

## 7. ประเภทบั๊กที่ใช้ในบทที่ 4

### 7.1 Action Bug: คำสั่งผิด

เงื่อนไขถูก แต่ทำคำสั่งผิด

```text
ถ้า [แครอทเปื้อนโคลน] → [ปล่อยผ่าน]
```

ควรแก้เป็น

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

### 7.2 Condition Bug: เงื่อนไขผิด

คำสั่งถูก แต่เงื่อนไขผิด

```text
ถ้า [แครอทสะอาด] → [ส่งเข้าเครื่องล้าง]
```

ควรแก้เป็น

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

### 7.3 Broad Condition Bug: เงื่อนไขกว้างเกินไป

เงื่อนไขไม่ละเอียดพอ ทำให้ตัวหลอกเข้าเงื่อนไขผิด

```text
ถ้า [ไข่ใบใหญ่] → [ถาดพรีเมียม]
```

ทำให้ไข่ใบใหญ่แต่ร้าวเข้าถาดพรีเมียมผิด

ควรแก้เป็น

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
```

### 7.4 Order Bug: ลำดับเงื่อนไขผิด

เงื่อนไขแต่ละอันถูก แต่เรียงผิด

```text
ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
```

ควรแก้เป็น

```text
ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
```

### 7.5 Missing Else Bug: ขาด Else

ระบบไม่รู้จะทำอะไรเมื่อไม่เข้าเงื่อนไขใด

```text
ถ้า [อุณหภูมิ > 35°C] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [อุณหภูมิ < 15°C] → [เปิดหลอดไฟ]
```

ควรเพิ่ม

```text
นอกเหนือจากนี้ → [ปิดระบบทั้งหมด]
```

### 7.6 Numeric Bug: ค่าตัวเลขหรือเครื่องหมายผิด

```text
ถ้า [อุณหภูมิ > 50°C] → [เปิดพัดลม]
```

ควรแก้เป็น

```text
ถ้า [อุณหภูมิ > 35°C] → [เปิดพัดลม]
```

### 7.7 If Pass-through Misunderstanding Bug

บั๊กเฉพาะที่เกี่ยวข้องกับบทที่ 3 เกม If

ตัวอย่างบั๊ก:

```text
แครอทมีตำหนิถูกส่งเข้าเครื่องล้าง
```

แต่โจทย์คือ

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

แครอทมีตำหนิแต่ไม่เปื้อนโคลนควร “ปล่อยผ่าน” ไม่ใช่ส่งเข้าเครื่องล้าง

บั๊กนี้ช่วยตอกย้ำว่าใน If การไม่เข้าเงื่อนไขแล้วผ่านไปเฉย ๆ คือพฤติกรรมที่ถูกต้อง

## 8. Core Game Loop รายละเอียด

### 8.1 Step 1: System Run

ระบบรันให้ดูโดยยังไม่เปิดแผงแก้ไข

สิ่งที่ต้องมี:

- Animation ระบบทำงาน
- วัตถุหรือเหตุการณ์ที่ผิดพลาด
- เสียงเตือนหรือไฟแดง
- ข้อความสั้น ๆ บอกอาการ

### 8.2 Step 2: Symptom Check

ผู้เรียนตอบคำถามสั้น ๆ ว่าเห็นอาการอะไร

ตัวเลือกตัวอย่าง:

- วัตถุถูกส่งผิดปลายทาง
- เครื่องจักรทำงานผิดเวลา
- ระบบไม่ทำงาน
- ตัวหลอกหลุดเข้าเงื่อนไข
- ระบบไม่มีคำสั่งสำหรับกรณีปกติ

ขั้นนี้ช่วยให้ผู้เรียนไม่ข้ามจาก “เห็นผิด” ไป “แก้มั่ว”

### 8.3 Step 3: Bug Hunt

ผู้เรียนใช้แว่นขยายหรือโหมดตรวจบั๊กคลิกบล็อกที่คิดว่าผิด

ถ้าคลิกถูก:

```text
เจอบั๊กแล้ว!
บั๊กซ่อนอยู่ในเงื่อนไขแถวแรก
```

ถ้าคลิกผิด:

```text
บล็อกนี้ยังไม่ใช่สาเหตุหลัก ลองดูว่าอะไรถูกส่งผิดปลายทาง
```

### 8.4 Step 4: Repair

ระบบเปิดช่องให้แก้เฉพาะส่วนที่เกี่ยวข้อง เช่น

- เปลี่ยน condition
- เปลี่ยน action
- สลับลำดับ rule
- เพิ่ม Else
- แก้ค่าตัวเลข

ไม่ควรให้เด็กแก้ทั้งโปรแกรมแบบอิสระทั้งหมดในด่านต้น ๆ เพราะจะยากเกินไป

### 8.5 Step 5: Retest

หลังแก้ ต้องกด “ทดสอบระบบอีกครั้ง”

ถ้าถูก ระบบแสดง animation สำเร็จ  
ถ้าผิด ระบบรันอาการผิดอีกครั้ง แต่ให้ feedback เพิ่มขึ้น

### 8.6 Step 6: Explain

หลังผ่านด่าน ต้องมีสรุปเชิงเหตุผล เช่น

```text
บั๊กนี้เกิดจากเงื่อนไขกว้างเกินไป
คำว่า “ไข่ใบใหญ่” รวมไข่ใบใหญ่ที่ร้าวด้วย
จึงต้องเปลี่ยนเป็น “ไข่ใบใหญ่และไม่ร้าว”
```

## 9. UI / UX ที่เสนอ

### 9.1 Layout หลัก

```text
┌──────────────────────────────────────────────┐
│ ชื่อด่าน | จำนวนครั้งแก้ | ดาว | สถานะระบบ   │
├──────────────────────────────────────────────┤
│                                              │
│     พื้นที่แสดงระบบฟาร์ม / สายพาน / โรงเรือน │
│                                              │
├───────────────────────┬──────────────────────┤
│ แผงบล็อกที่มีบั๊ก      │ เครื่องมือ Debug       │
│ ถ้า [____] → [____]    │ [ดูอาการซ้ำ]           │
│ Else If [____] → ...   │ [แว่นขยายจับบั๊ก]      │
│ Else → [____]          │ [ทดสอบอีกครั้ง]        │
└───────────────────────┴──────────────────────┘
```

### 9.2 เครื่องมือหลัก

| เครื่องมือ | หน้าที่ |
|---|---|
| ดูอาการซ้ำ | รันเหตุการณ์ผิดอีกครั้ง |
| แว่นขยายจับบั๊ก | เลือกบล็อกที่คิดว่าผิด |
| กล่องซ่อมบล็อก | ลาก condition/action ที่ถูกเข้าไปแทน |
| ทดสอบอีกครั้ง | รันระบบหลังแก้ |
| คำใบ้ | ช่วยแบบเป็นลำดับ ไม่เฉลยทันที |

### 9.3 การแสดงบล็อก

บล็อกควรใช้สีเหมือนบทที่ 3 เพื่อให้ผู้เรียนเชื่อมโยงได้

- Condition: น้ำเงิน
- Action: เขียว
- If / Else If / Else: ส้ม
- Bug highlight: แดง
- Fixed: เขียวอ่อน

### 9.4 UX สำคัญ

- ก่อนเจอบั๊ก ห้ามแก้บล็อกทันที
- หลังเจอบั๊ก จึงเปิดช่องซ่อม
- ถ้าเลือกบล็อกผิด ไม่ควรลงโทษรุนแรง แต่ควรลดโอกาสได้ 3 ดาว
- ต้องมีปุ่มดูอาการซ้ำ เพราะเด็ก ป.4 อาจดูรอบแรกไม่ทัน
- คำใบ้ต้องช่วยคิด ไม่ใช่เฉลยทันที

## 10. รายละเอียด Stage 10: บั๊กจิ๋วในโรงคัดแยก

**จุดเน้น:** บั๊กพื้นฐาน  
**ความยาก:** ง่าย  
**เป้าหมาย:** ให้ผู้เรียนแยกได้ว่า ปัญหาเกิดจาก condition, action หรือ condition ที่กว้างเกินไป

---

### 10-1 แครอทเลอะหลุดตะกร้า

**อาการ:** แครอทเปื้อนโคลนไม่เข้าเครื่องล้าง แต่ถูกปล่อยผ่าน

**กฎที่มีบั๊ก**

```text
ถ้า [แครอทเปื้อนโคลน] → [ปล่อยผ่าน]
```

**ประเภทบั๊ก:** Action Bug  
**วิธีแก้**

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

**จุดสอน:** เงื่อนไขถูก แต่คำสั่งผิด

**Animation:** แครอทเลอะหลุดไปตามสายพาน ตะกร้ามีคราบโคลน เมื่อแก้ถูก เครื่องล้างพ่นน้ำและแครอทสะอาด

---

### 10-2 แครอทสะอาดถูกส่งไปล้าง

**อาการ:** แครอทสะอาดถูกส่งเข้าเครื่องล้าง แต่แครอทเปื้อนโคลนหลุดผ่าน

**กฎที่มีบั๊ก**

```text
ถ้า [แครอทสะอาด] → [ส่งเข้าเครื่องล้าง]
```

**ประเภทบั๊ก:** Condition Bug  
**วิธีแก้**

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

**จุดสอน:** คำสั่งถูก แต่เงื่อนไขผิด

**เหตุผลที่ด่านนี้ดีกว่าเดิม:** ยังอยู่ในเกม If และย้ำว่า pass-through ของวัตถุไม่เข้าเงื่อนไขคือเรื่องปกติ

---

### 10-3 ไข่ร้าวปนถาดดี

**อาการ:** ไข่ใบใหญ่แต่ร้าวถูกส่งเข้าถาดพรีเมียม

**กฎที่มีบั๊ก**

```text
ถ้า [ไข่ใบใหญ่] → [ถาดพรีเมียม]
นอกเหนือจากนี้ถ้า [ไข่ใบเล็กและไม่ร้าว] → [ถาดมาตรฐาน]
นอกเหนือจากนี้ → [ถังคัดทิ้ง]
```

**ประเภทบั๊ก:** Broad Condition Bug  
**วิธีแก้**

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
นอกเหนือจากนี้ถ้า [ไข่ใบเล็กและไม่ร้าว] → [ถาดมาตรฐาน]
นอกเหนือจากนี้ → [ถังคัดทิ้ง]
```

**จุดสอน:** เงื่อนไขต้องละเอียดพอที่จะกันตัวหลอก

---

## 11. รายละเอียด Stage 11: โรงเรือนรวน ลำดับเพี้ยน

**จุดเน้น:** ลำดับเงื่อนไข ตัวเลข และ Else  
**ความยาก:** ปานกลาง  
**เป้าหมาย:** ให้ผู้เรียนเข้าใจว่าบั๊กไม่ได้เกิดจากบล็อกผิดตัวเดียวเท่านั้น แต่อาจเกิดจากลำดับหรือกรณีที่ตกหล่น

---

### 11-1 ฝนตกแต่ยังรดน้ำ

**อาการ:** ฝนตก แต่สปริงเกอร์ยังเปิด ทำให้แปลงน้ำท่วม

**กฎที่มีบั๊ก**

```text
ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
นอกเหนือจากนี้ → [รอดูสถานการณ์]
```

**ประเภทบั๊ก:** Order Bug  
**วิธีแก้**

```text
ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
นอกเหนือจากนี้ → [รอดูสถานการณ์]
```

**จุดสอน:** ลำดับเงื่อนไขมีผลต่อผลลัพธ์

---

### 11-2 โรงเรือนร้อนแต่ไม่เปิดพัดลม

**อาการ:** อุณหภูมิ 38°C แต่พัดลมไม่เปิด

**กฎที่มีบั๊ก**

```text
ถ้า [อุณหภูมิ > 50°C] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [อุณหภูมิ < 15°C] → [เปิดหลอดไฟ]
นอกเหนือจากนี้ → [ปิดระบบ]
```

**ประเภทบั๊ก:** Numeric Bug  
**วิธีแก้**

```text
ถ้า [อุณหภูมิ > 35°C] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [อุณหภูมิ < 15°C] → [เปิดหลอดไฟ]
นอกเหนือจากนี้ → [ปิดระบบ]
```

**จุดสอน:** ค่าตัวเลขผิดทำให้ระบบไม่เข้าเงื่อนไขที่ควรเข้า

---

### 11-3 อุณหภูมิปกติแต่ระบบค้าง

**อาการ:** อุณหภูมิ 25°C แต่ระบบไม่รู้จะทำอะไร

**กฎที่มีบั๊ก**

```text
ถ้า [อุณหภูมิ > 35°C] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [อุณหภูมิ < 15°C] → [เปิดหลอดไฟ]
```

**ประเภทบั๊ก:** Missing Else Bug  
**วิธีแก้**

```text
ถ้า [อุณหภูมิ > 35°C] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [อุณหภูมิ < 15°C] → [เปิดหลอดไฟ]
นอกเหนือจากนี้ → [ปิดระบบทั้งหมด]
```

**จุดสอน:** เมื่อไม่เข้าเงื่อนไขใดเลย ระบบต้องมี Else เพื่อรองรับกรณีปกติ

## 12. รายละเอียด Stage 12: ศูนย์ควบคุมฟาร์มฉุกเฉิน

**จุดเน้น:** Debug หลายระบบ  
**ความยาก:** สูงสุด  
**เป้าหมาย:** ให้ผู้เรียนใช้ความรู้ทั้งหมดในการวิเคราะห์หลายอาการ และแก้จุดสำคัญโดยไม่เดาสุ่ม

---

### 12-1 สัญญาณเตือนจากโรงคัดแยก

**อาการ:** กล่องพรีเมียมมีของเสีย และถังคัดทิ้งแทบไม่มีของ

**บั๊กหลัก:** เงื่อนไขกว้างเกินไป

**ตัวอย่างกฎผิด**

```text
ถ้า [ไข่ใบใหญ่] → [ถาดพรีเมียม]
```

**วิธีแก้**

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
```

**จุดสอน:** อาการหลายอย่างอาจเกิดจากสาเหตุเดียว

---

### 12-2 โรงเรือนกับสายพานเสียพร้อมกัน

**อาการ 1:** โรงเรือนร้อนแต่พัดลมไม่เปิด  
**อาการ 2:** ผลผลิตบางชิ้นถูกส่งผิดกล่อง  

**บั๊กที่ต้องแก้:**

1. Numeric Bug ในโรงเรือน  
2. Action Bug ในสายพาน  

**จุดสอน:** ระบบต่างกันอาจมีบั๊กคนละประเภท ต้องวิเคราะห์แยกกัน

---

### 12-3 วิกฤตฟาร์มก่อนส่งตลาด

**อาการรวม:**

- ไข่ร้าวเข้าถาดดี
- ฝนตกแต่สปริงเกอร์ยังเปิด
- อุณหภูมิปกติแต่ระบบค้าง

**บั๊กที่ต้องแก้:**

1. Broad Condition Bug  
2. Order Bug  
3. Missing Else Bug  

**เป้าหมาย:** แก้ 3 จุดหลักให้ฟาร์มกลับมาทำงานก่อนส่งผลผลิต

**จุดสอน:** Debugging จริงมักต้องตรวจหลายจุด แต่ต้องแก้อย่างมีเหตุผล

## 13. ระบบคะแนนและดาว

ไม่ควรเพิ่มเหรียญหรือคะแนนซับซ้อน ให้ใช้ดาวเป็นผลลัพธ์หลักเหมือนบทที่ 3

| ดาว | เงื่อนไข |
|---|---|
| 3 ดาว | เจอบั๊กถูกจุด แก้ถูกในครั้งแรก ไม่ใช้คำใบ้ |
| 2 ดาว | แก้สำเร็จ มีเลือกผิดหรือทดสอบซ้ำเล็กน้อย |
| 1 ดาว | แก้สำเร็จหลังลองหลายครั้งหรือใช้คำใบ้ |
| 0 ดาว | แก้ไม่สำเร็จ |

ตัวแปรที่ใช้คิดดาว:

- จำนวนครั้งที่เลือกบล็อกผิด
- จำนวนครั้งที่แก้ผิด
- จำนวนครั้งที่ใช้คำใบ้
- จำนวนครั้งที่ทดสอบซ้ำ
- ผ่านด่านหรือไม่

ลองตรวจเงื่อนไขในแถว If แถวแรก

เงื่อนไขควรมีคำว่า “ไม่ร้าว” เพิ่มเข้ามา

4. สร้างกฎที่ผิด
5. อธิบายอาการ
6. อธิบายวิธีแก้
7. ทดลองเล่นโจทย์บั๊ก
8. ส่งให้เพื่อนลองแก้

## 17. โครงสร้าง Engine

ควรสร้าง engine กลาง

```javascript
window.FarmMissions.smartFarmDebugger(config)
```

หน้าที่ของ engine:

1. โหลด config ด่าน
2. รันระบบที่มีบั๊กก่อนแก้
3. แสดงอาการผิด
4. แสดงบล็อกที่มีบั๊ก
5. ให้ผู้เรียนเลือกบล็อกผิด
6. ให้ผู้เรียนแก้บล็อก
7. ทดสอบระบบใหม่
8. คิดดาว
9. ส่งผลผ่าน `window.sendResult()`

## 24. สรุป

บทที่ 4 ควรเป็นเกมที่เปลี่ยนผู้เรียนจากผู้สร้างกฎในบทที่ 3 เป็นช่างซ่อมระบบฟาร์มที่ต้องจับบั๊ก แก้ไข และทดสอบใหม่

สิ่งที่ต้องยึดให้มั่นคือ

```text
ไม่ใช่แค่ตอบว่าบล็อกไหนถูก
แต่ต้องรู้ว่าอาการผิดคืออะไร
บั๊กอยู่ตรงไหน
แก้อย่างไร
และทำไมการแก้นั้นจึงถูก
```

เมื่อพัฒนาตามแผนนี้ บทที่ 4 จะไม่ซ้ำกับบทที่ 3 แต่จะต่อยอดอย่างแข็งแรง และช่วยให้ผู้เรียนเข้าใจกระบวนการ Debugging ได้อย่างเป็นรูปธรรม
