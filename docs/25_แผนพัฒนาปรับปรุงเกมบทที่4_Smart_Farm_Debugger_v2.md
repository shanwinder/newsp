# แผนการพัฒนาปรับปรุงเกมบทที่ 4
# Smart Farm Debugger v2: ภารกิจซ่อมระบบฟาร์มให้เล่นง่ายขึ้น

**บทเรียน:** บทที่ 4 การตรวจหาและแก้ไขข้อผิดพลาด  
**ชื่อบทเรียนในระบบ:** บทที่ 4: ช่างซ่อมระบบฟาร์มอัจฉริยะ  
**ชื่อเกมรวม:** Smart Farm Debugger: ภารกิจจับบั๊กกู้ฟาร์ม  
**แนวทางปรับปรุง:** ลดความซับซ้อนของเกม, ใช้ภาษาไทยง่าย, ยกระดับภาพประกอบให้เทียบเท่าบทที่ 3, แก้ปัญหาวัตถุซ้อนทับ, และเปลี่ยนแกนธีมจาก “สายพานโรงงาน” เป็น “ระบบฟาร์มเสียที่ต้องซ่อม”  
**กลุ่มเป้าหมาย:** นักเรียนชั้นประถมศึกษาปีที่ 4 โดยเฉพาะนักเรียนโรงเรียนชนบทที่ยังไม่คุ้นกับศัพท์เทคนิค  
**เป้าหมายสูงสุด:** ทำให้บทที่ 4 เป็นเกม Debugging ที่เข้าใจง่าย เล่นสนุก เห็นภาพชัด และไม่ซ้ำกับบทที่ 3

## 1. สรุปปัญหาที่ต้องแก้ไข

จากการตรวจสอบเกมบทที่ 4 พบประเด็นที่ต้องปรับปรุงอย่างจริงจัง ดังนี้

1. **ภาพประกอบและไอคอนยังดูไม่ประณีตเท่าบทที่ 3**  
   บทที่ 3 มีการออกแบบวัตถุ สายพาน และ UI ที่ดูเป็นระบบมากกว่า แต่บทที่ 4 ยังดูเหมือนใช้ไอคอนหรือสัญลักษณ์แบบชั่วคราว ทำให้คุณภาพภาพรวมไม่สม่ำเสมอกัน

2. **ภาษายังยากเกินไปสำหรับเด็ก ป.4**  
   มีคำศัพท์เชิงเทคนิค เช่น Debugging, Condition, Action, Numeric Bug, Missing Else ซึ่งเหมาะกับเอกสารพัฒนาหรือครู แต่ไม่เหมาะกับ UI สำหรับนักเรียน ป.4 โดยเฉพาะบริบทโรงเรียนชนบท

3. **การจัดวางวัตถุในกรอบเกมมีปัญหาซ้อนทับและบดบังกัน**  
   วัตถุ แผงคำสั่ง ป้ายเตือน หรือเครื่องมือบางส่วนอาจทับกัน ทำให้ผู้เล่นมองไม่ชัด และทำให้เกมดูไม่เรียบร้อย

4. **กลไกเกมยังยากเกินไปสำหรับเด็ก ป.4**  
   แนวคิด “หาบั๊กและซ่อมระบบ” ดีมาก แต่ขั้นตอนเดิมมีหลายชั้นเกินไป เช่น ดูอาการ วิเคราะห์ประเภทบั๊ก เลือกบล็อก แก้บล็อก สลับลำดับ ทดสอบ และอธิบาย ซึ่งอาจเกินระดับความพร้อมของเด็ก ป.4

5. **ธีมสายพานโรงงานไม่เหมาะกับทุกด่านของบทที่ 4**  
   บทที่ 3 ใช้สายพานได้ดีเพราะเป็นเกมคัดแยกตามเงื่อนไข แต่บทที่ 4 ควรเป็นเกมซ่อมระบบฟาร์ม ไม่ควรยึดสายพานเป็นแกนหลักทุกด่าน เพราะจะทำให้ซ้ำกับบทที่ 3 และบางด่านดูไม่เข้ากับภารกิจ Debugging

6. **มีความเสี่ยงว่าตัวเกม Debugger เองมีบั๊ก**  
   ถ้าเกมที่สอนเรื่องการหาบั๊กกลับมีบั๊กในตัวเกม เช่น คลิกแล้วไม่ตอบสนอง วัตถุซ้อนกัน หรือ feedback ไม่ตรง จะทำให้ความน่าเชื่อถือของบทเรียนลดลงมาก

## 2. เป้าหมายการปรับปรุง

การปรับปรุงบทที่ 4 เวอร์ชันนี้มีเป้าหมายหลัก 6 ด้าน

1. **ยกระดับภาพและไอคอนให้มีมาตรฐานเดียวกับบทที่ 3**  
   วัตถุ เครื่องมือ และฉากต้องดูเป็นเกมเดียวกัน ไม่ใช่คนละคุณภาพ

2. **ใช้ภาษาไทยง่ายสำหรับเด็ก ป.4**  
   ลดศัพท์เทคนิค ใช้คำสั้น เข้าใจง่าย เช่น “หาบั๊ก”, “จุดผิด”, “ซ่อม”, “ลองใหม่”, “ดูอาการซ้ำ”

3. **จัด layout ใหม่ให้ไม่ซ้อนทับกัน**  
   แยกพื้นที่ฉากฟาร์ม พื้นที่บล็อก และพื้นที่ปุ่มควบคุมให้ชัดเจน

4. **ลดความยากของกลไกเกม**  
   เริ่มจากการเลือกคำตอบ 2–3 ตัวเลือกก่อน แล้วค่อยเพิ่มความยากใน stage หลัง

5. **เปลี่ยนธีมหลักจากสายพานโรงงานเป็นระบบฟาร์มเสีย**  
   ใช้แปลงผัก โรงเรือน และแผนที่ฟาร์มอัจฉริยะเป็นฉากหลัก

6. **เพิ่ม checklist ตรวจบั๊กของเกมบทที่ 4 เอง**  
   เพื่อให้เกมที่สอนการหาบั๊กต้องทำงานเสถียรและน่าเชื่อถือ

## 3. แนวคิดหลักใหม่ของบทที่ 4

แนวคิดหลักของบทที่ 4 ควรเปลี่ยนจาก

```text
สายพานคัดแยกมีบั๊ก → แก้กฎสายพาน
```

เป็น

```text
ระบบฟาร์มทำงานผิด → แตะจุดที่เสีย → เลือกบล็อกที่ผิด → เลือกวิธีซ่อม → กดทดสอบ
```

รูปแบบการเล่นที่ควรล็อกคือ

```text
ดูภาพเสีย
→ แตะจุดที่น่าจะผิด
→ เลือกบล็อกที่ต้องซ่อม
→ เลือกบล็อกใหม่ที่ถูกต้อง
→ กด “ลองใหม่”
→ ระบบฟาร์มกลับมาทำงานถูกต้อง
```

เป้าหมายคือให้เด็กเข้าใจว่า

```text
ถ้าระบบทำงานผิด ให้ดูอาการก่อน แล้วค่อยหาจุดผิด
```

ไม่ใช่ให้เด็กต้องอ่านโค้ดยาว ๆ หรือวิเคราะห์แบบโปรแกรมเมอร์ผู้ใหญ่

## 4. ภาษาที่ใช้ในเกม

### 4.1 หลักการใช้ภาษา

ภาษาที่แสดงกับนักเรียนต้องเป็นภาษาไทยง่าย สั้น และตรงประเด็น

ควรใช้ประโยคแบบนี้

```text
เกิดอะไรขึ้น?
แครอทเปื้อนโคลนหลุดผ่านไป

น่าจะผิดตรงไหน?
ลองดูบล็อกคำสั่ง

ซ่อมอย่างไร?
เปลี่ยนเป็น “ส่งเข้าเครื่องล้าง”
```

ไม่ควรใช้ประโยคยาวหรือศัพท์เทคนิค เช่น

```text
ระบบมีข้อผิดพลาดจากการประเมินเงื่อนไขที่ไม่สอดคล้องกับผลลัพธ์ของโปรแกรม
```

### 4.2 คำที่ควรใช้แทนศัพท์เทคนิค

| คำเชิงเทคนิค | คำที่ใช้กับเด็ก |
|---|---|
| Debugging | หาบั๊ก / ซ่อมจุดผิด |
| Bug | บั๊ก / จุดผิด |
| Condition | เงื่อนไข |
| Action | คำสั่ง |
| Run | เริ่มระบบ |
| Retest | ลองใหม่ / ทดสอบอีกครั้ง |
| Numeric Bug | ตัวเลขผิด |
| Missing Else | ลืมกรณีอื่น |
| Broad Condition | เงื่อนไขกว้างเกินไป |
| Pass-through | ปล่อยผ่าน |
| System Error | ระบบรวน |
| Fix | ซ่อม |

### 4.3 คำที่ใช้ใน UI

ปุ่มในเกมควรใช้คำต่อไปนี้

```text
ดูอาการซ้ำ
หาบั๊ก
เลือกจุดผิด
ซ่อมบล็อก
ลองใหม่
ขอคำใบ้
ผ่านแล้ว
กลับไปเลือกด่าน
```

ไม่ควรใช้คำว่า Diagnose, Repair, Execute, Retest, Condition Bug หรือ Action Bug บนหน้าจอของนักเรียนโดยตรง

## 5. มาตรฐานภาพและ Asset

### 5.1 หลักการ

```javascript

    height: 96,
    description: 'ภาพแครอทเปื้อนโคลน ใช้ในด่านซ่อมระบบล้างผัก',
    futureSpriteSheet: {
      path: '../assets/img/debug/vegetable/dirty_carrot_sheet.png',
      frameWidth: 96,
      frameHeight: 96,
      animations: ['idle', 'wrong', 'fixed']
    }
  }
}
```

- ถ้ามี asset PNG ให้ใช้ภาพ PNG ก่อน

- ต้องรองรับ animation ในอนาคตผ่าน sprite sheet

## 7. กลไกเกมใหม่ที่ง่ายขึ้น

### 7.1 Flow ใหม่

ใช้ flow เดียวกันทั้งบท แต่เพิ่มความยากทีละน้อย

```text
ดูอาการเสีย
→ เลือกจุดผิด
→ เลือกบล็อกที่ควรแก้
→ กดลองใหม่
→ ดูผลลัพธ์
```

### 7.2 Stage 10: เลือกจากตัวเลือก

เด็กยังไม่ต้องลากแก้กฎเองเต็มรูปแบบ ให้เลือกจากตัวเลือก 2–3 ตัว

ตัวอย่าง

```text
แครอทเปื้อนโคลนหลุดผ่านไป
น่าจะผิดตรงไหน?

[เงื่อนไข] [คำสั่ง]
```

จากนั้นเลือกบล็อกแก้

```text
ควรเปลี่ยนคำสั่งเป็นอะไร?

[ปล่อยผ่าน] [ส่งเข้าเครื่องล้าง]
```

### 7.3 Stage 11: เริ่มซ่อมบล็อก

เด็กเริ่มลากหรือเลือกบล็อกที่ถูกไปแทน เช่น

```text
อุณหภูมิ 38°C แต่พัดลมไม่เปิด
ควรแก้ตัวเลขเป็นอะไร?

[มากกว่า 50°C] [มากกว่า 35°C] [น้อยกว่า 15°C]
```

### 7.4 Stage 12: แตะจุดเสียบนแผนที่ฟาร์ม

เด็กเห็นแผนที่ฟาร์ม มีจุดเสีย 2–3 จุด แล้วต้องแตะจุดที่เสียเพื่อซ่อมทีละจุด

ตัวอย่าง

```text
ฟาร์มมีระบบรวน 2 จุด
แตะจุดที่มีไฟแดงเพื่อซ่อม
```

### 7.5 จำกัดจำนวนสิ่งที่ต้องคิด

สำหรับเด็ก ป.4 แต่ละด่านไม่ควรมีตัวเลือกมากเกินไป

| Stage | จำนวนตัวเลือกที่เหมาะสม |
|---|---:|
| Stage 10 | 2 ตัวเลือก |
| Stage 11 | 3 ตัวเลือก |
| Stage 12 | 3 จุดเสีย หรือ 3 ตัวเลือกต่อจุด |

## 8. โครงสร้าง Stage ใหม่

### 8.1 Stage 10: ซ่อมระบบผักสวนครัว

**ชื่อในระบบ:** ซ่อมระบบผักสวนครัว  
**ธีม:** แปลงผัก / เครื่องล้าง / โต๊ะตรวจผัก  
**รูปแบบเล่น:** ดูอาการ → เลือกจุดผิด → เลือกบล็อกแก้  
**จุดเน้น:** คำสั่งผิด / เงื่อนไขผิด / เข้าใจ If แบบปล่อยผ่าน

#### ด่าน 10-1: แครอทเปื้อนโคลนไม่ถูกล้าง

**อาการในเกม:**

```text
แครอทเปื้อนโคลนหลุดผ่านไป
```

**บล็อกผิด:**

```text
ถ้า [แครอทเปื้อนโคลน] → [ปล่อยผ่าน]
```

**บล็อกที่ถูก:**

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

**กลไกสำหรับเด็ก:**

1. ดูแครอทเลอะหลุดผ่าน
2. เลือกว่า “คำสั่ง” เป็นจุดผิด
3. เลือกคำสั่งใหม่ “ส่งเข้าเครื่องล้าง”
4. กด “ลองใหม่”

**ข้อความง่าย:**

```text
แครอทเลอะต้องไปล้าง
แต่ระบบปล่อยผ่านไป
ซ่อมคำสั่งให้ถูกนะ
```

#### ด่าน 10-2: ผักกาดมีหนอนหลุดผ่าน

**อาการในเกม:**

```text
ผักกาดมีหนอนหลุดผ่านโต๊ะตรวจ
```

**บล็อกผิด:**

```text
ถ้า [ผักกาดใบแหว่ง] → [ส่งไปโต๊ะตรวจ]
```

**บล็อกที่ถูก:**

```text
ถ้า [ผักกาดมีหนอน] → [ส่งไปโต๊ะตรวจ]
```

**กลไกสำหรับเด็ก:**

1. ดูผักกาดมีหนอนหลุดผ่าน
2. เลือกว่า “เงื่อนไข” เป็นจุดผิด
3. เลือกเงื่อนไขใหม่ “ผักกาดมีหนอน”
4. กด “ลองใหม่”

**ข้อความง่าย:**

```text
ต้องจับผักที่มีหนอน
ไม่ใช่ผักใบแหว่งทุกใบ
```

#### ด่าน 10-3: แครอทมีตำหนิถูกส่งไปล้าง

**อาการในเกม:**

```text
แครอทมีตำหนิถูกส่งเข้าเครื่องล้าง
ทั้งที่ไม่ได้เปื้อนโคลน
```

**บล็อกผิด:**

```text
ถ้า [แครอทมีตำหนิ] → [ส่งเข้าเครื่องล้าง]
```

**บล็อกที่ถูก:**

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
```

**จุดสอน:**

ด่านนี้เน้นว่าเกม If ต้องจับเฉพาะเงื่อนไข ไม่ใช่จับทุกสิ่งที่ดูผิดปกติ  
ถ้าไม่เข้าเงื่อนไข ให้ปล่อยผ่าน

**ข้อความง่าย:**

```text
มีตำหนิไม่ใช่เปื้อนโคลน
ด่านนี้ล้างเฉพาะแครอทเปื้อนโคลนเท่านั้น
```

### 8.2 Stage 11: ซ่อมโรงเรือนอัจฉริยะ

**ชื่อในระบบ:** ซ่อมโรงเรือนอัจฉริยะ  
**ธีม:** โรงเรือน / ฝน / ความชื้น / อุณหภูมิ  
**รูปแบบเล่น:** ดูเซ็นเซอร์ → เลือกบล็อกผิด → เลือกวิธีซ่อม  
**จุดเน้น:** ลำดับ / ตัวเลข / กรณีอื่น

#### ด่าน 11-1: ฝนตกแต่สปริงเกอร์ยังเปิด

**อาการในเกม:**

```text
ฝนตกแล้ว แต่สปริงเกอร์ยังเปิด
แปลงผักน้ำท่วม
```

**บล็อกผิด:**

```text
ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
```

**บล็อกที่ถูก:**

```text
ถ้า [ฝนตก] → [ปิดสปริงเกอร์]
นอกเหนือจากนี้ถ้า [ดินแห้ง] → [เปิดสปริงเกอร์]
```

**กลไกสำหรับเด็ก:**

เด็กไม่ต้องเข้าใจคำว่า “ลำดับเงื่อนไข” ในทันที ให้ใช้ภาษาว่า

```text
ควรดูอะไรก่อน?
[ดินแห้ง] [ฝนตก]
```

**ข้อความง่าย:**

```text
ถ้าฝนตก ต้องหยุดรดน้ำก่อน
ไม่อย่างนั้นน้ำจะท่วมแปลง
```

#### ด่าน 11-2: โรงเรือนร้อนแต่พัดลมไม่เปิด

**อาการในเกม:**

```text
อุณหภูมิ 38°C
แต่พัดลมไม่เปิด
```

**บล็อกผิด:**

```text
ถ้า [อุณหภูมิสูงกว่า 50°C] → [เปิดพัดลม]
```

**บล็อกที่ถูก:**

```text
ถ้า [อุณหภูมิสูงกว่า 35°C] → [เปิดพัดลม]
```

**กลไกสำหรับเด็ก:**

```text
ตัวเลขไหนถูก?
[50°C] [35°C]
```

**ข้อความง่าย:**

```text
38°C ร้อนเกินแล้ว
ระบบควรเปิดพัดลมตั้งแต่เกิน 35°C
```

#### ด่าน 11-3: อากาศปกติแต่ระบบค้าง

**อาการในเกม:**

```text
อุณหภูมิ 25°C
ระบบไม่รู้ว่าจะทำอะไร
```

**บล็อกผิด:**

```text
ถ้า [ร้อนมาก] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [หนาวมาก] → [เปิดหลอดไฟ]
```

**บล็อกที่ถูก:**

```text
ถ้า [ร้อนมาก] → [เปิดพัดลม]
นอกเหนือจากนี้ถ้า [หนาวมาก] → [เปิดหลอดไฟ]
นอกเหนือจากนี้ → [ปิดระบบ]
```

**กลไกสำหรับเด็ก:**

```text
ถ้าอากาศปกติ ควรทำอะไร?
[เปิดพัดลม] [เปิดหลอดไฟ] [ปิดระบบ]
```

**ข้อความง่าย:**

```text
อากาศพอดีแล้ว
ไม่ต้องเปิดอะไรเพิ่ม
ให้ปิดระบบเพื่อประหยัดไฟ
```

### 8.3 Stage 12: ศูนย์ซ่อมฟาร์มฉุกเฉิน

**ชื่อในระบบ:** ศูนย์ซ่อมฟาร์มฉุกเฉิน  
**ธีม:** แผนที่ฟาร์ม มีจุดเสียให้แตะ  
**รูปแบบเล่น:** แตะจุดเสีย → ซ่อมทีละระบบ → ฟาร์มกลับมาปกติ  
**จุดเน้น:** รวมความรู้แบบง่าย ไม่ซับซ้อนเกินไป

#### ด่าน 12-1: ไฟแดงที่แปลงผัก

**อาการ:**

มีไฟแดง 2 จุด แต่เสียจริง 1 จุด  
เด็กต้องแตะจุดที่ผิดจากภาพ

**จุดเสียจริง:**

แครอทเปื้อนโคลนหลุดผ่านเครื่องล้าง

**ตัวหลอก:**

เครื่องล้างมีเสียงดัง แต่ยังทำงานปกติ

**เป้าหมาย:**

เด็กต้องดูอาการ ไม่ใช่แตะตามไฟอย่างเดียว

#### ด่าน 12-2: โรงเรือนกับแปลงผักรวนพร้อมกัน

**อาการ:**

1. ฝนตกแต่สปริงเกอร์เปิด  
2. ผักกาดมีหนอนหลุดผ่านโต๊ะตรวจ  

**เป้าหมาย:**

ซ่อม 2 จุด โดยแต่ละจุดใช้กลไกง่าย ๆ แบบ Stage 10 และ 11

**จำกัดจำนวนครั้ง:**

ซ่อมผิดได้ไม่เกิน 2 ครั้ง หากเกินยังเล่นต่อได้ แต่ได้ดาวลดลง

#### ด่าน 12-3: ฟาร์มต้องพร้อมก่อนส่งผลผลิต

**อาการ:**

1. แครอทมีตำหนิถูกส่งไปล้างผิด  
2. โรงเรือนร้อนแต่พัดลมไม่เปิด  
3. อากาศปกติแต่ระบบค้าง  

**เป้าหมาย:**

ซ่อม 3 จุดก่อนจบเกม

**ข้อความง่าย:**

```text
ช่วยซ่อมฟาร์มให้พร้อมส่งผลผลิต
แตะจุดที่มีปัญหา แล้วเลือกวิธีซ่อม
```

```text
แครอทเปื้อนโคลนควรถูกส่งเข้าเครื่องล้าง
```

## 11. ระบบดาว

ใช้ดาวเป็นคะแนนหลักเหมือนบทที่ 3

| ดาว | เงื่อนไข |
|---|---|
| 3 ดาว | ซ่อมถูกครั้งแรก ไม่ใช้คำใบ้ |
| 2 ดาว | ซ่อมสำเร็จ แต่เลือกผิดเล็กน้อยหรือใช้คำใบ้ 1 ครั้ง |
| 1 ดาว | ซ่อมสำเร็จหลังลองหลายครั้ง |
| 0 ดาว | ยังซ่อมไม่สำเร็จ |

ไม่ต้องมีเหรียญ คอมโบ หรือคะแนนตัวเลข เพื่อไม่ให้ UI รกเกินไป

```javascript
window.FarmMissions.smartFarmDebuggerLite(config)
```

ระบบสร้างชิ้นงานยังควรใช้แนว “สร้างโจทย์บั๊กฟาร์มของฉัน” แต่ต้องทำให้ง่ายกว่าเดิม

### 16.1 Flow สำหรับเด็ก

```text
เลือกฉาก
→ เลือกอาการเสีย
→ เลือกจุดผิด
→ เลือกคำตอบที่ถูก
→ ทดลองเล่น
→ ส่งให้เพื่อนลองซ่อม
```

ไม่ควรให้เด็กสร้าง rules ซับซ้อนเองตั้งแต่ต้น

### 16.2 ตัวอย่างชิ้นงาน

```text
ชื่อโจทย์: แครอทเลอะไม่ถูกล้าง

อาการเสีย:
แครอทเปื้อนโคลนหลุดผ่านเครื่องล้าง

จุดผิด:
คำสั่ง

คำตอบที่ถูก:
ส่งเข้าเครื่องล้าง
```

```json

  "project_type": "smart_farm_debug_lite_challenge",

  "theme": "vegetable_repair",
  "problemText": "แครอทเปื้อนโคลนหลุดผ่านเครื่องล้าง",
  "bugTarget": "action",
  "correctFix": "wash",
  "choices": {
    "bugTargetChoices": ["condition", "action"],
    "fixChoices": ["pass", "wash"]
  },
  "tested": true
}
```

## 21. สรุป

แนวคิดของบทที่ 4 คือ “หาบั๊กและซ่อมระบบฟาร์ม” ยังเป็นแนวคิดที่ดี แต่ต้องปรับการนำเสนอให้เหมาะกับเด็ก ป.4 มากขึ้น

สิ่งที่ต้องเปลี่ยนสำคัญคือ

1. **ไม่ยึดสายพานโรงงานเป็นแกนหลัก**  
   ให้เปลี่ยนเป็นระบบฟาร์มเสีย เช่น แปลงผัก โรงเรือน และแผนที่ฟาร์ม

2. **เล่นง่ายขึ้น**  
   เด็กควรทำเพียง ดูอาการ เลือกจุดผิด เลือกวิธีซ่อม แล้วลองใหม่

3. **ภาษาไทยง่ายขึ้น**  
   ใช้คำว่า หาบั๊ก จุดผิด ซ่อม ลองใหม่ ดูอาการซ้ำ

4. **ภาพต้องดีเท่าบทที่ 3**  
   ทุกวัตถุต้องมี asset structure พร้อมเปลี่ยนเป็นภาพจริงในอนาคต

5. **layout ต้องไม่ซ้อนทับกัน**  
   ใช้ grid, anchor point, z-index และ safe area

เมื่อปรับตามแผนนี้ บทที่ 4 จะกลายเป็นเกม Debugging ที่เข้าใจง่าย สนุกขึ้น เหมาะกับเด็ก ป.4 และต่อยอดจากบทที่ 3 ได้อย่างมีคุณภาพ
