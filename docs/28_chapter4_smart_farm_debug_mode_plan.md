# แผนการพัฒนาปรับปรุงเกมบทที่ 4
## Smart Farm Debug Mode: ซ่อมกฎฟาร์มอัจฉริยะ
### แนวทางใหม่: ยกเกมบทที่ 3 มาใช้เป็นฐาน แล้วปรับเป็นโหมดตรวจสอบและแก้ไขข้อผิดพลาด

เอกสารนี้จัดทำขึ้นเพื่อใช้เป็นแผนพัฒนาปรับปรุง **บทที่ 4: การตรวจสอบและแก้ไขข้อผิดพลาด** ของระบบแบบฝึกทักษะออนไลน์ โดยเปลี่ยนแนวทางจากการสร้างเกมบทที่ 4 แยกใหม่ เป็นการนำเกมสายพาน/โรงคัดแยกของ **บทที่ 3** มาใช้เป็นฐานทั้งระบบ แล้วปรับรูปแบบการเล่นจาก **Build Mode: สร้างกฎ** เป็น **Debug Mode: ตรวจสอบและซ่อมกฎที่ผิด**

แนวคิดสำคัญคือ ผู้เรียนไม่ต้องเรียนรู้ UI ใหม่ แต่ใช้รูปแบบเกมที่คุ้นเคยจากบทที่ 3 แล้วเพิ่มภารกิจใหม่ คือการตรวจสอบระบบที่มีข้อผิดพลาด ทดลองรันระบบเดิม สังเกตผลลัพธ์ที่ผิด วิเคราะห์ว่าบั๊กอยู่ตรงไหน แล้วแก้กฎให้ระบบทำงานถูกต้อง

---

# 1. สรุปแนวทางใหม่

## 1.1 แนวคิดหลัก

บทที่ 3 และบทที่ 4 ควรใช้เกมแกนเดียวกัน แต่เป้าหมายการคิดต่างกันอย่างชัดเจน

```text
บทที่ 3 = สร้างกฎให้ระบบทำงานถูกต้อง
บทที่ 4 = ตรวจสอบกฎที่ผิด แล้วแก้ไขให้ถูกต้อง
```

ดังนั้น บทที่ 4 ไม่ควรสร้างเกมใหม่ที่แยกออกจากบทที่ 3 แต่ควรใช้ engine, layout, assets, item queue, machines, rule blocks, expected logic และ scoring ของบทที่ 3 เป็นฐาน แล้วเพิ่มความสามารถเฉพาะของการ debug เช่น

- มีกฎที่ผิดถูกวางไว้ล่วงหน้า
- ให้ผู้เรียนกดรันทดสอบกฎเดิมก่อน
- ระบบแสดงรายการที่ถูกคัดแยกผิด
- ระบบแสดงรายงานข้อผิดพลาด
- ผู้เรียนแก้บล็อกที่ผิด
- ผู้เรียนกดทดสอบซ้ำ
- ผ่านด่านเมื่อผลผลิตทุกชิ้นถูกส่งไปยังปลายทางที่ถูกต้อง

---

## 1.2 ชื่อเกมบทที่ 4 ที่เสนอ

**Smart Farm Debug Mode: ซ่อมกฎฟาร์มอัจฉริยะ**

ชื่อภาษาไทยที่แสดงกับผู้เรียน:

```text
บทที่ 4: ช่างซ่อมกฎฟาร์มอัจฉริยะ
```

คำอธิบายสั้นสำหรับหน้าเลือกด่าน:

```text
ระบบฟาร์มอัจฉริยะทำงานผิดพลาด เพราะกฎบางข้อถูกตั้งไว้ไม่ถูกต้อง
มาช่วยกันตรวจสอบ ทดลอง และซ่อมกฎให้ฟาร์มกลับมาทำงานถูกต้องกันเถอะ
```

# 2. เหตุผลที่ควรยกเกมบทที่ 3 มาใช้ในบทที่ 4

## 2.1 เกมบทที่ 3 เหมาะกับการ debug อยู่แล้ว

เกมบทที่ 3 มีองค์ประกอบที่สอดคล้องกับการตรวจสอบและแก้ไขข้อผิดพลาดครบถ้วน ได้แก่

1. มีวัตถุหรือผลผลิตไหลผ่านสายพาน
2. มีเครื่องสแกนหรือตัวตรวจสอบคุณสมบัติ
3. มีบล็อกเงื่อนไขและคำสั่ง
4. มีปลายทางของผลผลิตแต่ละประเภท
5. มี expected logic หรือผลลัพธ์ที่ควรถูกต้อง
6. มีการประเมินว่าผลผลิตถูกส่งถูกทางหรือไม่
7. มี feedback และคะแนนจากผลการทำงานของระบบ

องค์ประกอบเหล่านี้เหมาะกับบทที่ 4 มาก เพราะการ debug ต้องเริ่มจากการเห็นผลลัพธ์ที่ผิดก่อน แล้วจึงย้อนกลับไปตรวจว่ากฎหรือเงื่อนไขใดทำให้เกิดความผิดพลาด

---

## 2.2 ผู้เรียนไม่ต้องเรียนรู้เกมใหม่

ถ้าบทที่ 4 ใช้ UI ใหม่ทั้งหมด ผู้เรียนต้องเสียเวลาเรียนรู้วิธีเล่นใหม่ แต่ถ้ายกบทที่ 3 มาใช้ ผู้เรียนจะคุ้นเคยกับ

- สายพาน
- วัตถุที่ไหลผ่าน
- เครื่องคัดแยก
- กฎ IF / ELSE
- บล็อกคำสั่ง
- ปลายทางของผลผลิต
- ปุ่มเริ่มการทำงาน

บทที่ 4 จึงสามารถเปลี่ยนจุดเน้นจาก “วิธีเล่น” ไปเป็น “วิธีคิดตรวจสอบข้อผิดพลาด” ได้ทันที

---

## 2.3 ลดปัญหา UI และวัตถุทับซ้อน

เกมบทที่ 4 ที่ทำแยกออกมาก่อนหน้านี้พบปัญหาเรื่องกรอบเกม วัตถุ และบล็อกคำสั่งทับซ้อนกัน แต่เกมบทที่ 3 มี layout ที่เป็นระบบกว่าอยู่แล้ว การใช้ฐานจากบทที่ 3 จะช่วยลดปัญหาเหล่านี้ เพราะมีโครงสร้างพื้นที่ที่ชัดเจน เช่น

- พื้นที่สายพาน
- พื้นที่โปรแกรม/กฎ
- พื้นที่เครื่องปลายทาง
- พื้นที่ feedback
- พื้นที่คะแนน

---

## 2.4 บทที่ 3 และบทที่ 4 จะต่อเนื่องกันทางการเรียนรู้

บทที่ 3 สอนให้ผู้เรียนสร้างกฎ ส่วนบทที่ 4 จะสอนให้ผู้เรียนตรวจสอบและแก้ไขกฎ ความต่อเนื่องนี้ช่วยให้ลำดับการเรียนรู้สมเหตุสมผลมากขึ้น

```text
บทที่ 3: นักเรียนเรียนรู้ว่า “จะสร้างกฎอย่างไร”
บทที่ 4: นักเรียนเรียนรู้ว่า “ถ้ากฎผิด จะตรวจและแก้อย่างไร”
```

# 3. เป้าหมายการเรียนรู้ของบทที่ 4

หลังเรียนด้วยเกมบทที่ 4 ผู้เรียนควรสามารถ

1. รันทดสอบระบบที่มีกฎผิดพลาดได้
2. สังเกตผลลัพธ์ที่ไม่ตรงกับเป้าหมายได้
3. ระบุได้ว่าข้อผิดพลาดเกิดจากเงื่อนไขผิด คำสั่งผิด ลำดับผิด หรือเงื่อนไขกว้าง/แคบเกินไป
4. แก้ไขบล็อกคำสั่งหรือเงื่อนไขให้ถูกต้องได้
5. ทดสอบซ้ำหลังแก้ไขได้
6. อธิบายได้ว่าการแก้ไขทำให้ผลลัพธ์ถูกต้องอย่างไร
7. เข้าใจความสัมพันธ์ระหว่างกฎ เงื่อนไข คำสั่ง และผลลัพธ์

# 4. เปรียบเทียบ Build Mode และ Debug Mode

| ประเด็น | บทที่ 3: Build Mode | บทที่ 4: Debug Mode |
|---|---|---|
| จุดเริ่มต้น | กระดานกฎว่าง | มีกฎผิดวางไว้แล้ว |
| งานของผู้เรียน | สร้างกฎให้ถูก | ตรวจและแก้กฎที่ผิด |
| การเรียนรู้หลัก | เข้าใจโครงสร้างเงื่อนไข | เข้าใจการตรวจหาข้อผิดพลาด |
| ปุ่มหลัก | เริ่มสายพาน | ทดสอบกฎเดิม / ทดสอบหลังแก้ไข |
| Feedback | ถูก/ผิดจากการสร้างกฎ | รายงานข้อผิดพลาดและจุดน่าสงสัย |
| ความรู้ที่เน้น | IF, ELSE, ELSE IF | Debugging, Testing, Fixing |
| ภาพรวมบทบาท | ผู้จัดการฟาร์ม | ช่างซ่อมระบบฟาร์ม |

# 5. Core Gameplay Loop ของบทที่ 4

## 5.1 Flow การเล่นต่อด่าน

```text
1. อ่านภารกิจ
2. ดูกฎเดิมที่ระบบตั้งไว้ผิด
3. กด “ทดสอบกฎเดิม”
4. สายพานเริ่มทำงาน
5. ระบบแสดงผลผลิตที่ถูกส่งผิดทาง
6. เปิด Debug Report
7. ผู้เรียนเลือกหรือแก้บล็อกที่น่าสงสัย
8. กด “ทดสอบหลังแก้ไข”
9. ถ้าถูกต้อง ระบบแสดงผลสำเร็จและให้ดาว
10. ถ้ายังผิด ระบบให้ feedback และเปิดโอกาสให้แก้ใหม่
```

---

## 5.2 Flow แบบย่อสำหรับเด็ก

ข้อความที่แสดงในเกมควรเรียบง่าย เช่น

```text
1. ลองเปิดระบบเดิม
2. ดูว่าผลผลิตชิ้นไหนไปผิดทาง
3. หาบล็อกที่ทำให้ระบบผิด
4. ซ่อมกฎ
5. ลองเปิดระบบอีกครั้ง
```

# 6. โครงสร้างบทที่ 4 ใหม่

บทที่ 4 มี 3 stage หลัก โดยยกเนื้อหาจากบทที่ 3 มาใช้เป็นฐาน

| Stage ใหม่ | ฐานจากบทที่ 3 | แนวคิด | โครงสร้าง |
|---|---|---|---|
| Stage 10 | Stage 7 | ซ่อมกฎ IF | IF |
| Stage 11 | Stage 8 | ซ่อมกฎ IF / ELSE | IF / ELSE |
| Stage 12 | Stage 9 | ซ่อมกฎ IF / ELSE IF / ELSE | IF / ELSE IF / ELSE |

แต่ละ stage มี 3 ด่านย่อย รวมทั้งหมด 9 ด่าน

# 7. Stage 10: ซ่อมกฎ IF

## 7.1 แนวคิดของ Stage 10

Stage 10 ใช้ฐานจาก Stage 7 ของบทที่ 3 ซึ่งเป็นด่าน IF แบบพื้นฐาน ผู้เรียนจะเจอกฎ IF ที่ระบบตั้งผิดไว้ แล้วต้องแก้เงื่อนไขหรือคำสั่งให้ถูกต้อง

โครงสร้างที่ใช้:

```text
ถ้า [เงื่อนไข] → [คำสั่ง]
```

เป้าหมายคือให้ผู้เรียนเข้าใจว่า กฎ IF ต้องตรวจเงื่อนไขที่ถูกต้องและส่งไปยังปลายทางที่เหมาะสม

---

## 7.2 ด่าน 10-1: แครอทเปื้อนโคลน

### ชื่อด่าน

```text
10-1 ซ่อมกฎล้างแครอทเปื้อนโคลน
```

### สถานการณ์

ระบบล้างแครอททำงานผิดพลาด แครอทบางลูกที่ไม่ได้เปื้อนโคลนถูกส่งไปล้าง ส่วนแครอทที่เปื้อนโคลนจริงบางลูกอาจหลุดผ่านระบบ

### กฎเสียเริ่มต้น

```text
ถ้า แครอทสีเข้ม → ส่งเข้าเครื่องล้าง
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
ระบบดูจาก “สีเข้ม” แทนที่จะดูว่า “เปื้อนโคลน” หรือไม่
จึงอาจส่งแครอทสะอาดบางลูกไปล้างผิด และจับแครอทเปื้อนโคลนได้ไม่แม่นยำ
```

### กฎที่ถูกต้อง

```text
ถ้า แครอทเปื้อนโคลน → ส่งเข้าเครื่องล้าง
```

### ประเภทบั๊ก

```text
wrong_condition
```

### สิ่งที่ผู้เรียนต้องทำ

แก้เงื่อนไขจาก “แครอทสีเข้ม” เป็น “แครอทเปื้อนโคลน”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! ระบบควรตรวจว่าแครอทเปื้อนโคลนหรือไม่ ไม่ใช่ดูจากสีเข้มอย่างเดียว
```

### ความรู้ที่ได้

เงื่อนไขที่ดูคล้ายกันอาจทำให้ระบบตัดสินใจผิด ต้องเลือกเงื่อนไขที่ตรงกับปัญหาจริง

---

## 7.3 ด่าน 10-2: ผักกาดมีหนอน

### ชื่อด่าน

```text
10-2 ซ่อมกฎตรวจผักกาดมีหนอน
```

### สถานการณ์

ระบบตรวจศัตรูพืชของผักกาดทำงานผิดพลาด เพราะตรวจจากรอยบนใบแทนที่จะตรวจว่ามีหนอนจริงหรือไม่

### กฎเสียเริ่มต้น

```text
ถ้า ผักกาดมีรูเล็ก ๆ → ส่งไปโต๊ะตรวจศัตรูพืช
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
ผักกาดบางใบมีรูเล็ก ๆ แต่ไม่มีหนอนก็ถูกส่งไปตรวจ
ขณะเดียวกัน ระบบยังไม่แม่นพอในการจับผักกาดที่มีหนอนจริง
```

### กฎที่ถูกต้อง

```text
ถ้า ผักกาดมีหนอน → ส่งไปโต๊ะตรวจศัตรูพืช
```

### ประเภทบั๊ก

```text
too_broad_condition
```

### สิ่งที่ผู้เรียนต้องทำ

แก้เงื่อนไขให้ตรวจ “มีหนอน” โดยตรง ไม่ใช่ตรวจเพียงรอยรูบนใบ

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! รอยบนใบอาจเกิดจากหลายสาเหตุ แต่สิ่งที่ต้องตรวจจริงคือมีหนอนหรือไม่
```

### ความรู้ที่ได้

เงื่อนไขที่กว้างเกินไปทำให้ระบบจับของผิดกลุ่ม

---

## 7.4 ด่าน 10-3: มันฝรั่งมีหน่องอก

### ชื่อด่าน

```text
10-3 ซ่อมกฎคัดแยกมันฝรั่งงอก
```

### สถานการณ์

ระบบคัดแยกมันฝรั่งทำงานผิด เพราะเข้าใจว่ามันฝรั่งเปื้อนดินคือมันฝรั่งที่ต้องคัดออก ทั้งที่ปัญหาจริงคือมันฝรั่งมีหน่องอก

### กฎเสียเริ่มต้น

```text
ถ้า มันฝรั่งเปื้อนดิน → ส่งไปคัดแยกพิเศษ
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
มันฝรั่งเปื้อนดินถูกคัดแยก ทั้งที่อาจล้างได้
แต่มันฝรั่งมีหน่องอกซึ่งควรถูกคัดแยกจริง อาจไม่ถูกตรวจพบ
```

### กฎที่ถูกต้อง

```text
ถ้า มันฝรั่งมีหน่องอก → ส่งไปคัดแยกพิเศษ
```

### ประเภทบั๊ก

```text
wrong_condition
```

### สิ่งที่ผู้เรียนต้องทำ

เปลี่ยนเงื่อนไขจาก “เปื้อนดิน” เป็น “มีหน่องอก”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! มันฝรั่งเปื้อนดินยังล้างได้ แต่ถ้ามีหน่องอกต้องคัดแยกพิเศษ
```

### ความรู้ที่ได้

ต้องแยกแยะอาการที่คล้ายกัน และเลือกเงื่อนไขที่ตรงกับปัญหาจริง

# 8. Stage 11: ซ่อมกฎ IF / ELSE

## 8.1 แนวคิดของ Stage 11

Stage 11 ใช้ฐานจาก Stage 8 ของบทที่ 3 ซึ่งเป็นด่าน IF / ELSE ผู้เรียนต้องแก้กฎที่มี 2 ทางเลือก คือกรณีที่เข้าเงื่อนไข และกรณีที่ไม่เข้าเงื่อนไข

โครงสร้างที่ใช้:

```text
ถ้า [เงื่อนไข] → [คำสั่งที่ 1]
นอกเหนือจากนี้ → [คำสั่งที่ 2]
```

เป้าหมายคือให้ผู้เรียนเข้าใจว่า ELSE ต้องส่งสิ่งที่ไม่เข้า IF ไปยังปลายทางที่ถูกต้อง ไม่ใช่ปล่อยให้ทำงานผิดเหมือน IF

---

## 8.2 ด่าน 11-1: ส้มลูกใหญ่และส้มลูกเล็ก

### ชื่อด่าน

```text
11-1 ซ่อมกฎคัดส้มพรีเมียม
```

### สถานการณ์

โรงคัดส้มต้องส่งส้มลูกใหญ่ไปกล่องพรีเมียม และส่งส้มที่ไม่ใช่ลูกใหญ่ไปกล่องแปรรูปน้ำส้ม แต่ระบบตั้งค่า ELSE ผิด

### กฎเสียเริ่มต้น

```text
ถ้า ส้มลูกใหญ่ → กล่องพรีเมียม
นอกเหนือจากนี้ → กล่องพรีเมียม
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
ส้มลูกเล็กก็ถูกส่งเข้ากล่องพรีเมียม ทำให้กล่องพรีเมียมปนของผิดเกณฑ์
```

### กฎที่ถูกต้อง

```text
ถ้า ส้มลูกใหญ่ → กล่องพรีเมียม
นอกเหนือจากนี้ → กล่องแปรรูปน้ำส้ม
```

### ประเภทบั๊ก

```text
wrong_else_action
```

### สิ่งที่ผู้เรียนต้องทำ

แก้ปลายทางของ ELSE จาก “กล่องพรีเมียม” เป็น “กล่องแปรรูปน้ำส้ม”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! ส้มที่ไม่ใช่ลูกใหญ่ไม่ควรเข้ากล่องพรีเมียม แต่ควรส่งไปแปรรูป
```

### ความรู้ที่ได้

ELSE คือทางเลือกสำหรับสิ่งที่ไม่เข้าเงื่อนไข IF

---

## 8.3 ด่าน 11-2: กล้วยสุกและกล้วยดิบ

### ชื่อด่าน

```text
11-2 ซ่อมกฎคัดกล้วยสุก
```

### สถานการณ์

ระบบคัดกล้วยส่งขายหน้าร้านทำงานผิด เพราะเงื่อนไขกว้างเกินไป ทำให้กล้วยที่ยังไม่สุกเต็มที่ถูกส่งไปขาย

### กฎเสียเริ่มต้น

```text
ถ้า กล้วยมีสีเหลืองบางส่วน → ส่งไปขายหน้าร้าน
นอกเหนือจากนี้ → ส่งไปห้องบ่มต่อ
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
กล้วยเหลืองอมเขียวหรือกล้วยที่ปลายยังเขียวถูกส่งไปขาย ทั้งที่ยังควรบ่มต่อ
```

### กฎที่ถูกต้อง

```text
ถ้า กล้วยสุกสีเหลือง → ส่งไปขายหน้าร้าน
นอกเหนือจากนี้ → ส่งไปห้องบ่มต่อ
```

### ประเภทบั๊ก

```text
too_broad_condition
```

### สิ่งที่ผู้เรียนต้องทำ

แก้เงื่อนไขให้เฉพาะเจาะจงขึ้น จาก “มีสีเหลืองบางส่วน” เป็น “สุกสีเหลือง”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! กล้วยที่ยังเขียวบางส่วนควรถูกบ่มต่อ ไม่ควรส่งขายทันที
```

### ความรู้ที่ได้

เงื่อนไขที่กว้างเกินไปอาจทำให้ระบบเลือกของที่ยังไม่ผ่านเกณฑ์

---

## 8.4 ด่าน 11-3: แตงโมเกรดดี

### ชื่อด่าน

```text
11-3 ซ่อมกฎคัดแตงโมเกรด A
```

### สถานการณ์

ระบบคัดแตงโมเกรด A ทำงานผิด เพราะตรวจแค่น้ำหนักพอดี แต่ไม่ตรวจเสียงกังวาน ทำให้แตงโมบางลูกถูกติดเกรด A ผิดพลาด

### กฎเสียเริ่มต้น

```text
ถ้า แตงโมน้ำหนักพอดี → ติดสติ๊กเกอร์เกรด A
นอกเหนือจากนี้ → ส่งไปแปรรูป
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
แตงโมบางลูกน้ำหนักพอดี แต่เสียงไม่กังวาน จึงยังไม่ควรเป็นเกรด A
```

### กฎที่ถูกต้อง

```text
ถ้า แตงโมน้ำหนักพอดีและเสียงกังวาน → ติดสติ๊กเกอร์เกรด A
นอกเหนือจากนี้ → ส่งไปแปรรูป
```

### ประเภทบั๊ก

```text
too_broad_condition
```

### สิ่งที่ผู้เรียนต้องทำ

เพิ่มเงื่อนไขให้ครบทั้ง “น้ำหนักพอดี” และ “เสียงกังวาน”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! แตงโมเกรด A ต้องทั้งน้ำหนักพอดีและเสียงกังวาน
```

### ความรู้ที่ได้

บางกฎต้องใช้หลายเงื่อนไขร่วมกัน จึงจะตัดสินใจได้ถูกต้อง

---

# 9. Stage 12: ซ่อมกฎ IF / ELSE IF / ELSE

## 9.1 แนวคิดของ Stage 12

Stage 12 ใช้ฐานจาก Stage 9 ของบทที่ 3 ซึ่งเป็นด่านหลายเงื่อนไข ผู้เรียนต้องตรวจลำดับของกฎ ความถูกต้องของเงื่อนไข และปลายทางของแต่ละกรณี

โครงสร้างที่ใช้:

```text
ถ้า [เงื่อนไขที่ 1] → [คำสั่งที่ 1]
หรือถ้า [เงื่อนไขที่ 2] → [คำสั่งที่ 2]
นอกเหนือจากนี้ → [คำสั่งที่ 3]
```

เป้าหมายคือให้ผู้เรียนเข้าใจว่าระบบหลายเงื่อนไขต้องตรวจทีละกรณี และบั๊กอาจเกิดจากเงื่อนไขกว้างเกินไป ลำดับผิด หรือขอบเขตตัวเลขผิด

---

## 9.2 ด่าน 12-1: คัดเกรดไข่ไก่

### ชื่อด่าน

```text
12-1 ซ่อมกฎคัดเกรดไข่ไก่
```

### สถานการณ์

ระบบคัดเกรดไข่ไก่ต้องแยกไข่พรีเมียม ไข่มาตรฐาน และไข่คัดทิ้ง แต่กฎพรีเมียมกว้างเกินไป ทำให้ไข่ใบใหญ่ที่เปลือกแตกถูกส่งเข้าถาดพรีเมียม

### กฎเสียเริ่มต้น

```text
ถ้า ไข่ใบใหญ่ → ถาดไข่พรีเมียม
หรือถ้า ไข่ใบเล็กแต่ไม่แตก → ถาดไข่มาตรฐาน
นอกเหนือจากนี้ → ถาดคัดทิ้ง
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
ไข่ใบใหญ่แต่เปลือกแตกถูกส่งเข้าถาดพรีเมียม ทั้งที่ควรถูกคัดทิ้ง
```

### กฎที่ถูกต้อง

```text
ถ้า ไข่ใบใหญ่และเปลือกสมบูรณ์ → ถาดไข่พรีเมียม
หรือถ้า ไข่ใบเล็กแต่ไม่แตก → ถาดไข่มาตรฐาน
นอกเหนือจากนี้ → ถาดคัดทิ้ง
```

### ประเภทบั๊ก

```text
too_broad_condition
```

### สิ่งที่ผู้เรียนต้องทำ

แก้เงื่อนไขพรีเมียมให้ตรวจทั้ง “ใบใหญ่” และ “เปลือกสมบูรณ์”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! ไข่พรีเมียมต้องใบใหญ่และเปลือกสมบูรณ์เท่านั้น
```

### ความรู้ที่ได้

เงื่อนไขแรกของ IF / ELSE IF ต้องแม่นยำ เพราะถ้ากว้างเกินไป ระบบจะส่งของผิดตั้งแต่กฎแรก

---

## 9.3 ด่าน 12-2: คัดคุณภาพขนแกะ

### ชื่อด่าน

```text
12-2 ซ่อมกฎคัดคุณภาพขนแกะ
```

### สถานการณ์

ระบบคัดขนแกะต้องแยกขนแกะพรีเมียม ขนแกะที่ต้องทำความสะอาด และขนแกะที่ต้องรีไซเคิล แต่ระบบตั้งเงื่อนไขพรีเมียมกว้างเกินไป

### กฎเสียเริ่มต้น

```text
ถ้า ขนแกะสะอาด → ม้วนไหมพรมพรีเมียม
หรือถ้า ขนแกะมีเศษหญ้า → ส่งไปเครื่องทำความสะอาด
นอกเหนือจากนี้ → ถังรีไซเคิล
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
ขนแกะที่สะอาดแต่ไม่นุ่มถูกส่งไปเป็นพรีเมียม ทั้งที่ควรตรวจคุณภาพเพิ่ม
```

### กฎที่ถูกต้อง

```text
ถ้า ขนแกะสะอาดและนุ่ม → ม้วนไหมพรมพรีเมียม
หรือถ้า ขนแกะมีเศษหญ้า → ส่งไปเครื่องทำความสะอาด
นอกเหนือจากนี้ → ถังรีไซเคิล
```

### ประเภทบั๊ก

```text
too_broad_condition
```

### สิ่งที่ผู้เรียนต้องทำ

แก้เงื่อนไขพรีเมียมให้ตรวจทั้ง “สะอาด” และ “นุ่ม”

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! ขนแกะพรีเมียมต้องทั้งสะอาดและนุ่ม
```

### ความรู้ที่ได้

กฎที่มีหลายเงื่อนไขต้องครบตามเกณฑ์ ไม่ใช่ตรวจเพียงบางส่วน

---

## 9.4 ด่าน 12-3: ตรวจคุณภาพน้ำนม

### ชื่อด่าน

```text
12-3 ซ่อมกฎตรวจอุณหภูมิน้ำนม
```

### สถานการณ์

ระบบตรวจคุณภาพน้ำนมต้องแยกนมพร้อมขาย นมที่ต้องทำความเย็น และนมที่ต้องคัดทิ้ง แต่ระบบกำหนดขอบเขตอุณหภูมิผิด

### กฎเสียเริ่มต้น

```text
ถ้า อุณหภูมินมไม่เกิน 8°C → บรรจุขวดพร้อมขาย
หรือถ้า อุณหภูมินมต่ำกว่า 15°C → ส่งเข้าห้องทำความเย็น
นอกเหนือจากนี้ → คัดทิ้ง
```

### ปัญหาที่เกิดเมื่อรันทดสอบ

```text
นมที่อุณหภูมิ 8°C ถูกจัดกลุ่มไม่ชัดเจน และนมช่วง 8-15°C อาจถูกตัดสินผิดเกณฑ์
```

### กฎที่ถูกต้อง

```text
ถ้า อุณหภูมินมต่ำกว่า 8°C → บรรจุขวดพร้อมขาย
หรือถ้า อุณหภูมิ 8-15°C → ส่งเข้าห้องทำความเย็น
นอกเหนือจากนี้ → คัดทิ้ง
```

### ประเภทบั๊ก

```text
boundary_error
```

### สิ่งที่ผู้เรียนต้องทำ

แก้ขอบเขตอุณหภูมิให้ชัดเจน โดยแยกนมต่ำกว่า 8°C กับนมช่วง 8-15°C ออกจากกัน

### Feedback เมื่อแก้ถูก

```text
ถูกต้อง! นมพร้อมขายต้องเย็นต่ำกว่า 8°C ส่วน 8-15°C ต้องส่งไปทำความเย็นก่อน
```

### ความรู้ที่ได้

บั๊กอาจเกิดจากตัวเลขหรือขอบเขตของเงื่อนไขผิดเพียงเล็กน้อย

# 10. สิ่งที่ต้องเพิ่มในระบบเพื่อรองรับ Debug Mode

## 10.1 เพิ่ม `brokenLogic`

ในบทที่ 3 เดิมมี `expectedLogic` เพื่อบอกผลลัพธ์ที่ถูกต้องอยู่แล้ว บทที่ 4 ควรเพิ่ม `brokenLogic` เพื่อกำหนดกฎผิดที่เริ่มต้นด่าน

ตัวอย่าง

```javascript
brokenLogic: [
  { type: 'if', condition: 'dark_carrot', action: 'wash' }
],
expectedLogic: [
  { type: 'if', condition: 'muddy_carrot', action: 'wash' }
]
```

## 10.2 เพิ่ม `bugType`

ใช้ระบุประเภทของข้อผิดพลาด เพื่อใช้แสดง feedback และเก็บข้อมูลการเรียนรู้

```javascript
bugType: 'wrong_condition'
```

ประเภทที่ควรรองรับ

| bugType | ความหมาย |
|---|---|
| `wrong_condition` | เงื่อนไขผิด |
| `wrong_action` | คำสั่งหรือปลายทางผิด |
| `too_broad_condition` | เงื่อนไขกว้างเกินไป |
| `too_narrow_condition` | เงื่อนไขแคบเกินไป |
| `wrong_else_action` | คำสั่ง ELSE ผิด |
| `wrong_order` | ลำดับ IF / ELSE IF ผิด |
| `boundary_error` | ขอบเขตตัวเลขผิด |

## 10.3 เพิ่ม `debugReport`

หลังจากผู้เรียนกดทดสอบกฎเดิม ระบบควรแสดงรายงานข้อผิดพลาด เช่น

```javascript
debugReport: {
  title: 'พบข้อผิดพลาดในระบบล้างแครอท',
  summary: 'ระบบใช้เงื่อนไขแครอทสีเข้ม แทนที่จะตรวจว่าเปื้อนโคลนหรือไม่',
  hints: [
    'ลองดูว่าแครอทที่ควรล้างมีลักษณะอะไร',
    'สีเข้มกับเปื้อนโคลนเป็นสิ่งเดียวกันหรือไม่',
    'ควรเปลี่ยนเงื่อนไขเป็นแครอทเปื้อนโคลน'
  ]
}
```

## 10.4 เพิ่ม `debugGoal`

ใช้บอกผู้เรียนว่าด่านนี้ต้องซ่อมอะไร

```javascript
debugGoal: 'แก้เงื่อนไขให้ระบบล้างเฉพาะแครอทที่เปื้อนโคลน'
```

## 10.5 เพิ่ม `suspiciousBlocks`

ใช้ highlight บล็อกที่น่าสงสัยหลังจากรันทดสอบกฎเดิม

```javascript
suspiciousBlocks: ['condition']
```

หรือ

```javascript
suspiciousBlocks: ['elseAction']
```

| คะแนน | ดาว |
|---|---|
| 90–100 | 3 ดาว |
| 70–89 | 2 ดาว |
| 50–69 | 1 ดาว |

## 13.3 การนับ Attempt

ควรนับ attempt เมื่อ

- ผู้เรียนทดสอบหลังแก้ไขแล้วผลยังผิด

- ผู้เรียนเลือก action หรือปลายทางผิด

แต่ไม่ควรนับ attempt จากการรันทดสอบกฎเดิมครั้งแรก เพราะการทดสอบกฎเดิมเป็นขั้นตอนบังคับของบทที่ 4

# 17. Acceptance Criteria

- Stage 11 เป็นการ debug กฎ IF / ELSE
- Stage 12 เป็นการ debug กฎ IF / ELSE IF / ELSE

- มีคะแนนและดาว
- เล่นครบทุกด่านได้

- Feedback ไม่ตำหนิเด็ก แต่ชี้ให้สังเกตและแก้ไข

# 19. สรุป

แนวทางใหม่ที่เหมาะสมที่สุดคือ **ยกเกมบทที่ 3 มาใช้ในบทที่ 4 ทั้งชุด** แต่เปลี่ยนเป้าหมายจากการสร้างกฎเป็นการตรวจสอบและแก้ไขกฎที่ผิดพลาด

ข้อดีของแนวทางนี้คือ

- ไม่ต้องสร้างเกมใหม่ตั้งแต่ศูนย์
- ใช้ UI และ engine ที่ผู้เรียนคุ้นเคยแล้ว
- ลดปัญหาวัตถุและบล็อกคำสั่งทับซ้อน
- ทำให้บทที่ 3 และบทที่ 4 ต่อเนื่องกันอย่างเป็นธรรมชาติ
- สอดคล้องกับเนื้อหา “ตรวจสอบและแก้ไขข้อผิดพลาด” มากกว่าแนวทางเดิม
- เปิดโอกาสให้ผู้เรียนเห็นบั๊กจากผลลัพธ์จริง แล้วแก้ไขผ่านกระบวนการทดสอบซ้ำ

บทที่ 3 คือการเป็น “ผู้จัดการฟาร์มที่สร้างกฎ” ส่วนบทที่ 4 คือการเป็น “ช่างซ่อมระบบฟาร์มที่ตรวจและแก้กฎผิด” ใช้เกมเดียวกันได้ แต่เป้าหมายการคิดต่างกันชัดเจน และเหมาะกับนักเรียนชั้นประถมศึกษาปีที่ 4 มากกว่า
