หัวใจของระบบคือ

## 2. ความสอดคล้องกับเกมบทที่ 3 เวอร์ชันปัจจุบัน

เกมบทที่ 3 ล่าสุดถูกจัดธีมใหม่เป็น 3 เกมหลัก ได้แก่

| เกม | ธีม | ตรรกะหลัก |
|---|---|---|
| เกมที่ 1 | ผลผลิตจากพืชผัก | If |
| เกมที่ 2 | ผลผลิตจากผลไม้ | If / Else |
| เกมที่ 3 | ผลผลิตจากสัตว์ | If / Else If / Else |

ระบบสร้างชิ้นงานควรใช้โครงสร้างเดียวกัน เพื่อให้เด็กสามารถเลือกสร้างเกมได้ตามระดับความเข้าใจของตนเอง

| ประเภทชิ้นงาน | เหมาะกับผู้เรียน | รูปแบบกฎ |
|---|---|---|
| แบบง่าย | เข้าใจ If | ถ้า...แล้ว... |
| แบบกลาง | เข้าใจ If / Else | ถ้า...แล้ว... / นอกเหนือจากนี้... |
| แบบท้าทาย | เข้าใจ If / Else If / Else | ถ้า... / นอกเหนือจากนี้ถ้า... / นอกเหนือจากนี้... |

## 4. เป้าหมายการเรียนรู้ของระบบสร้างชิ้นงาน

เมื่อสร้างชิ้นงานเสร็จ นักเรียนควรสามารถ

1. อธิบายได้ว่าเงื่อนไข If ใช้ตรวจกรณีพิเศษอย่างไร
2. อธิบายได้ว่า Else คือกลุ่มที่เหลือทั้งหมด
3. อธิบายได้ว่า Else If ใช้ตรวจเงื่อนไขลำดับถัดไป
4. สร้างวัตถุที่มีคุณสมบัติตรงตามเงื่อนไขได้
5. สร้างตัวหลอกที่ไม่เข้าเงื่อนไขได้
6. จับคู่เงื่อนไขกับคำสั่งได้ถูกต้อง
7. ออกแบบปลายทางของวัตถุให้สอดคล้องกับคำสั่ง
8. ทดสอบด่านของตนเองและแก้ไขข้อผิดพลาดได้
9. สร้างเกมที่เพื่อนสามารถเล่นได้จริง
10. สะท้อนเหตุผลของกฎที่ตนเองออกแบบได้

# 6. ขั้นตอนที่ 1: เลือกชนิดเกม

## 6.1 ตัวเลือกที่ 1: เกม If

**ชื่อในระบบ:** โรงคัดผักสวนครัว  
**เหมาะสำหรับ:** นักเรียนที่ต้องการสร้างเกมง่าย  
**โครงสร้างกฎ:**

```text
ถ้า [เงื่อนไข] → [คำสั่ง]
ไม่เข้าเงื่อนไข → ปล่อยผ่านอัตโนมัติ
```

**ตัวอย่าง:**

```text
ถ้า [ผักกาดมีหนอน] → [ส่งไปโต๊ะตรวจศัตรูพืช]
```

---

## 6.2 ตัวเลือกที่ 2: เกม If / Else

**ชื่อในระบบ:** โรงคัดผลไม้แสนอร่อย  
**เหมาะสำหรับ:** นักเรียนที่ต้องการแยกวัตถุเป็น 2 กลุ่ม  
**โครงสร้างกฎ:**

```text
ถ้า [เงื่อนไข] → [คำสั่ง A]
นอกเหนือจากนี้ → [คำสั่ง B]
```

**ตัวอย่าง:**

```text
ถ้า [กล้วยสุก] → [ส่งไปขายหน้าร้าน]
นอกเหนือจากนี้ → [ส่งไปบ่มต่อ]
```

---

## 6.3 ตัวเลือกที่ 3: เกม If / Else If / Else

**ชื่อในระบบ:** โรงคัดผลผลิตจากฟาร์มสัตว์  
**เหมาะสำหรับ:** นักเรียนที่ต้องการสร้างเกมท้าทาย  
**โครงสร้างกฎ:**

```text
ถ้า [เงื่อนไข A] → [คำสั่ง A]
นอกเหนือจากนี้ถ้า [เงื่อนไข B] → [คำสั่ง B]
นอกเหนือจากนี้ → [คำสั่ง C]
```

**ตัวอย่าง:**

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
นอกเหนือจากนี้ถ้า [ไข่ใบเล็กและไม่ร้าว] → [ถาดมาตรฐาน]
นอกเหนือจากนี้ → [ถังคัดทิ้ง]
```

---

## 6.4 UI ขั้นตอนเลือกชนิดเกม

ควรแสดงเป็นการ์ด 3 ใบ

```text
┌────────────────────┐
│ เกม If             │
│ โรงคัดผักสวนครัว   │
│ เหมาะกับมือใหม่    │
│ [เลือก]            │
└────────────────────┘

┌────────────────────┐
│ เกม If / Else      │
│ โรงคัดผลไม้        │
│ แยกเป็น 2 ทาง      │
│ [เลือก]            │
└────────────────────┘

┌────────────────────┐
│ เกม If / Else If   │
│ ผลผลิตจากสัตว์     │
│ คัดหลายระดับ       │
│ [เลือก]            │
└────────────────────┘
```

เมื่อเลือกชนิดเกมแล้ว ระบบต้องสร้างโครงสร้างแผงกฎให้เหมาะสมโดยอัตโนมัติ

# 7. ขั้นตอนที่ 2: ตั้งชื่อด่านและเรื่องราว

นักเรียนควรกรอกข้อมูลสั้น ๆ ได้แก่

1. ชื่อด่าน
2. คำอธิบายภารกิจ
3. เป้าหมายของผู้เล่น
4. ข้อความแนะนำก่อนเริ่มเกม

ตัวอย่าง:

```text
ชื่อด่าน: โรงคัดไข่ไก่บ้านนาอุดม
ภารกิจ: ช่วยคัดไข่ไก่ก่อนส่งตลาด
เป้าหมาย: คัดไข่ให้ถูกถาดอย่างน้อย 80%
คำแนะนำ: สังเกตขนาดของไข่และรอยร้าวให้ดี
```

ระบบควรจำกัดความยาวข้อความ เพื่อไม่ให้เด็กเขียนยาวเกินไป เช่น

- ชื่อด่าน ไม่เกิน 60 ตัวอักษร
- ภารกิจ ไม่เกิน 150 ตัวอักษร
- คำแนะนำ ไม่เกิน 150 ตัวอักษร

# 8. ขั้นตอนที่ 3: เลือกหรือสร้างวัตถุบนสายพาน

## 8.1 หลักการ

วัตถุคือสิ่งที่จะวิ่งบนสายพาน เช่น ผัก ผลไม้ ไข่ นม หรือขนแกะ นักเรียนควรเลือกจากคลังวัตถุแทนการพิมพ์เองทั้งหมด เพื่อให้ผลงานมีภาพและโครงสร้างข้อมูลชัดเจน

## 8.2 คลังวัตถุแนะนำ

### กลุ่มพืชผัก

- แครอทสะอาด
- แครอทเปื้อนโคลน
- ผักกาดปกติ
- ผักกาดมีหนอน
- มันฝรั่งปกติ
- มันฝรั่งมีหน่องอก
- แตงกวาปกติ
- แตงกวามีรอยช้ำ

### กลุ่มผลไม้

- ส้มลูกใหญ่
- ส้มลูกเล็ก
- กล้วยสุก
- กล้วยดิบ
- แตงโมเกรดดี
- แตงโมไม่ผ่านเกณฑ์
- มะม่วงสุก
- มะม่วงดิบ

### กลุ่มผลผลิตจากสัตว์

- ไข่ใบใหญ่ไม่ร้าว
- ไข่ใบเล็กไม่ร้าว
- ไข่ร้าว
- ขนแกะสะอาด
- ขนแกะมีเศษหญ้า
- ขนแกะสกปรกมาก
- ขวดนมเย็นคุณภาพดี
- ขวดนมอุณหภูมิสูง
- ขวดนมเสีย

## 8.3 การกำหนดคุณสมบัติของวัตถุ

เมื่อเลือกวัตถุ ระบบควรมีคุณสมบัติมาให้ เช่น

```json
{
  "id": "egg_big_good",
  "label": "ไข่ใบใหญ่ไม่ร้าว",
  "fallbackIcon": "🥚",
  "props": {
    "category": "animal_product",
    "type": "egg",
    "size": "big",
    "cracked": false
  }
}
```

เด็กไม่ควรต้องแก้ JSON เอง แต่ UI ควรแสดงเป็นช่องติ๊กหรือป้ายคุณสมบัติ เช่น

```text
ประเภท: ไข่
ขนาด: ใหญ่
รอยร้าว: ไม่มี
```

## 8.4 จำนวนวัตถุขั้นต่ำ

ระบบควรกำหนดจำนวนขั้นต่ำตามชนิดเกม

| ชนิดเกม | วัตถุขั้นต่ำ | กลุ่มเข้าเงื่อนไข | ตัวหลอก / กลุ่มอื่น |
|---|---:|---:|---:|
| If | 4 ชิ้น | อย่างน้อย 1 | อย่างน้อย 2 |
| If / Else | 6 ชิ้น | อย่างน้อย 2 | อย่างน้อย 2 |
| If / Else If / Else | 9 ชิ้น | If ≥ 2, Else If ≥ 2 | Else ≥ 2 |

# 9. ระบบตัวหลอก

## 9.1 เหตุผลที่ต้องมีตัวหลอก

ถ้าไม่มีตัวหลอก เกมจะง่ายเกินไป เพราะวัตถุทุกชิ้นจะเข้าเงื่อนไขชัดเจนจนผู้เล่นไม่ต้องคิดมาก ตัวหลอกช่วยให้ด่านมีความท้าทาย และทำให้นักเรียนเข้าใจเงื่อนไขอย่างแท้จริง

ตัวหลอกไม่ควรเป็นวัตถุที่ผิดธีม แต่ควรเป็นวัตถุที่ “ดูคล้าย” หรือ “ใกล้เคียง” กับเงื่อนไข เช่น

- แครอทสะอาด เป็นตัวหลอกของแครอทเปื้อนโคลน
- ผักกาดมีรอยดิน เป็นตัวหลอกของผักกาดมีหนอน
- กล้วยเหลืองอมเขียว เป็นตัวหลอกของกล้วยสุก
- ไข่ใบใหญ่แต่ร้าว เป็นตัวหลอกของไข่ใบใหญ่ไม่ร้าว
- ขนแกะสะอาดแต่เปียก เป็นตัวหลอกของขนแกะสะอาดและนุ่ม

---

## 9.2 ประเภทตัวหลอก

### ตัวหลอกแบบภาพคล้าย

วัตถุดูคล้ายกัน แต่คุณสมบัติต่างกัน เช่น

```text
ไข่ใบใหญ่ไม่ร้าว
ไข่ใบใหญ่แต่ร้าว
```

### ตัวหลอกแบบคุณสมบัติใกล้เคียง

มีคุณสมบัติบางส่วนตรง แต่ไม่ครบ เช่น

```text
แตงโมน้ำหนักดี แต่เสียงไม่กังวาน
```

### ตัวหลอกแบบไม่เกี่ยวข้อง

อยู่ในสายพาน แต่ไม่เข้าเงื่อนไขใด เช่น

```text
มะม่วงดิบปนมากับกล้วยสุก
```

ควรใช้ตัวหลอกแบบไม่เกี่ยวข้องอย่างระมัดระวัง ไม่ให้ทำให้เด็กสับสนเกินไป

---

## 9.3 UI สำหรับเพิ่มตัวหลอก

ในหน้าสร้างวัตถุ ควรมีป้ายหรือ checkbox

```text
[ ] ตั้งเป็นตัวหลอก
```

แต่ไม่ควรให้เด็กตั้งแบบสุ่ม ควรมีคำอธิบายสั้น ๆ

```text
ตัวหลอกคือวัตถุที่ดูคล้ายของที่เข้าเงื่อนไข แต่จริง ๆ ไม่ควรถูกส่งไปทางเดียวกัน
```

ระบบควรมีปุ่มแนะนำ

```text
[เพิ่มตัวหลอกอัตโนมัติ]
```

เช่น ถ้าเด็กเลือก “ไข่ใบใหญ่ไม่ร้าว” ระบบเสนอ “ไข่ใบใหญ่แต่ร้าว” ให้เป็นตัวหลอก

# 10. แถบรายการวัตถุด้านล่างของกรอบเกม

## 10.1 ปัญหาที่ต้องแก้

ในเกมหลักและระบบทดลองเล่น มีแถบรายการวัตถุด้านล่างของกรอบเกม ซึ่งบางครั้งอาจบดบังวัตถุที่อยู่มุมล่างของพื้นที่เกมเล็กน้อย ทำให้ผู้เล่นมองวัตถุหรือปลายทางบางส่วนไม่ชัด

## 10.2 แนวทางแก้ layout

แถบรายการวัตถุควรอยู่ในพื้นที่ของตนเอง ไม่ทับซ้อนกับพื้นที่สายพานหรือวัตถุในเกม

โครงสร้างที่เสนอ:

```text
┌──────────────────────────────┐
│ พื้นที่สายพานและเครื่องจักร │
│                              │
│                              │
├──────────────────────────────┤
│ แถบรายการวัตถุที่จะมา        │
│ [🥕] [🥬] [🥚] [🍌]           │
└──────────────────────────────┘
```

ต้องแยกด้วย container คนละส่วน ไม่ควรใช้ position absolute ซ้อนทับบน board

ตัวอย่าง CSS:

```css
.game-board {
  display: grid;
  grid-template-rows: minmax(0, 1fr) auto;
  overflow: hidden;
}

.conveyor-play-area {
  position: relative;
  min-height: 360px;
  overflow: hidden;
}

.item-preview-bar {
  position: relative;
  z-index: 1;
  flex-shrink: 0;
  padding: 10px 12px;
  border-top: 1px solid rgba(120, 90, 40, 0.2);
  background: rgba(255, 255, 255, 0.92);
}
```

## 10.3 เว้น Safe Area ด้านล่าง

ถ้าเกมใช้ animation ที่วัตถุอาจเคลื่อนใกล้ขอบล่าง ต้องกำหนด safe area เช่น

```javascript
const BOARD_SAFE_PADDING = {
  top: 24,
  right: 24,
  bottom: 84,
  left: 24
};
```

วัตถุใน Phaser หรือ DOM animation ไม่ควรเคลื่อนลงไปในพื้นที่ของ preview bar

## 10.4 คลิกแถบรายการวัตถุเพื่อดูรายละเอียด

แถบรายการวัตถุด้านล่างควรมีประโยชน์มากกว่าแค่แสดงลำดับวัตถุ ควรสามารถคลิกเพื่อดูรายละเอียดได้ว่า

1. วัตถุชิ้นนี้คืออะไร
2. มีคุณสมบัติอะไร
3. ควรเข้าเงื่อนไขใด
4. เป็นตัวหลอกหรือไม่
5. ปลายทางที่ถูกต้องคืออะไร

เมื่อคลิกวัตถุใน preview bar ควรเปิด modal หรือ popover เช่น

```text
ไข่ใบใหญ่แต่ร้าว

คุณสมบัติ:
- ประเภท: ไข่
- ขนาด: ใหญ่
- รอยร้าว: มี

สถานะ: ตัวหลอก
เหตุผล: แม้จะเป็นไข่ใบใหญ่ แต่มีรอยร้าว จึงไม่ควรเข้าถาดพรีเมียม

ปลายทางที่ถูกต้อง:
ถังคัดทิ้ง
```

## 10.5 โหมดแสดงคำใบ้หรือเฉลย

ควรแยกเป็น 2 โหมด

### โหมดก่อนเล่น

แสดงเฉพาะข้อมูลพื้นฐาน เช่น

- ชื่อวัตถุ
- คุณสมบัติที่มองเห็น
- มีตัวหลอกหรือไม่แบบไม่เฉลยทั้งหมด

ตัวอย่าง:

```text
วัตถุนี้อาจเป็นตัวหลอก ลองสังเกตรอยร้าวให้ดี
```

### โหมดหลังเล่นหรือโหมดครู

แสดงเฉลยเต็ม เช่น

- เงื่อนไขที่ถูกต้อง
- ปลายทางที่ถูกต้อง
- เหตุผล
- ตัวหลอกหรือไม่

เพื่อป้องกันไม่ให้นักเรียนกดดูเฉลยก่อนเล่น

# 11. ขั้นตอนที่ 4: สร้างเงื่อนไข คำสั่ง และปลายทาง

## 11.1 เงื่อนไข

ระบบควรให้เด็กเลือกเงื่อนไขจากวัตถุที่สร้างไว้ เช่น

```text
ไข่ใบใหญ่และไม่ร้าว
ไข่ใบเล็กและไม่ร้าว
ไข่ร้าว
```

เงื่อนไขควรเกิดจาก props ของวัตถุ ไม่ใช่พิมพ์ข้อความลอย ๆ เพื่อให้ engine ตรวจได้จริง

ตัวอย่าง condition config:

```json
{
  "id": "egg_big_good",
  "label": "ไข่ใบใหญ่และไม่ร้าว",
  "type": "match",
  "props": {
    "type": "egg",
    "size": "big",
    "cracked": false
  }
}
```

## 11.2 คำสั่ง

คำสั่งคือสิ่งที่ระบบจะทำเมื่อเข้าเงื่อนไข เช่น

```text
ส่งไปถาดพรีเมียม
ส่งไปถาดมาตรฐาน
ส่งไปถังคัดทิ้ง
```

คำสั่งควรเชื่อมกับปลายทางชัดเจน

## 11.3 ปลายทาง

ปลายทางคือจุดสิ้นสุดของวัตถุ เช่น

```text
เครื่องล้าง
ตะกร้าปกติ
กล่องพรีเมียม
ถังปุ๋ยหมัก
ถาดไข่พรีเมียม
ถังคัดทิ้ง
```

ปลายทางควรมีภาพหรือไอคอนประกอบ

# 12. ขั้นตอนที่ 5: สร้างกฎด้วย Drag & Drop

## 12.1 หลักการ

แผงสร้างกฎในระบบสร้างชิ้นงานควรใช้ component เดียวกับตัวเกมบทที่ 3 เพื่อให้ประสบการณ์เหมือนกัน

เด็กต้องลากบล็อกเงื่อนไขและบล็อกคำสั่งไปวางในช่อง เช่น

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
นอกเหนือจากนี้ถ้า [ไข่ใบเล็กและไม่ร้าว] → [ถาดมาตรฐาน]
นอกเหนือจากนี้ → [ถังคัดทิ้ง]
```

## 12.2 Validation ก่อนทดสอบ

ก่อนกดทดลองเล่น ระบบต้องตรวจว่า

- มีเงื่อนไขครบตามชนิดเกม
- มีคำสั่งครบทุกแถว
- มีปลายทางครบทุกคำสั่ง
- มีวัตถุอย่างน้อย 1 ชิ้นที่ตรงกับแต่ละเงื่อนไข
- มีตัวหลอกขั้นต่ำ
- ไม่มี action ที่ไม่มีปลายทาง
- ไม่มี condition ที่ไม่มีวัตถุรองรับ

ตัวอย่างข้อความเตือน:

```text
ยังไม่มีวัตถุที่เข้าเงื่อนไข “ไข่ใบเล็กและไม่ร้าว”
กรุณาเพิ่มวัตถุหรือเปลี่ยนเงื่อนไข
```

# 13. ขั้นตอนที่ 6: ทดลองเล่นด่านของฉัน

## 13.1 เป้าหมาย

การทดลองเล่นต้องใช้ engine เดียวกับเกมหลัก เพื่อพิสูจน์ว่าชิ้นงานของนักเรียนเล่นได้จริง

นักเรียนกดปุ่ม

```text
[ทดลองเล่นด่านของฉัน]
```

แล้วระบบโหลด config ที่นักเรียนสร้างเข้า `conveyorLogic(config)`

## 13.2 สิ่งที่ต้องแสดงระหว่างทดลองเล่น

- ชื่อด่าน
- ภารกิจ
- สายพาน
- วัตถุที่นักเรียนเลือก
- เครื่องสแกน
- ปลายทาง
- กฎที่นักเรียนสร้าง
- ผลถูก/ผิด
- สรุปดาวหลังเล่น

## 13.3 เงื่อนไขก่อนอนุญาตให้ส่งงาน

ระบบควรบังคับให้เด็กทดสอบอย่างน้อย 1 ครั้งก่อนส่ง และถ้าด่านมีปัญหาหนัก เช่น ไม่มีวัตถุเข้าเงื่อนไขเลย ควรไม่ให้ส่ง

```text
กรุณาทดลองเล่นด่านของตนเองก่อนส่งผลงาน
```

# 20. Prompt สำหรับสั่ง Codex พัฒนา

```text
พัฒนาระบบสร้างชิ้นงานบทที่ 3 ให้เป็น Smart Farm Mini Game Builder แทนแบบฟอร์มเดิม

เป้าหมาย:
ให้นักเรียนสร้างด่านเกมสายพานตรรกะของตนเองที่เล่นได้จริง โดยใช้ engine เดียวกับเกม Smart Farm Manager บทที่ 3

1. นักเรียนเลือกชนิดเกมได้ 3 แบบ:
   - If: โรงคัดผักสวนครัว
   - If / Else: โรงคัดผลไม้
   - If / Else If / Else: โรงคัดผลผลิตจากสัตว์
2. นักเรียนตั้งชื่อด่าน ภารกิจ และคำแนะนำสั้น ๆ ได้
3. นักเรียนเลือกวัตถุจากคลังได้ เช่น ผัก ผลไม้ ไข่ นม ขนแกะ
4. นักเรียนกำหนดตัวหลอกได้ และระบบควรตรวจว่ามีตัวหลอกขั้นต่ำ
5. นักเรียนสร้างเงื่อนไขจาก props ของวัตถุได้
6. นักเรียนสร้างคำสั่งและปลายทางได้
7. นักเรียนลากบล็อกเงื่อนไขและบล็อกคำสั่งลงแผงกฎได้ด้วย Drag & Drop
8. ระบบต้อง validate ก่อนทดสอบ เช่น กฎครบ วัตถุครบ ตัวหลอกครบ ปลายทางครบ
9. เพิ่มปุ่มทดลองเล่นด่านของฉัน โดยแปลง work_data เป็น levelConfig และโหลดเข้า conveyorLogic engine
10. ต้องทดสอบด่านอย่างน้อย 1 ครั้งก่อนส่งงาน
11. บันทึกข้อมูลทั้งหมดลง work_data เป็น JSON โดยใช้ project_type = smart_farm_mini_game
12. แถบรายการวัตถุด้านล่างต้องไม่บดบังพื้นที่เกม ให้แยกเป็น container คนละส่วน
13. แถบรายการวัตถุคลิกได้ เพื่อดูรายละเอียด คุณสมบัติ ตัวหลอก และปลายทางที่ถูกต้อง
14. ก่อนเล่นให้แสดงข้อมูลแบบไม่เฉลยทั้งหมด หลังเล่นหรือโหมดครูจึงแสดงเฉลยเต็ม
15. หน้า Showcase ต้องมีปุ่มเล่นด่านของเพื่อน โดยโหลด work_data เป็น mini level
16. ครูต้องดูเฉลย กฎ และผลการทดสอบได้
17. หลีกเลี่ยงการเปลี่ยน database ถ้าไม่จำเป็น ใช้ work_data เดิมให้มากที่สุด
```

## 22. สรุป

ระบบสร้างชิ้นงานบทที่ 3 ควรพัฒนาเป็น **Smart Farm Mini Game Builder** ที่ให้นักเรียนสร้างด่านสายพานตรรกะของตนเองได้จริง โดยใช้โครงสร้างเดียวกับเกมหลักของบทที่ 3

สิ่งสำคัญที่สุดคือ

> นักเรียนต้องสร้าง “เกมที่เล่นได้” ไม่ใช่ “คำอธิบายของเกม”

ระบบนี้จะทำให้บทที่ 3 สมบูรณ์ขึ้น เพราะนักเรียนจะได้เปลี่ยนบทบาทจากผู้เล่นเป็นผู้ออกแบบด่าน เรียนรู้การสร้างเงื่อนไขจากวัตถุจริง ฝึกออกแบบตัวหลอก และตรวจสอบตรรกะของตนเองผ่านการทดลองเล่น
