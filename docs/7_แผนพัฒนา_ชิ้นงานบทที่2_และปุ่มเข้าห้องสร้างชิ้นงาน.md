# แผนพัฒนาระบบสร้างชิ้นงานบทที่ 2 และการปรับปรุงปุ่มเข้าห้องสร้างชิ้นงาน

> เอกสารนี้เป็นแผนพัฒนาสำหรับบทเรียนที่ 2 “เส้นทางเดินรถไถ” โดยครอบคลุม 2 ส่วนสำคัญ  
> 1) การออกแบบระบบสร้างชิ้นงานหลังเรียนจบบท  
> 2) การปรับปรุงปุ่มเข้าห้องสร้างชิ้นงานให้มองเห็นง่ายและใช้งานสะดวกขึ้น

---

## 1. ที่มาและปัญหาที่พบ

หลังจากนักเรียนเล่นเกมในบทเรียนจบครบทุกด่าน ระบบจะพากลับมาที่หน้าเลือกด่าน `game_select.php` ซึ่งในปัจจุบันมีการแสดงปุ่ม “เข้าห้องสร้างชิ้นงาน” เฉพาะเมื่อผ่านครบทุกด่านแล้ว

อย่างไรก็ตาม ปัญหาที่พบคือปุ่มดังกล่าวอยู่ด้านล่างของหน้า หลังรายการการ์ดด่านทั้งหมด ทำให้นักเรียนต้องเลื่อนหน้าจอลงไปจึงจะเห็นปุ่ม ส่งผลให้ประสบการณ์ใช้งานยังไม่ดีพอ โดยเฉพาะบนแท็บเล็ตหรือโทรศัพท์มือถือ

ดังนั้นควรปรับให้ระบบสื่อสารกับนักเรียนชัดเจนขึ้นว่า

> “เมื่อผ่านครบทุกด่านแล้ว ขั้นต่อไปคือการสร้างชิ้นงาน”

และควรทำให้ปุ่มเข้าห้องสร้างชิ้นงานมองเห็นได้ทันที โดยไม่ต้องเลื่อนค้นหาเอง

---

## 2. เป้าหมายการพัฒนา

### 2.1 เป้าหมายด้านระบบสร้างชิ้นงาน

พัฒนาระบบสร้างชิ้นงานของบทเรียนที่ 2 ให้ผู้เรียนสามารถออกแบบ “โจทย์เส้นทางรถไถ” ด้วยตนเอง โดยอาศัยความรู้จากเกมทั้ง 3 ด่านของบทเรียน ได้แก่

| ด่าน | เนื้อหา | สิ่งที่นำไปใช้สร้างชิ้นงาน |
|---|---|---|
| เดินหน้าสู่แปลงนา | สั่งรถไถไปถึงจุดหมาย | วางจุดเริ่มต้นและเป้าหมาย |
| หลบหลีกกองฟาง | เดินหลบสิ่งกีดขวาง | วางกองฟาง/หินเป็นอุปสรรค |
| เก็บเกี่ยวผลผลิต | เก็บผลผลิตก่อนกลับโรงนา | วางผลผลิตและโรงนา |

### 2.2 เป้าหมายด้านปุ่มเข้าห้องสร้างชิ้นงาน

ปรับหน้า `game_select.php` ให้แสดงปุ่มเข้าห้องสร้างชิ้นงานอย่างเด่นชัดขึ้น เมื่อผู้เรียนผ่านครบทุกด่าน โดยไม่ต้องเลื่อนลงไปค้นหา

---

## 3. แนวคิดหลักของระบบสร้างชิ้นงานบทที่ 2

ชื่อที่แนะนำ:

> **สตูดิโอออกแบบเส้นทางรถไถ**

หรือ

> **สร้างชิ้นงาน: ออกแบบภารกิจเส้นทางรถไถ**

แนวคิดคือให้นักเรียนเปลี่ยนบทบาทจาก “ผู้เล่นเกม” เป็น “ผู้ออกแบบโจทย์” โดยต้องสร้างแผนที่ วางเส้นทาง วางวัตถุ และกำหนดคำสั่งเฉลยที่ทำให้รถไถผ่านภารกิจได้จริง

ชิ้นงานไม่ควรเป็นเพียงภาพวาด แต่ควรเป็น “โจทย์ลำดับคำสั่งที่ทดสอบได้” เพื่อสะท้อนว่านักเรียนเข้าใจการแก้ปัญหาอย่างเป็นขั้นตอน

---

## 4. ขอบเขตของชิ้นงานบทที่ 2

ในเวอร์ชันแรก ให้นักเรียนสร้างชิ้นงาน 1 ชิ้น โดยมีองค์ประกอบดังนี้

| องค์ประกอบ | รายละเอียด |
|---|---|
| แผนที่ | ตารางขนาด 6x5 เหมือนเกมบทที่ 2 |
| จุดเริ่มต้น | วางรถไถได้ 1 คัน |
| จุดหมาย | วางตะกร้าหรือโรงนาได้ 1 จุด |
| สิ่งกีดขวาง | วางกองฟาง/หินได้ตามจำนวนที่กำหนด |
| ผลผลิต | วางผลผลิตได้เฉพาะภารกิจเก็บเกี่ยว |
| คำสั่งเฉลย | ใช้ปุ่มลูกศรสร้างลำดับคำสั่ง |
| คำอธิบายวิธีคิด | นักเรียนอธิบายเส้นทางและเหตุผลของคำสั่ง |

## 5. ประเภทภารกิจที่ให้นักเรียนเลือก

ควรให้เลือกประเภทภารกิจได้ 3 แบบ เพื่อให้ล้อกับเกมทั้ง 3 ด่านของบทเรียนที่ 2

### 5.1 ภารกิจแบบที่ 1: ไปถึงจุดหมาย

เหมาะสำหรับผู้เรียนที่ต้องการออกแบบโจทย์พื้นฐาน

เงื่อนไข:

- ต้องวางรถไถ 1 คัน
- ต้องวางตะกร้าเป้าหมาย 1 จุด
- ไม่จำเป็นต้องมีสิ่งกีดขวาง
- คำสั่งเฉลยต้องพารถไถไปถึงเป้าหมายได้

### 5.2 ภารกิจแบบที่ 2: หลบสิ่งกีดขวาง

เหมาะสำหรับผู้เรียนที่ต้องการออกแบบโจทย์ระดับกลาง

เงื่อนไข:

- ต้องวางรถไถ 1 คัน
- ต้องวางเป้าหมาย 1 จุด
- ต้องวางกองฟางหรือหินอย่างน้อย 1 จุด
- คำสั่งเฉลยต้องพารถไถไปถึงเป้าหมายโดยไม่ชนสิ่งกีดขวาง

### 5.3 ภารกิจแบบที่ 3: เก็บเกี่ยวผลผลิต

เหมาะสำหรับผู้เรียนที่ต้องการออกแบบโจทย์ท้าทาย

เงื่อนไข:

- ต้องวางรถไถ 1 คัน
- ต้องวางโรงนา 1 จุด
- ต้องวางผลผลิตอย่างน้อย 1 จุด
- อาจวางสิ่งกีดขวางเพิ่มเติมได้
- คำสั่งเฉลยต้องพารถไถเก็บผลผลิตครบก่อนกลับโรงนา

## 7. เครื่องมือในหน้า Studio

### 7.1 Toolbox ด้านซ้าย

ควรมีเครื่องมือดังนี้

| เครื่องมือ | ใช้สำหรับ | จำนวนสูงสุด |
|---|---|---:|
| 🚜 รถไถ | จุดเริ่มต้น | 1 |
| 🧺 ตะกร้า | เป้าหมายทั่วไป | 1 |
| 🏚️ โรงนา | จุดหมายของภารกิจเก็บเกี่ยว | 1 |
| 🌾 กองฟาง | สิ่งกีดขวาง | 5 |
| 🪨 หิน | สิ่งกีดขวาง | 5 |
| 🌽 ผลผลิต | สิ่งที่ต้องเก็บ | 3 |
| 🧹 ยางลบ | ลบวัตถุในช่อง | ไม่จำกัด |

หมายเหตุ: ใช้ emoji เป็นต้นแบบได้ แต่ต้องออกแบบ code ให้เปลี่ยนเป็น asset ภาพจริงภายหลังได้ง่าย

### 7.2 กระดานแผนที่

- ใช้ grid 6x5
- วางวัตถุได้ทีละช่อง
- ห้ามวางวัตถุซ้อนกัน
- คลิกช่องเพื่อวางวัตถุที่เลือกจาก toolbox
- คลิกยางลบแล้วคลิกช่องเพื่อลบ
- ควรแสดงตำแหน่งด้วยภาพ/emoji ชัดเจน

### 7.3 แผงคำสั่งเฉลย

มีปุ่มคำสั่ง

- ⬆️ ขึ้น
- ⬇️ ลง
- ⬅️ ซ้าย
- ➡️ ขวา

ฟังก์ชันที่ต้องมี

- เพิ่มคำสั่ง
- ลบคำสั่งรายตัว
- ล้างคำสั่งทั้งหมด
- แสดงจำนวนคำสั่ง เช่น `5 / 15`
- จำกัดคำสั่งไม่เกิน 15 คำสั่ง

### 7.4 ปุ่มทดสอบเส้นทาง

ต้องมีปุ่ม

> ▶️ ทดสอบเส้นทาง

เมื่อกดแล้ว ระบบต้องให้รถไถวิ่งตามคำสั่งจริงบนแผนที่

ผลการทดสอบมี 2 แบบ

| ผล | การตอบสนอง |
|---|---|
| ผ่าน | แสดงข้อความ “โจทย์นี้ใช้งานได้ พร้อมส่งชิ้นงานแล้ว” |
| ไม่ผ่าน | แสดงเหตุผล เช่น “รถไถชนกองฟาง” หรือ “ยังเก็บผลผลิตไม่ครบ” |

## 8. เงื่อนไขก่อนส่งชิ้นงาน

ระบบต้องตรวจสอบก่อนส่งเสมอ

### 8.1 เงื่อนไขพื้นฐาน

1. ต้องวางรถไถ 1 จุด
2. ต้องมีเป้าหมายหรือโรงนา 1 จุด
3. ต้องมีคำสั่งเฉลยอย่างน้อย 1 คำสั่ง
4. ห้ามวางวัตถุซ้อนกันในช่องเดียว
5. ต้องเขียนคำอธิบายวิธีคิด
6. ต้องกดทดสอบเส้นทางและผ่านก่อนส่ง

### 8.2 เงื่อนไขตามประเภทภารกิจ

| ประเภทภารกิจ | เงื่อนไขเพิ่มเติม |
|---|---|
| ไปถึงจุดหมาย | ต้องมีตะกร้า 1 จุด |
| หลบสิ่งกีดขวาง | ต้องมีสิ่งกีดขวางอย่างน้อย 1 จุด |
| เก็บเกี่ยวผลผลิต | ต้องมีผลผลิตอย่างน้อย 1 จุด และมีโรงนา 1 จุด |

### 8.3 ข้อความแจ้งเตือนที่ควรใช้

| กรณี | ข้อความ |
|---|---|
| ไม่วางรถไถ | ยังไม่มีรถไถจุดเริ่มต้นนะ |
| ไม่มีเป้าหมาย | อย่าลืมวางเป้าหมายให้รถไถไปถึง |
| ไม่มีคำสั่ง | เพิ่มคำสั่งลูกศรก่อนส่งชิ้นงานนะ |
| ยังไม่ได้ทดสอบ | ลองกด “ทดสอบเส้นทาง” ให้ผ่านก่อนส่งนะ |
| ทดสอบไม่ผ่าน | เส้นทางยังไม่สำเร็จ ลองปรับคำสั่งหรือแผนที่อีกครั้ง |
| ไม่เขียนคำอธิบาย | เขียนอธิบายวิธีคิดของตนเองก่อนส่งนะ |

## 10. การบันทึก `description`

ช่อง `description` ควรใช้เก็บคำอธิบายของนักเรียน เช่น

```text
[ประเภทภารกิจ: หลบสิ่งกีดขวาง]

ฉันออกแบบให้รถไถเริ่มจากมุมล่างซ้าย แล้วต้องเดินอ้อมกองฟางตรงกลางก่อนถึงตะกร้า
ลำดับคำสั่งที่ใช้คือ เดินขวา ขึ้น ขวา ขึ้น แล้วเดินไปทางขวาจนถึงเป้าหมาย
```

ควรเก็บประเภทภารกิจไว้ใน description ด้วย เพื่อให้ครูอ่านง่าย แม้ข้อมูลหลักจะอยู่ใน `work_data`

## 11. การสร้างไฟล์ใหม่

ควรสร้างไฟล์ใหม่

```text
pages/create_project_algo.php
```

หน้าที่ของไฟล์นี้คือห้องสร้างชิ้นงานของบทที่ 2

ควรยึดโครงสร้างจาก `create_project_logic.php` ได้ แต่ต้องเปลี่ยนแนวคิดจากการวาง item แบบ target/decoy เป็นการออกแบบเส้นทางรถไถ

### สิ่งที่ควรนำจาก `create_project_logic.php` มาใช้ต่อ

- session check
- การเรียก `includes/db.php`
- การเรียก `includes/context.php`
- การดึงงานเดิมของผู้เรียน
- การแสดง feedback เมื่อครูส่งกลับให้แก้ไข
- layout studio 3 คอลัมน์
- การใช้ Phaser สำหรับ canvas
- การส่งข้อมูลไป `api/save_work.php`
- การใช้ `student_works` เป็นตารางจัดเก็บ

### สิ่งที่ต้องเปลี่ยน

| จากเดิม | เปลี่ยนเป็น |
|---|---|
| game_id = 1 | game_id = 2 |
| คลังไอเทมตรรกะ | คลังวัตถุเส้นทางรถไถ |
| target/decoy | start/target/obstacle/crop/barn |
| ตั้งโจทย์เงื่อนไขการคลิก | อธิบายเส้นทางและลำดับคำสั่ง |

## 12. การปรับปรุงปุ่มเข้าห้องสร้างชิ้นงานใน `game_select.php`

### 12.1 ปัญหาปัจจุบัน

### 12.2 แนวทางใหม่

1. Completion Hero ด้านบน

## 13. Completion Hero ด้านบน

ให้แสดงการ์ดใหญ่ใต้หัวข้อบทเรียน ก่อนรายการด่าน

### 13.1 ตำแหน่ง

ควรวางหลังส่วนหัวข้อบทเรียน และก่อน `<div class="row g-4 justify-content-center">` ของการ์ดด่าน

โครงหน้าใหม่ควรเป็น

```text
ปุ่มกลับ / ดูผลงานเพื่อน
หัวข้อบทเรียน
Completion Hero ถ้าผ่านครบแล้ว
รายการด่าน
```

### 13.2 ข้อความที่ใช้

กรณียังไม่สร้างชิ้นงาน

```text
🎉 เยี่ยมมาก! คุณผ่านบทเรียนนี้ครบทุกด่านแล้ว
ขั้นต่อไป: ออกแบบภารกิจเส้นทางรถไถของคุณเอง
[เข้าห้องสร้างชิ้นงาน]
```

กรณีส่งแล้ว

```text
✅ คุณสร้างชิ้นงานแล้ว
สามารถเข้าไปดูหรือปรับปรุงชิ้นงานล่าสุดได้
[ดู/แก้ไขชิ้นงาน]
```

กรณีครูส่งกลับให้แก้

```text
⚠️ คุณครูส่งชิ้นงานกลับให้แก้ไข
อ่านคำแนะนำแล้วปรับปรุงผลงานให้ดีขึ้น
[แก้ไขชิ้นงาน]
```

### 13.3 สีที่แนะนำ

| สถานะ | สี |
|---|---|
| ยังไม่สร้าง | gradient เหลือง/ส้ม |
| ส่งแล้ว | เขียว/น้ำเงิน |
| revision | แดง/ส้ม |
| reviewed | เขียว |

## 17. สิ่งที่ต้องแก้ใน `game_select.php`

### 17.1 ย้าย block ปุ่มสร้างชิ้นงาน

จากเดิมที่แสดงท้ายหน้า ให้แยก logic ออกเป็นตัวแปรก่อนแสดง HTML

ควรคำนวณก่อน render ส่วนรายการด่านว่า

```php
$can_create_project = ($passed_count >= $total_stages && $total_stages > 0);
```

แต่เนื่องจากปัจจุบัน `$passed_count` ถูกคำนวณระหว่าง while loop การ render การ์ดด่าน อาจต้องปรับเป็น

1. ดึง stage ทั้งหมดมาเก็บใน array ก่อน
2. คำนวณ passed_count และ total_stages
3. คำนวณ project status
4. render Completion Hero
5. render stage cards

### 17.2 เหตุผลที่ควรดึง stage เป็น array ก่อน

เพื่อให้สามารถแสดง Completion Hero ด้านบนได้ก่อนรายการด่าน โดยไม่ต้องรอ while loop จบ

โครง logic ใหม่

```php
$stages = [];
$passed_count = 0;
$total_stages = 0;

while ($stage = $stages_result->fetch_assoc()) {
    $stages[] = $stage;
    $total_stages++;
    if (($stage['score'] ?? 0) > 0) {
        $passed_count++;
    }
}

$can_create_project = ($passed_count >= $total_stages && $total_stages > 0);
```

จากนั้นค่อย render cards ด้วย

```php
foreach ($stages as $stage) {
    // render stage card
}
```

## 19. สิ่งที่ยังไม่ทำในรอบนี้

ยังไม่ต้องทำ

- ระบบให้เพื่อนเข้ามาเล่นโจทย์ที่นักเรียนสร้าง
- ระบบแชร์ชิ้นงานแบบ public
- ระบบให้คะแนนอัตโนมัติจากความยากของแผนที่
- ระบบสร้างแผนที่ขนาดอื่น
- ระบบลากวัตถุอิสระแบบไม่มี grid
- ระบบ import/export ชิ้นงาน
- ระบบ asset HD จริง

ให้ทำเฉพาะการสร้างชิ้นงานที่บันทึกได้ ทดสอบเส้นทางได้ และครูตรวจได้ก่อน

## 21. สรุป

ระบบสร้างชิ้นงานของบทที่ 2 ควรออกแบบให้ผู้เรียนสร้าง “โจทย์เส้นทางรถไถที่ทดสอบได้จริง” โดยมีแผนที่ จุดเริ่มต้น เป้าหมาย อุปสรรค ผลผลิต และคำสั่งเฉลย

ส่วนหน้าเลือกด่านต้องปรับให้ปุ่มเข้าห้องสร้างชิ้นงานมองเห็นง่ายขึ้น โดยเพิ่ม Completion Hero ด้านบน Sticky CTA ด้านล่าง และ Modal แจ้งเตือนหลังผ่านครบทุกด่าน เพื่อให้ผู้เรียนเข้าใจเส้นทางการเรียนรู้ต่อเนื่องจาก “เล่นเกม” ไปสู่ “สร้างชิ้นงาน” อย่างชัดเจน
