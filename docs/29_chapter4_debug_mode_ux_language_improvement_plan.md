# แผนการพัฒนาปรับปรุงเกมบทที่ 4  
## Smart Farm Debug Mode: ซ่อมกฎฟาร์มอัจฉริยะ  
### เน้นปรับ UX, ความสอดคล้องของข้อมูล และภาษาที่เข้าใจง่ายสำหรับเด็ก ป.4

เอกสารนี้จัดทำเพื่อใช้เป็นแผนสั่งงานและตรวจงานพัฒนาเกมบทที่ 4 หลังจากนำแนวทาง “ยกตัวเกมบทที่ 3 มาใช้เป็นฐาน แล้วปรับเป็นโหมดตรวจสอบและแก้ไขข้อผิดพลาด” ไปให้ Codex ดำเนินการแล้วพบปัญหาที่ต้องแก้ไขเพิ่มเติม

จุดเน้นของรอบนี้คือ **ไม่รื้อระบบใหม่** แต่ปรับให้บทที่ 4 ใช้งานง่ายขึ้น สื่อสารกับเด็กชัดขึ้น และข้อมูลตั้งแต่หน้า Dashboard จนถึงหน้าเล่นเกมตรงกับตัวเกมจริงทั้งหมด

---

# 1. สรุปสถานการณ์ปัจจุบัน

บทที่ 4 ได้เปลี่ยนแนวทางจากเกม Debugger แยกต่างหาก มาเป็นการใช้ฐานเกมสายพานของบทที่ 3 แล้วปรับเป็น **Debug Mode** หรือ “โหมดซ่อมกฎ” ซึ่งถือว่าเป็นแนวทางที่เหมาะสมกว่า เพราะเด็กคุ้นกับรูปแบบเกมจากบทที่ 3 อยู่แล้ว

แนวคิดหลักของบทที่ 4 คือ

```text
บทที่ 3: สร้างกฎให้ระบบฟาร์มทำงานถูกต้อง
บทที่ 4: ตรวจสอบกฎที่ผิด แล้วซ่อมให้ระบบฟาร์มกลับมาทำงานถูกต้อง
```

ดังนั้นบทที่ 4 ไม่ควรเป็นเพียงการเล่นซ้ำแบบบทที่ 3 แต่ต้องมีความแตกต่างชัดเจน คือ

1. มีกฎที่ผิดถูกใส่ไว้ตั้งแต่เริ่มต้น
2. ผู้เรียนต้องกดทดสอบกฎเดิมก่อน
3. ระบบแสดงผลลัพธ์ที่ผิดให้สังเกต
4. ผู้เรียนวิเคราะห์ว่าผิดตรงเงื่อนไข คำสั่ง หรือลำดับกฎ
5. ผู้เรียนแก้บล็อกกฎ
6. ผู้เรียนทดสอบซ้ำ
7. ผ่านด่านเมื่อระบบคัดแยกผลผลิตได้ถูกต้อง

# 2. ปัญหาที่พบหลังให้ Codex ดำเนินการ

จากการตรวจสอบรอบล่าสุด พบปัญหาหลัก 3 ประการ

## 2.1 แถบคำแนะนำ/สถานะใต้ “แผนผังซ่อมระบบ” เยอะเกินไป

บริเวณใต้หัวข้อ “แผนผังซ่อมระบบ” มีการแสดงคำแนะนำ สถานะ รายงาน หรือคำอธิบายหลายรายการพร้อมกัน ทำให้หน้าจอดูรก ยืดยาว และดึงความสนใจของเด็กออกจากภารกิจหลัก

ผลกระทบที่เกิดขึ้นคือ

- เด็กอ่านไม่หมด
- เด็กไม่รู้ว่าข้อความไหนสำคัญที่สุด
- พื้นที่เล่นเกมถูกดันลงล่าง
- หน้าเกมดูแน่นเกินไป
- บนมือถือยิ่งยาวและใช้งานยาก

## 2.2 ช่อง “ประเภทบั๊ก” ยังไม่เป็นภาษาไทย

ช่อง “ประเภทบั๊ก” บางกรณียังแสดงเป็น key ภาษาอังกฤษ เช่น

```text
wrong_condition
wrong_action
too_broad_condition
boundary_error
```

คำเหล่านี้เหมาะกับนักพัฒนา แต่ไม่เหมาะกับนักเรียนชั้น ป.4 เพราะเด็กไม่เข้าใจความหมาย และทำให้เกมดูเป็นระบบโปรแกรมมากเกินไป

## 2.3 ข้อมูลตั้งแต่ Dashboard ถึงหน้าเล่นเกมอาจไม่ตรงกับตัวเกมจริง

เนื่องจากบทที่ 4 เปลี่ยนแนวทางหลายรอบ จึงมีโอกาสที่ข้อมูลบางหน้าค้างจากแนวคิดเดิม เช่น

- ชื่อบทเรียน
- คำอธิบายบทเรียน
- ชื่อ Stage
- ชื่อด่านย่อย
- ข้อความบนปุ่ม
- ชื่อโหมด
- ข้อความใน Dashboard
- ข้อมูลในฐานข้อมูลหรือ seed SQL
- ข้อความในหน้าเลือกบทเรียน
- ข้อความในหน้าเลือกด่าน
- ข้อความในหน้าเล่นเกม

ปัญหานี้สำคัญมาก เพราะหากหน้า Dashboard บอกอย่างหนึ่ง แต่หน้าเล่นเกมแสดงอีกอย่าง เด็กจะสับสน และครูเองก็อธิบายการใช้สื่อได้ยาก

# 3. เป้าหมายของการปรับปรุงรอบนี้

## 3.1 เป้าหมายด้าน UX

1. ลดความรกของหน้าเกม
2. ทำให้ข้อมูลสำคัญมองเห็นชัด
3. แสดงรายงานข้อผิดพลาดแบบสั้นก่อน
4. ซ่อนรายละเอียดที่ยาวไว้ในปุ่ม “ดูรายละเอียด”
5. ทำให้หน้าเกมไม่ยืดยาวเกินไป
6. ใช้งานได้ดีทั้งคอมพิวเตอร์และมือถือ

## 3.2 เป้าหมายด้านภาษา

1. ใช้ภาษาไทยง่าย เหมาะกับเด็ก ป.4
2. ลดศัพท์เทคนิคภาษาอังกฤษบนหน้าจอ
3. แปลง key ทางเทคนิคให้เป็นคำไทย
4. ใช้ข้อความสั้น ชัดเจน และบอกให้เด็กทำอะไรต่อ
5. ใช้น้ำเสียงเป็นมิตร ไม่ตำหนิเด็กเมื่อทำผิด
6. ให้ feedback ที่ช่วยให้เด็กคิดต่อ

## 3.3 เป้าหมายด้านความสอดคล้องของข้อมูล

1. Dashboard แสดงชื่อบทที่ 4 ตรงกับหน้าเลือกบทเรียน
2. หน้าเลือกบทเรียนแสดงชื่อและคำอธิบายตรงกับหน้าเลือกด่าน
3. หน้าเลือกด่านแสดงชื่อ Stage 10–12 ตรงกับ config จริง
4. หน้าเล่นเกมแสดงชื่อด่านย่อยตรงกับเนื้อหาจริง
5. ไม่มีคำเก่าหรือชื่อระบบเก่าค้างอยู่ใน UI
6. ข้อมูลในฐานข้อมูลหรือไฟล์ seed ตรงกับเกมจริง

# 4. แนวทางปรับปรุงข้อ 1  
## แถบคำแนะนำ/สถานะใต้ “แผนผังซ่อมระบบ”

## 4.1 หลักการใหม่

ไม่ควรแสดงข้อมูลทั้งหมดพร้อมกันใต้แผนผังซ่อมระบบ เพราะทำให้รกและยาวเกินไป

ให้เปลี่ยนเป็นรูปแบบ

```text
สรุปสั้น ๆ ก่อน
รายละเอียดซ่อนไว้
เด็กกดดูเมื่อจำเป็น
```

รูปแบบที่แนะนำคือ

```text
รายงานการทดสอบ
พบของไปผิดทาง 3 จาก 8 ชิ้น
จุดที่ควรตรวจ: เงื่อนไขผิด
[ดูรายละเอียด]
```

เมื่อกด “ดูรายละเอียด” จึงแสดงข้อมูลเต็ม เช่น

```text
รายละเอียดที่พบ
- แครอทเปื้อนโคลนถูกปล่อยผ่าน
- แครอทสีเข้มแต่สะอาดถูกส่งเข้าเครื่องล้าง

กฎเดิมที่น่าสงสัย:
ถ้า แครอทสีเข้ม → ส่งเข้าเครื่องล้าง

เป้าหมาย:
ถ้า แครอทเปื้อนโคลน → ส่งเข้าเครื่องล้าง
```

## 4.2 รูปแบบ UI ที่แนะนำ

### แบบที่ 1: Accordion

เหมาะที่สุดสำหรับระบบปัจจุบัน เพราะทำง่าย ไม่กระทบ layout มาก

```text
┌─────────────────────────────────────┐
│ รายงานการทดสอบ                     │
│ พบของไปผิดทาง 3 จาก 8 ชิ้น          │
│ จุดที่ควรตรวจ: เงื่อนไขผิด           │
│ [ดูรายละเอียด]                      │
└─────────────────────────────────────┘
```

เมื่อกด “ดูรายละเอียด” กล่องจะขยายลงมา

ข้อดีคือทำง่ายและไม่ต้องสร้าง modal ใหม่

### แบบที่ 2: Modal

เหมาะกับข้อมูลยาวมาก เช่น รายการวัตถุผิดหลายรายการ

```text
[ดูรายละเอียด]
```

เมื่อกดแล้วเปิดหน้าต่างกลางจอ

ข้อดีคือไม่ดันพื้นที่เกมให้ยาว

### แบบที่ 3: Side Drawer

เหมาะกับหน้าจอคอมพิวเตอร์ แต่บนมือถืออาจต้องปรับมากกว่า

```text
คลิกดูรายละเอียด → แผงเลื่อนจากด้านขวา
```

ข้อดีคือดูทันสมัย แต่พัฒนายากกว่า accordion

## 4.3 แนวทางที่แนะนำให้ใช้จริง

ให้ใช้ **Accordion เป็นค่าเริ่มต้น** เพราะเหมาะกับระบบเดิมและพัฒนาง่ายที่สุด

หลักการคือ

1. รายงานย่อแสดงตลอด
2. รายละเอียดซ่อนด้วย class เช่น `.is-collapsed`
3. ปุ่มเปลี่ยนข้อความระหว่าง “ดูรายละเอียด” และ “ซ่อนรายละเอียด”
4. หากเป็นมือถือ รายละเอียดต้องไม่ยาวจนเกินไป
5. ถ้ารายการผิดมีมากกว่า 3 รายการ ให้แสดงแค่ 3 รายการแรก แล้วต่อท้ายว่า “และรายการอื่น ๆ”

## 4.4 ข้อความที่ควรแสดงในรายงานย่อ

รายงานย่อควรมีไม่เกิน 3 บรรทัด

```text
พบของไปผิดทาง 3 จาก 8 ชิ้น
จุดที่ควรตรวจ: เงื่อนไขผิด
ลองซ่อมกฎ แล้วทดสอบอีกครั้ง
```

หรือกรณีแก้ถูกแล้ว

```text
ซ่อมสำเร็จ!
ผลผลิตทั้งหมดไปถูกทางแล้ว
ไปด่านถัดไปกันเลย
```

## 4.5 สิ่งที่ไม่ควรแสดงในรายงานย่อ

ไม่ควรแสดงข้อมูลเหล่านี้พร้อมกันในหน้าแรก

- รายการวัตถุผิดทุกชิ้น
- กฎเสียทุกบรรทัด
- expectedLogic เต็มชุด
- brokenLogic เต็มชุด
- key ภาษาอังกฤษ
- คำอธิบายเชิงเทคนิคยาว ๆ
- JSON หรือ object ดิบ
- คำว่า debug แบบไม่มีคำอธิบาย

ข้อมูลเหล่านี้ให้เก็บไว้ใน “ดูรายละเอียด” เท่านั้น

# 5. แนวทางปรับปรุงข้อ 2  
## ช่อง “ประเภทบั๊ก” ให้เป็นภาษาไทย

## 5.1 หลักการ

ทุกข้อความที่นักเรียนเห็นบนหน้าจอต้องเป็นภาษาไทยที่เข้าใจง่าย หากยังจำเป็นต้องใช้ key ภาษาอังกฤษใน code ให้เก็บไว้เฉพาะภายใน logic เท่านั้น

## 5.2 Mapping ประเภทบั๊ก

ให้เพิ่มฟังก์ชันแปลง `bugType` เป็นภาษาไทย

| bugType ในระบบ | ข้อความภาษาไทยสำหรับ UI | คำอธิบายสำหรับเด็ก |
|---|---|---|
| wrong_condition | เงื่อนไขผิด | ระบบตรวจของผิดชนิด |
| wrong_action | คำสั่งผิด | ระบบตรวจเจอแล้ว แต่ส่งไปผิดที่ |
| too_broad_condition | เงื่อนไขกว้างเกินไป | ระบบจับของที่ไม่ควรถูกจับด้วย |
| too_narrow_condition | เงื่อนไขแคบเกินไป | ระบบจับของที่ควรจับได้ไม่ครบ |
| wrong_else_action | คำสั่งกรณีอื่นผิด | ของที่ไม่เข้าเงื่อนไขถูกส่งผิดทาง |
| wrong_order | ลำดับกฎผิด | ระบบตรวจเงื่อนไขผิดลำดับ |
| boundary_error | ขอบเขตตัวเลขผิด | ตั้งค่ามากกว่า/น้อยกว่าผิด |
| missing_condition | ขาดเงื่อนไข | ยังไม่ได้บอกให้ระบบตรวจบางกรณี |
| missing_action | ขาดคำสั่ง | ตรวจเจอแล้ว แต่ยังไม่ได้บอกให้ทำอะไร |
| extra_rule | มีกฎเกินจำเป็น | มีกฎที่ไม่จำเป็น ทำให้ระบบสับสน |

## 5.3 ตัวอย่างฟังก์ชันที่ควรเพิ่ม

```javascript
function bugTypeLabel(type) {
  const labels = {
    wrong_condition: 'เงื่อนไขผิด',
    wrong_action: 'คำสั่งผิด',
    too_broad_condition: 'เงื่อนไขกว้างเกินไป',
    too_narrow_condition: 'เงื่อนไขแคบเกินไป',
    wrong_else_action: 'คำสั่งกรณีอื่นผิด',
    wrong_order: 'ลำดับกฎผิด',
    boundary_error: 'ขอบเขตตัวเลขผิด',
    missing_condition: 'ขาดเงื่อนไข',
    missing_action: 'ขาดคำสั่ง',
    extra_rule: 'มีกฎเกินจำเป็น'
  };

  return labels[type] || 'บั๊กของกฎระบบ';
}
```

## 5.4 จุดที่ต้องใช้ฟังก์ชันนี้

ให้ใช้กับทุกจุดที่แสดงประเภทบั๊ก เช่น

```text
mission-bug-type
debug-report-summary
debug-report-detail
stage summary
hint panel
teacher/admin preview
```

ห้ามแสดง key ดิบ เช่น `wrong_condition` บน UI ที่เด็กเห็น

# 6. แนวทางปรับปรุงข้อ 3  
## ตรวจความสอดคล้องจาก Dashboard ถึงหน้าเล่นเกม

## 6.1 ชื่อบทที่ 4 ที่ควรใช้ให้ตรงกัน

ให้ใช้ชื่อหลักเดียวกันทุกหน้า

```text
บทที่ 4: ตรวจสอบและแก้ไขข้อผิดพลาด
```

หากเป็นหน้าเล่นเกม สามารถเพิ่มชื่อธีมได้ว่า

```text
ซ่อมกฎฟาร์มอัจฉริยะ
```

รูปแบบเต็มที่แนะนำคือ

```text
บทที่ 4: ตรวจสอบและแก้ไขข้อผิดพลาด
ซ่อมกฎฟาร์มอัจฉริยะ
```

## 6.2 คำอธิบายบทที่ 4

ให้ใช้คำอธิบายเดียวกันหรือใกล้เคียงกันทุกหน้า

```text
ฝึกตรวจสอบกฎที่ผิดพลาด ทดลองรันระบบ สังเกตผลลัพธ์ที่ผิด แล้วแก้ไขเงื่อนไขและคำสั่งให้ระบบฟาร์มทำงานถูกต้อง
```

ถ้าต้องการข้อความสั้นสำหรับการ์ดบน Dashboard ให้ใช้

```text
ทดสอบระบบฟาร์ม หาเหตุที่ผิด แล้วซ่อมกฎให้ถูกต้อง
```

## 6.3 ชื่อ Stage ที่ควรใช้

| Stage | ชื่อที่ควรแสดง | คำอธิบายสั้น |

# 8. ตัวอย่างภาษาที่ควรใช้ในแต่ละช่วงของเกม

## 8.1 ตอนเริ่มด่าน

ใช้ข้อความสั้น ๆ อธิบายภารกิจ

```text
ระบบฟาร์มมีกฎบางอย่างผิดอยู่
ลองทดสอบกฎเดิม แล้วช่วยซ่อมให้ถูกต้อง
```

หรือ

```text
ภารกิจของเธอคือช่วยซ่อมกฎของระบบฟาร์ม
เริ่มจากกด “ทดสอบกฎเดิม” เพื่อดูว่าเกิดอะไรขึ้น
```

## 8.2 ตอนยังไม่ได้ทดสอบกฎเดิม

```text
กดปุ่ม “ทดสอบกฎเดิม” ก่อน
แล้วสังเกตว่าผลผลิตชิ้นไหนไปผิดทาง
```

## 8.3 ตอนพบข้อผิดพลาด

```text
พบของไปผิดทาง 3 ชิ้น
ลองดูว่ากฎข้อไหนทำให้ระบบส่งของผิด
```

## 8.4 ตอนเด็กเลือกบล็อกผิด

```text
ยังไม่ใช่จุดนี้
ลองดูผลผลิตที่ไปผิดทางอีกครั้ง
```

หรือ

```text
ใกล้แล้ว!
กฎที่ต้องซ่อมเกี่ยวกับเงื่อนไขของผลผลิตชิ้นนี้
```

## 8.5 ตอนเด็กแก้ถูก

```text
ซ่อมสำเร็จ!
ระบบฟาร์มส่งผลผลิตถูกทางแล้ว
```

## 8.6 ตอนขอคำใบ้

คำใบ้ควรมี 3 ระดับ

### คำใบ้ระดับ 1: ชี้ให้สังเกต

```text
ลองดูว่าผลผลิตที่ผิดมีลักษณะอะไรเหมือนกัน
```

### คำใบ้ระดับ 2: ชี้จุดที่ควรตรวจ

```text
จุดที่ควรตรวจอยู่ที่เงื่อนไขของกฎ IF
```

### คำใบ้ระดับ 3: บอกแนวทางซ่อม

```text
กฎควรตรวจคำว่า “เปื้อนโคลน” ไม่ใช่ “สีเข้ม”
```

# 14. Acceptance Criteria

## 14.1 ด้าน UX

- รายงานใต้แผนผังซ่อมระบบแสดงแบบย่อ

- ปุ่มกดง่าย
- ไม่มีข้อมูลเทคนิคยาว ๆ แสดงทันที

- ประเภทบั๊กเป็นภาษาไทย
- ปุ่มหลักเป็นภาษาไทยที่เข้าใจง่าย
- Feedback เป็นมิตร
- คำใบ้ช่วยให้คิดต่อ
- ข้อความแต่ละจุดไม่ยาวเกินไป

## 14.3 ด้านความสอดคล้องของข้อมูล

- ฐานข้อมูล/seed ไม่ขัดกับ UI

- `window.sendResult()` ยังทำงาน
- game logs ยังบันทึกได้
- stage 10–12 โหลดได้
- ไม่มี error ใน console

- responsive ยังใช้งานได้
