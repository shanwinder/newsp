# แผนการพัฒนาระบบให้รองรับ Multi-School ผ่านระบบสมัครสมาชิก

> เอกสารนี้เป็นแผนการพัฒนาระบบ “เกมแบบฝึกทักษะออนไลน์ เรื่อง การแก้ปัญหาอย่างเป็นขั้นตอน สำหรับนักเรียนชั้นประถมศึกษาปีที่ 4” ให้สามารถรองรับการใช้งานจากครูและโรงเรียนหลายแห่งได้อย่างเป็นระบบ ปลอดภัย และสามารถขยายผลได้จริงในอนาคต

---

## 1. ที่มาและความจำเป็น

ระบบเดิมถูกออกแบบในลักษณะเหมาะกับการใช้งานภายในห้องเรียนหรือโรงเรียนเดียว โดยผู้สร้างสื่อเป็นผู้จัดการข้อมูลนักเรียน กำหนดการเข้าเล่น ควบคุมด่าน ตรวจชิ้นงาน และดูผลคะแนนของนักเรียนในระบบ

อย่างไรก็ตาม เมื่อผลงานถูกพัฒนาเป็นสื่อเทคโนโลยีดิจิทัลสำหรับเผยแพร่ผ่าน OBEC Content Center และอาจมีครูจากหลากหลายโรงเรียนสนใจนำไปใช้ ระบบจำเป็นต้องได้รับการปรับปรุงให้รองรับผู้ใช้หลายโรงเรียนอย่างเหมาะสม โดยต้องแยกข้อมูลของแต่ละโรงเรียน ครู ห้องเรียน และนักเรียนออกจากกันอย่างชัดเจน

การปรับปรุงระบบในระยะนี้จึงมีเป้าหมายเพื่อเปลี่ยนจากระบบแบบ “โรงเรียนเดียว/ห้องเรียนเดียว” ไปสู่ระบบแบบ **Multi-School Controlled Access** หรือระบบที่รองรับหลายโรงเรียนภายใต้การควบคุมสิทธิ์และการอนุมัติของผู้ดูแลระบบ

---

## 2. สภาพปัจจุบันของระบบ

จากการทบทวนโครงสร้างระบบปัจจุบัน พบว่าระบบมีจุดแข็งหลายประการ เช่น

- มีระบบผู้ใช้พื้นฐาน แบ่งบทบาทนักเรียนและผู้ดูแล
- มีระบบเข้าสู่ระบบแบบนักเรียนรายบุคคลและแบบกลุ่ม
- มี Teacher Dashboard สำหรับควบคุมการเข้าเล่นและติดตามนักเรียน
- มีระบบบันทึกคะแนน ความก้าวหน้า เวลา และจำนวนครั้งที่พยายาม
- มีระบบล็อก/ปลดล็อกด่าน
- มีระบบสร้างชิ้นงานหลังผ่านด่าน
- มีระบบตรวจชิ้นงานและส่งข้อเสนอแนะกลับให้นักเรียน

แต่ยังมีข้อจำกัดสำคัญเมื่อต้องรองรับหลายโรงเรียน ได้แก่

### 2.1 ข้อมูลผู้ใช้ยังไม่แยกตามโรงเรียน

ตารางผู้ใช้เดิมมีข้อมูลนักเรียนและผู้ดูแล แต่ยังไม่มีโครงสร้างสำหรับแยกโรงเรียน ครู ห้องเรียน หรือรอบการใช้งาน ทำให้หากเปิดให้หลายโรงเรียนใช้งาน ข้อมูลนักเรียนอาจปะปนกัน

### 2.2 Dashboard ดึงข้อมูลนักเรียนรวมทั้งระบบ

ระบบติดตามนักเรียนปัจจุบันมีแนวโน้มดึงนักเรียนทั้งหมดที่มีบทบาทเป็น student โดยไม่ได้กรองตามโรงเรียน ครู หรือห้องเรียน ส่งผลให้ครูแต่ละโรงเรียนอาจเห็นข้อมูลนักเรียนของโรงเรียนอื่น

### 2.3 การล็อก/ปลดล็อกด่านเป็นค่ากลางของระบบ

สถานะควบคุมห้องเรียน เช่น การอนุญาตให้เข้าเล่น หรือการล็อก/ปลดล็อกด่าน ยังเป็นค่ากลางของระบบ หากครูหลายโรงเรียนใช้พร้อมกัน การควบคุมของครูคนหนึ่งอาจกระทบผู้ใช้ของโรงเรียนอื่น

### 2.4 คะแนนยังไม่ผูกกับห้องเรียนหรือรอบการใช้งาน

ตารางคะแนนสามารถบันทึกผลรายผู้เรียนและรายด่านได้ แต่ยังไม่มีบริบทว่าเป็นคะแนนของโรงเรียนใด ห้องใด ครูใด หรือรอบการทดลองใด จึงไม่เหมาะต่อการวิเคราะห์ผลแบบหลายโรงเรียน

### 2.5 ยังไม่มีระบบสมัครสมาชิกสำหรับครู

หากครูจากโรงเรียนอื่นต้องการใช้งาน ยังไม่มีระบบสมัครสมาชิก สร้างโรงเรียน สร้างห้องเรียน เพิ่มนักเรียน และจัดการข้อมูลของตนเอง

## 3. เป้าหมายการพัฒนา

### 3.1 เป้าหมายหลัก

พัฒนาระบบให้รองรับครูและโรงเรียนหลายแห่ง โดยให้ครูสามารถสมัครสมาชิก สร้างห้องเรียน เพิ่มนักเรียน จัดกิจกรรมการเรียนรู้ ติดตามผล และส่งออกข้อมูลได้ โดยข้อมูลของแต่ละโรงเรียนต้องแยกจากกันอย่างปลอดภัย

### 3.2 เป้าหมายเชิงระบบ

1. ครูจากโรงเรียนต่าง ๆ สามารถสมัครสมาชิกได้
2. ผู้ดูแลระบบสามารถอนุมัติหรือระงับบัญชีครู/โรงเรียนได้
3. ครูสามารถสร้างห้องเรียนของตนเองได้
4. ครูสามารถเพิ่มนักเรียนรายห้องเรียนได้
5. นักเรียนสามารถเข้าใช้งานภายใต้ห้องเรียนของตนเองได้
6. ครูเห็นเฉพาะข้อมูลนักเรียนในโรงเรียน/ห้องเรียนของตนเอง
7. การล็อก/ปลดล็อกด่านต้องเป็นรายห้องเรียน ไม่ใช่ค่ากลางทั้งระบบ
8. คะแนนและความก้าวหน้าต้องผูกกับโรงเรียน ห้องเรียน ครู และรอบการเรียนรู้
9. ระบบสามารถส่งออกข้อมูลผลการเรียนรู้เป็น CSV/Excel ได้
10. มีบัญชี Demo สำหรับเผยแพร่ใน OBEC Content Center โดยไม่กระทบข้อมูลจริง

## 5. แนวคิดการออกแบบระบบใหม่

ระบบใหม่ควรออกแบบเป็นโครงสร้างลำดับชั้นดังนี้

```text
Super Admin
  └── School
        └── Teacher
              └── Classroom
                    └── Student
                          └── Progress / Logs / Works
```

หลักสำคัญคือ ข้อมูลทุกส่วนที่เกี่ยวข้องกับผู้เรียนต้องผูกกับบริบทอย่างน้อย 4 ระดับ ได้แก่

```text
school_id
teacher_id
classroom_id
learning_session_id
```

เพื่อให้สามารถแยกข้อมูล ป้องกันข้อมูลปะปน และนำข้อมูลไปวิเคราะห์ผลการใช้งานได้ถูกต้อง

## 6. บทบาทผู้ใช้ในระบบใหม่

### 6.1 Super Admin

ผู้ดูแลระบบหลักหรือผู้สร้างสื่อ มีหน้าที่

- ดูภาพรวมทุกโรงเรียน
- อนุมัติ/ระงับบัญชีครู
- จัดการข้อมูลโรงเรียน
- จัดการเนื้อหาเกมและด่าน
- ตรวจสอบข้อมูล Demo
- ดูสถิติภาพรวมของระบบ
- ดูแลความปลอดภัยและความถูกต้องของข้อมูล

### 6.2 Teacher / School Admin

ครูผู้สมัครใช้งานระบบ มีหน้าที่

- สมัครสมาชิก
- กรอกข้อมูลโรงเรียน
- สร้างห้องเรียน
- เพิ่มนักเรียนหรือนำเข้า CSV
- กำหนด PIN/รหัสผ่านให้นักเรียน
- ควบคุมสถานะการเรียนในห้อง
- ล็อก/ปลดล็อกด่านรายห้องเรียน
- ตรวจชิ้นงาน
- ให้ข้อเสนอแนะ
- ดูคะแนนและความก้าวหน้าของนักเรียน
- ส่งออกข้อมูลผลการเรียนรู้

### 6.3 Student

นักเรียน มีหน้าที่

- เข้าสู่ระบบผ่านห้องเรียนของตนเอง
- เล่นเกมแบบฝึกทักษะออนไลน์
- ทำกิจกรรมตามด่าน
- สะสมคะแนนหรือดาว
- สร้างชิ้นงานหลังผ่านกิจกรรม
- อ่านข้อเสนอแนะจากครูและปรับปรุงงาน

### 6.4 Demo User

บัญชีทดลองสำหรับกรรมการหรือครูผู้สนใจ มีหน้าที่

- ทดลองเข้าสู่ระบบ
- ทดลองเล่นเกม
- ทดลองดูหน้าครู/นักเรียนบางส่วน
- ไม่สามารถลบหรือแก้ไขข้อมูลจริง
- ใช้ข้อมูลสมมติเท่านั้น

## 7. โครงสร้างฐานข้อมูลที่ควรเพิ่มหรือปรับปรุง

## 7.1 ตาราง `schools`

ใช้เก็บข้อมูลโรงเรียน

```sql
CREATE TABLE schools (
    id INT AUTO_INCREMENT PRIMARY KEY,
    school_code VARCHAR(50) NULL,
    school_name VARCHAR(255) NOT NULL,
    district VARCHAR(150) NULL,
    province VARCHAR(150) NULL,
    affiliation VARCHAR(255) NULL,
    contact_name VARCHAR(255) NULL,
    contact_email VARCHAR(255) NULL,
    contact_phone VARCHAR(50) NULL,
    status ENUM('pending','approved','suspended') DEFAULT 'pending',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NULL
);
```

### สถานะที่ใช้

| สถานะ | ความหมาย |
|---|---|
| pending | รอตรวจสอบ/รออนุมัติ |
| approved | ได้รับอนุมัติให้ใช้งาน |
| suspended | ระงับการใช้งาน |

---

## 7.2 ตาราง `classrooms`

ใช้เก็บข้อมูลห้องเรียนหรือกลุ่มเรียนของครู

```sql
CREATE TABLE classrooms (
    id INT AUTO_INCREMENT PRIMARY KEY,
    school_id INT NOT NULL,
    teacher_id INT NOT NULL,
    classroom_name VARCHAR(255) NOT NULL,
    grade_level VARCHAR(50) DEFAULT 'ป.4',
    academic_year VARCHAR(20) NULL,
    join_code VARCHAR(20) NOT NULL UNIQUE,
    status ENUM('active','archived') DEFAULT 'active',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NULL,
    FOREIGN KEY (school_id) REFERENCES schools(id),
    FOREIGN KEY (teacher_id) REFERENCES users(user_id)
);
```

### ตัวอย่างชื่อห้องเรียน

- ป.4 โรงเรียนบ้านนาอุดม ปี 2569
- ป.4/1 ภาคเรียนที่ 1
- กลุ่มทดลองใช้สื่อ
- ห้องเรียน Demo สำหรับครูทดลอง

---

## 7.3 ตาราง `learning_sessions`

ใช้แยกรอบการใช้สื่อหรือรอบการทดลอง

```sql
CREATE TABLE learning_sessions (
    id INT AUTO_INCREMENT PRIMARY KEY,
    school_id INT NOT NULL,
    classroom_id INT NOT NULL,
    teacher_id INT NOT NULL,
    session_name VARCHAR(255) NOT NULL,
    class_status ENUM('active','paused') DEFAULT 'active',
    navigation_status ENUM('locked','unlocked') DEFAULT 'locked',
    start_date DATE NULL,
    end_date DATE NULL,
    status ENUM('active','closed','archived') DEFAULT 'active',
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    updated_at DATETIME NULL,
    FOREIGN KEY (school_id) REFERENCES schools(id),
    FOREIGN KEY (classroom_id) REFERENCES classrooms(id),
    FOREIGN KEY (teacher_id) REFERENCES users(user_id)
);
```

### เหตุผลที่ต้องมีตารางนี้

- แยกข้อมูลแต่ละรอบการสอน
- ป้องกันคะแนนเก่าปนกับคะแนนใหม่
- ใช้สำหรับการทดลองสื่อและการคำนวณ E1/E2
- ควบคุมสถานะห้องเรียนรายรอบ
- ใช้ล็อก/ปลดล็อกด่านเฉพาะห้องเรียน

---

## 7.4 ปรับตาราง `users`

ตารางผู้ใช้เดิมควรเพิ่มข้อมูลเพื่อรองรับหลายโรงเรียน

```sql
ALTER TABLE users
ADD COLUMN school_id INT NULL,
ADD COLUMN classroom_id INT NULL,
ADD COLUMN teacher_id INT NULL,
ADD COLUMN email VARCHAR(255) NULL,
ADD COLUMN status ENUM('pending','active','suspended') DEFAULT 'active',
ADD COLUMN approved_at DATETIME NULL,
ADD COLUMN approved_by INT NULL;
```

### ปรับบทบาทผู้ใช้

จากเดิมที่มีเพียง `student` และ `admin` ควรปรับเป็น

| role | ความหมาย |
|---|---|
| super_admin | ผู้ดูแลระบบหลัก |
| teacher | ครูผู้ใช้งาน |
| student | นักเรียน |
| demo | ผู้ใช้ทดลอง |

หมายเหตุ: หากต้องการคงโครงสร้างเดิมไว้ก่อน อาจใช้ `admin` แทน `super_admin` ในระยะเปลี่ยนผ่าน แต่ควรออกแบบให้รองรับ role ใหม่ในอนาคต

---

## 7.5 ปรับตาราง `progress`

ตารางคะแนนควรเพิ่มบริบทของโรงเรียน ห้องเรียน ครู และรอบการเรียนรู้

```sql
ALTER TABLE progress
ADD COLUMN school_id INT NULL,
ADD COLUMN classroom_id INT NULL,
ADD COLUMN teacher_id INT NULL,
ADD COLUMN learning_session_id INT NULL;
```

### หลักการบันทึกคะแนนใหม่

ทุกครั้งที่บันทึกคะแนน ต้องบันทึกข้อมูลต่อไปนี้พร้อมกัน

- user_id
- stage_id
- score
- duration_seconds
- attempts
- school_id
- classroom_id
- teacher_id
- learning_session_id

ADD COLUMN teacher_id INT NULL,

## 8. Workflow การสมัครสมาชิก

## 8.1 ขั้นตอนสมัครของครู

1. ครูเข้าหน้า “สมัครใช้งานสำหรับครู”
2. กรอกข้อมูลส่วนตัว
   - ชื่อ-นามสกุล
   - Email
   - เบอร์โทรศัพท์
   - รหัสผ่าน
3. กรอกข้อมูลโรงเรียน
   - ชื่อโรงเรียน
   - อำเภอ
   - จังหวัด
   - สังกัด
4. ระบุระดับชั้นหรือวัตถุประสงค์การใช้งาน
5. ส่งคำขอสมัครสมาชิก
6. ระบบสร้างบัญชีสถานะ `pending`
7. แสดงข้อความว่า “รอผู้ดูแลระบบอนุมัติ”

---

## 8.2 ขั้นตอนอนุมัติโดย Super Admin

1. ผู้ดูแลระบบเข้าสู่หน้า Super Admin Dashboard
2. ดูรายการครู/โรงเรียนที่รออนุมัติ
3. ตรวจสอบข้อมูล
4. เลือกอย่างใดอย่างหนึ่ง
   - อนุมัติ
   - ส่งกลับให้แก้ไขข้อมูล
   - ปฏิเสธ
   - ระงับ
5. เมื่ออนุมัติ สถานะครูเปลี่ยนเป็น `active`
6. ครูเข้าสู่ระบบและเริ่มสร้างห้องเรียนได้

---

## 8.3 ขั้นตอนสร้างห้องเรียน

1. ครูเข้าสู่ระบบ
2. เลือกเมนู “ห้องเรียนของฉัน”
3. กด “สร้างห้องเรียน”
4. กรอกข้อมูล
   - ชื่อห้องเรียน
   - ระดับชั้น
   - ปีการศึกษา
   - ชื่อรอบการใช้งาน
5. ระบบสร้าง `join_code`
6. ครูนำรหัสนี้ไปใช้กับนักเรียนหรือสร้างบัญชีนักเรียนให้

---

## 8.4 ขั้นตอนเพิ่มนักเรียน

ควรมี 2 วิธี

### วิธีที่ 1 เพิ่มทีละคน

ครูกรอก

- รหัสนักเรียน
- ชื่อ-นามสกุล
- ชั้นเรียน
- PIN/รหัสผ่าน

### วิธีที่ 2 นำเข้า CSV

รูปแบบไฟล์ CSV ที่แนะนำ

```csv
student_id,name,class_level,pin
4001,เด็กชายตัวอย่าง หนึ่ง,ป.4,1234
4002,เด็กหญิงตัวอย่าง สอง,ป.4,1234
```

หลังนำเข้า ระบบต้องแสดงผลว่า

- นำเข้าสำเร็จกี่คน
- แถวใดมีปัญหา
- มีรหัสซ้ำหรือไม่
- ข้อมูลไม่ครบหรือไม่

---

## 8.5 ขั้นตอนเข้าใช้งานของนักเรียน

นักเรียนควรเข้าใช้งานโดยใช้ข้อมูลดังนี้

- รหัสห้องเรียน หรือ Join Code
- รหัสนักเรียน/ชื่อผู้เรียน
- PIN

ตัวอย่างขั้นตอน

1. นักเรียนเข้าหน้าเข้าสู่ระบบ
2. กรอก Join Code ของห้องเรียน
3. เลือกชื่อหรือกรอกรหัสนักเรียน
4. กรอก PIN
5. ระบบตรวจสอบว่านักเรียนอยู่ในห้องเรียนนั้นจริง
6. ระบบตั้งค่า session ของนักเรียน เช่น school_id, classroom_id, teacher_id, learning_session_id
7. เข้าสู่หน้าเกม

## 9. การปรับปรุง Teacher Dashboard

Dashboard เดิมควรปรับให้เป็น Dashboard รายห้องเรียน โดยมีโครงสร้างใหม่ดังนี้

### 9.1 หน้าแรกของครู

แสดง

- ชื่อครู
- โรงเรียน
- จำนวนห้องเรียน
- จำนวนผู้เรียนทั้งหมด
- ห้องเรียนล่าสุด
- ปุ่มสร้างห้องเรียน
- ปุ่มนำเข้านักเรียน
- ปุ่มดูรายงานผล

### 9.2 หน้าเลือกห้องเรียน

ครูต้องเลือกห้องเรียนก่อนจึงจะดูข้อมูลนักเรียนได้

ข้อมูลที่แสดง

- ชื่อห้องเรียน
- ระดับชั้น
- ปีการศึกษา
- Join Code
- จำนวนนักเรียน
- สถานะห้องเรียน
- ปุ่มจัดการ

### 9.3 Dashboard รายห้องเรียน

แสดงเฉพาะข้อมูลของห้องเรียนนั้น

- รายชื่อนักเรียน
- สถานะ online/offline
- คะแนนรวม
- คะแนนรายหน่วย
- ด่านที่ผ่านแล้ว
- ด่านที่ยังไม่ผ่าน
- ชิ้นงานที่รอตรวจ
- ปุ่มล็อก/ปลดล็อกด่าน
- ปุ่ม pause/resume
- ปุ่มส่งออกคะแนน

### 9.4 หลักการกรองข้อมูล

ทุก query ที่เกี่ยวข้องกับนักเรียน คะแนน log หรือชิ้นงาน ต้องมีเงื่อนไขอย่างน้อย

```sql
WHERE school_id = ?
AND classroom_id = ?
AND teacher_id = ?
```

หรือในกรณีใช้ `learning_session_id`

```sql
WHERE learning_session_id = ?
```

ห้ามใช้ query แบบดึงนักเรียนทั้งหมดในระบบโดยกรองเพียง `role = 'student'`

## 10. การปรับปรุงระบบล็อก/ปลดล็อกด่าน

ระบบเดิมใช้ค่ากลาง เช่น `class_status` และ `navigation_status` ในตาราง system settings ซึ่งไม่เหมาะกับหลายโรงเรียน

ระบบใหม่ควรย้ายสถานะเหล่านี้ไปไว้ใน `learning_sessions` หรือ `classroom_settings`

### แนวทางที่แนะนำ

ใช้ตาราง `learning_sessions`

```text
learning_sessions.class_status
learning_sessions.navigation_status
```

เมื่อครูห้อง A กดล็อกด่าน จะกระทบเฉพาะ session ของห้อง A เท่านั้น ไม่กระทบห้อง B หรือโรงเรียนอื่น

## 11. การปรับปรุงระบบคะแนน

ระบบคะแนนใหม่ควรรองรับการวิเคราะห์ผลได้ 3 ระดับ

### 11.1 ระดับนักเรียน

- คะแนนรวม
- คะแนนรายด่าน
- เวลาที่ใช้
- จำนวนครั้งที่พยายาม
- ด่านที่ยังไม่ผ่าน
- ชิ้นงานที่ส่งแล้ว

### 11.2 ระดับห้องเรียน

- คะแนนเฉลี่ยทั้งห้อง
- ร้อยละนักเรียนที่ผ่านแต่ละด่าน
- ค่าเฉลี่ยเวลาในการเล่น
- จำนวนครั้งเฉลี่ยที่พยายาม
- รายชื่อนักเรียนที่ต้องช่วยเหลือ

### 11.3 ระดับโรงเรียน/ภาพรวม

- จำนวนห้องเรียนที่ใช้งาน
- จำนวนนักเรียนทั้งหมด
- ค่าเฉลี่ยคะแนนรวม
- สถิติการใช้งานรายโรงเรียน
- จำนวนครูที่ใช้งานจริง

## 14. แผนการพัฒนาเป็นระยะ

- สำรองฐานข้อมูลก่อนปรับโครงสร้าง

### ผลลัพธ์ที่ต้องได้

- ฐานข้อมูลรองรับหลายโรงเรียน
- ข้อมูลเดิมไม่สูญหาย
- ระบบเดิมยังสามารถใช้งานได้ในโหมดโรงเรียนเริ่มต้น

## ระยะที่ 3: พัฒนาระบบสมัครสมาชิกครู

### งานที่ต้องทำ

- สร้างหน้า register teacher

- สร้างหน้าแจ้งรออนุมัติ

### ผลลัพธ์ที่ต้องได้

- ครูสมัครสมาชิกได้

- ครูที่ยังไม่อนุมัติไม่สามารถเข้าจัดการห้องเรียนได้

## ระยะที่ 4: พัฒนาระบบห้องเรียน

### งานที่ต้องทำ

### ผลลัพธ์ที่ต้องได้

- ครูสร้างห้องเรียนได้เอง
- ห้องเรียนผูกกับโรงเรียนและครู

- ระบบพร้อมรองรับการจัดกิจกรรมแยกห้อง

## ระยะที่ 5: พัฒนาระบบจัดการนักเรียนรายห้อง

### งานที่ต้องทำ

- ปรับหน้าเพิ่มนักเรียนให้ผูกกับ classroom_id
- ปรับระบบ import CSV ให้เพิ่มนักเรียนในห้องที่เลือก
- ป้องกันรหัสนักเรียนซ้ำในห้องเดียวกัน
- สร้างระบบ reset PIN
- สร้างบัตรรหัสนักเรียนสำหรับพิมพ์แจก

### ผลลัพธ์ที่ต้องได้

- ครูเพิ่มนักเรียนเฉพาะห้องของตนเองได้
- ข้อมูลนักเรียนไม่ปนกับโรงเรียนอื่น
- นักเรียนเข้าสู่ระบบด้วย join code และ PIN ได้

---

## ระยะที่ 6: ปรับระบบ Login และ Session

### งานที่ต้องทำ

- ปรับหน้า login ให้รองรับครูและนักเรียน
- นักเรียนต้องระบุ join code ก่อนเข้าสู่ระบบ
- เมื่อ login สำเร็จ ต้องบันทึก session:
  - user_id
  - role
  - school_id
  - classroom_id
  - teacher_id
  - learning_session_id
- ป้องกันการเข้าถึงข้อมูลข้ามห้องเรียน
- ปรับ logout ให้ล้าง session ครบถ้วน

### ผลลัพธ์ที่ต้องได้

- ผู้ใช้แต่ละคนเข้าสู่ระบบตามบริบทของตนเอง
- ครูเห็นเฉพาะข้อมูลของตนเอง
- นักเรียนเห็นเฉพาะห้องเรียนและคะแนนของตนเอง

---

## ระยะที่ 7: ปรับ Dashboard และ API ให้กรองข้อมูล

### งานที่ต้องทำ

- ปรับ Teacher Dashboard ให้เลือกห้องเรียนก่อน
- ปรับ API get_students_monitor ให้กรองด้วย classroom_id

- ปรับระบบตรวจชิ้นงานให้กรองตามห้องเรียน
- ปรับระบบลบ/แก้ไขนักเรียนให้ตรวจสิทธิ์ก่อนทุกครั้ง
- ปรับระบบคะแนนรวมให้คิดเฉพาะบริบทของห้องเรียน

### ผลลัพธ์ที่ต้องได้

- Dashboard แสดงข้อมูลถูกต้องเฉพาะห้อง
- API ไม่เปิดเผยข้อมูลข้ามโรงเรียน
- คะแนนและ log ถูกบันทึกพร้อมบริบทครบถ้วน

## ระยะที่ 8: ปรับระบบล็อก/ปลดล็อกด่านรายห้องเรียน

- ย้าย class_status และ navigation_status ไปผูกกับ learning_session

- ปรับ check_nav_status ให้ตรวจจาก session ของห้องเรียนนั้น
- ปรับ student side ให้รับสถานะจาก session ของตนเอง
- ทดสอบหลายห้องเรียนพร้อมกัน

### ผลลัพธ์ที่ต้องได้

- ครูแต่ละห้องควบคุมด่านของตนเองได้
- การล็อกด่านของห้องหนึ่งไม่กระทบอีกห้อง
- ใช้งานพร้อมกันหลายโรงเรียนได้ปลอดภัยขึ้น

## ระยะที่ 9: พัฒนาระบบรายงานและ Export

- สร้างรายงานคะแนนรายด่าน
- สร้างรายงานนักเรียนที่ยังไม่ผ่าน

## 16. หลักการด้านความปลอดภัยและข้อมูลส่วนบุคคล

การรองรับหลายโรงเรียนทำให้ระบบเกี่ยวข้องกับข้อมูลนักเรียนจำนวนมากขึ้น จึงต้องให้ความสำคัญกับความปลอดภัย

### 16.1 หลักการสำคัญ

- ครูต้องเห็นเฉพาะนักเรียนของตนเอง
- นักเรียนต้องเห็นเฉพาะข้อมูลของตนเอง
- Super Admin เท่านั้นที่ดูข้อมูลภาพรวมได้
- ห้ามใช้ข้อมูลนักเรียนจริงในระบบ Demo
- ห้ามเผยแพร่ SQL dump ที่มีชื่อจริงนักเรียน
- ต้องใช้ `password_hash()` และ `password_verify()` สำหรับรหัสผ่าน
- ทุกหน้าที่สำคัญต้องตรวจ session และ role
- ทุก query ที่แก้ไข/ลบข้อมูลต้องตรวจ owner/context ก่อน

### 16.2 การลดความเสี่ยง

- ใช้ข้อมูลสมมติใน GitHub และไฟล์ตัวอย่าง
- แยก production database กับ demo database
- จำกัดสิทธิ์บัญชี Demo
- สำรองฐานข้อมูลก่อน migration
- ตรวจสอบช่องโหว่ SQL injection ทุกจุด
- ใช้ prepared statements ให้ครบทุก query สำคัญ

## 19. แนวทางการพัฒนาที่แนะนำ

ควรพัฒนาแบบค่อยเป็นค่อยไป ไม่ควรแก้ทุกอย่างพร้อมกัน เพราะระบบเกี่ยวข้องกับฐานข้อมูล การเข้าสู่ระบบ และคะแนนของนักเรียน

ลำดับที่แนะนำคือ

```text
สำรองฐานข้อมูล
    ↓
เพิ่มโครงสร้าง schools/classrooms/sessions
    ↓
migrate ข้อมูลเดิมเข้าโรงเรียนเริ่มต้น
    ↓
ปรับ login/session context
    ↓
ปรับ dashboard/API ให้กรองข้อมูล
    ↓
เพิ่มระบบสมัครครูและอนุมัติ
    ↓
เพิ่มระบบห้องเรียนและนักเรียน
    ↓
ปรับคะแนน/log/ชิ้นงานให้ผูกกับ session
    ↓
เพิ่ม export และ demo
    ↓
ทดสอบหลายโรงเรียนพร้อมกัน
```

## 21. สรุป

การพัฒนาระบบให้รองรับ Multi-School เป็นการยกระดับจากสื่อที่ใช้ในห้องเรียนของผู้สร้าง ไปสู่สื่อดิจิทัลที่สามารถเผยแพร่และขยายผลสู่ครูหลายโรงเรียนได้อย่างเป็นระบบ

หัวใจสำคัญของการปรับปรุงครั้งนี้ไม่ใช่เพียงการเพิ่มหน้าสมัครสมาชิก แต่คือการออกแบบระบบใหม่ให้ข้อมูลทุกส่วนแยกตามโรงเรียน ครู ห้องเรียน และรอบการใช้งานอย่างถูกต้อง เพื่อป้องกันข้อมูลปะปน ลดความเสี่ยงด้านข้อมูลส่วนบุคคล และทำให้ระบบพร้อมใช้เป็นหลักฐานเชิงประจักษ์ในการพัฒนาสื่อและการจัดการเรียนรู้

เมื่อดำเนินการตามแผนนี้ ระบบจะมีความพร้อมมากขึ้นทั้งในด้านการใช้งานจริง การเผยแพร่ผ่าน OBEC Content Center การเก็บข้อมูลผลการใช้สื่อ และการต่อยอดเป็นผลงาน Best Practice ประเภทผู้สร้างสื่อเทคโนโลยีดิจิทัล
