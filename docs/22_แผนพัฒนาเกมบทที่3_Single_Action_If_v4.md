## 1. หลักการสำคัญที่ต้องแก้ไข

ในเกมบทที่ 3 โครงสร้างการเรียนรู้ถูกแบ่งเป็น 3 เกมหลัก ได้แก่

| เกม | ตรรกะหลัก | ความหมายเชิงการสอน |
|---|---|---|
| เกมที่ 1 | If | ถ้าเข้าเงื่อนไข ให้ทำคำสั่งพิเศษ ถ้าไม่เข้าเงื่อนไข ปล่อยผ่าน |
| เกมที่ 2 | If / Else | ถ้าเข้าเงื่อนไข ไปทางหนึ่ง นอกเหนือจากนี้ ไปอีกทางหนึ่ง |
| เกมที่ 3 | If / Else If / Else | ตรวจหลายเงื่อนไขตามลำดับ แล้วส่งไปหลายปลายทาง |

ปัญหาที่ต้องแก้คือ เกมที่ 1 ซึ่งควรสอน `If` อย่างบริสุทธิ์ ไม่ควรถูกออกแบบให้มีปลายทาง 2 ทางเทียบเท่า `If / Else` เพราะจะทำให้ผู้เรียนเข้าใจผิดว่า `If` ต้องมี “ทางเลือกอีกทาง” เสมอ ทั้งที่จริงแล้ว `If` คือการทำงานเฉพาะเมื่อเงื่อนไขเป็นจริงเท่านั้น

ดังนั้น เกมที่ 1 ต้องปรับเป็นแนวคิดนี้

```text
ถ้า [เข้าเงื่อนไข] → [ทำคำสั่งพิเศษ]
ถ้าไม่เข้าเงื่อนไข → ปล่อยผ่านอัตโนมัติ
```

ตัวอย่าง

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้างน้ำ]
```

แครอทที่ไม่เปื้อนโคลน เช่น แครอทสะอาด แครอทมีตำหนิ แครอทรูปร่างแปลก หรือแครอทหัวเล็ก ถือว่า “ไม่เข้าเงื่อนไข” และควรปล่อยผ่านไปเองโดยไม่ต้องมีบล็อก `Else`

---

## 2. เป้าหมายของการปรับปรุง

การปรับปรุงครั้งนี้มีเป้าหมายดังนี้

1. ทำให้เกมที่ 1 สอน `If` ได้ถูกต้องตามหลักการ
2. แยกความต่างระหว่าง `If`, `If / Else`, และ `If / Else If / Else` ให้ชัดเจน
3. ลดความสับสนของผู้เรียนเรื่องปลายทางของวัตถุ
4. ปรับ engine ให้รองรับ `logic_type = if` แบบ Single-Action If
5. ปรับระบบสร้างชิ้นงานให้ไม่บังคับเด็กสร้าง Else ในเกม If
6. ปรับ scoring ให้ถือว่า “ปล่อยผ่านอัตโนมัติ” เป็นคำตอบที่ถูกสำหรับวัตถุไม่เข้าเงื่อนไข
7. ปรับ UI ให้แสดงว่ากรณีไม่เข้าเงื่อนไขเป็นพฤติกรรมของระบบ ไม่ใช่คำสั่งที่เด็กต้องสร้าง
8. รองรับตัวหลอกในเกม If ได้อย่างถูกต้อง โดยตัวหลอกต้องเป็นวัตถุที่ “ไม่ควรถูกส่งเข้าเครื่องพิเศษ”
9. ปรับระบบ Debugger ของบทที่ 4 ให้เข้าใจว่า If ไม่มี Else และ pass-through คือพฤติกรรมปกติ

## 3. นิยามใหม่ของเกมทั้ง 3 ในบทที่ 3

### 3.1 เกมที่ 1: โรงคัดผักสวนครัว

**ตรรกะหลัก:** If  
**รูปแบบ:** Single-Action If  
**จำนวนปลายทางที่เด็กควบคุม:** 1 ปลายทางพิเศษ  
**พฤติกรรมวัตถุไม่เข้าเงื่อนไข:** ปล่อยผ่านอัตโนมัติ

ตัวอย่าง

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้างน้ำ]
ไม่เข้าเงื่อนไข → ปล่อยผ่านอัตโนมัติ
```

เกมนี้ต้องทำให้เด็กเข้าใจว่า `If` ใช้สำหรับตรวจจับ “กรณีพิเศษ” และสั่งให้ระบบทำบางอย่างเฉพาะกรณีนั้น

---

### 3.2 เกมที่ 2: โรงคัดผลไม้แสนอร่อย

**ตรรกะหลัก:** If / Else  
**รูปแบบ:** Two-Way Routing  
**จำนวนปลายทางที่เด็กควบคุม:** 2 ปลายทาง

ตัวอย่าง

```text
ถ้า [ส้มลูกใหญ่] → [กล่องพรีเมียม]
นอกเหนือจากนี้ → [กล่องแปรรูป]
```

เกมนี้ต้องทำให้เด็กเข้าใจว่า `Else` คือ “กลุ่มที่เหลือทั้งหมด” และต้องมีปลายทางของกลุ่มที่เหลืออย่างชัดเจน

---

### 3.3 เกมที่ 3: โรงคัดผลผลิตจากฟาร์มสัตว์

**ตรรกะหลัก:** If / Else If / Else  
**รูปแบบ:** Multi-Way Routing  
**จำนวนปลายทางที่เด็กควบคุม:** 3 ปลายทางขึ้นไป

ตัวอย่าง

```text
ถ้า [ไข่ใบใหญ่และไม่ร้าว] → [ถาดพรีเมียม]
นอกเหนือจากนี้ถ้า [ไข่ใบเล็กและไม่ร้าว] → [ถาดมาตรฐาน]
นอกเหนือจากนี้ → [ถังคัดทิ้ง]
```

เกมนี้ต้องทำให้เด็กเข้าใจว่าระบบจะตรวจเงื่อนไขจากบนลงล่าง และ `Else` คือกรณีที่ไม่เข้าเงื่อนไขใดเลย

---

## 4. รายละเอียดการปรับเกมที่ 1 ให้เป็น Single-Action If

### 4.1 โครงสร้างการเล่นใหม่

เกมที่ 1 ต้องมีองค์ประกอบดังนี้

1. สายพานลำเลียงวัตถุ
2. จุดสแกนเงื่อนไข
3. เครื่องจักรหรือปลายทางพิเศษเพียง 1 จุด
4. ทางผ่านอัตโนมัติสำหรับวัตถุที่ไม่เข้าเงื่อนไข
5. บล็อกกฎเพียง 1 แถว คือ `ถ้า...แล้ว...`
6. ไม่มีบล็อก `Else`
7. ไม่มีช่องคำสั่งสำหรับวัตถุที่ไม่เข้าเงื่อนไข

ภาพแนวคิด

```text
วัตถุบนสายพาน
      ↓
จุดสแกน
      ↓
เข้าเงื่อนไขหรือไม่?
      ├─ ใช่ → ส่งเข้าเครื่องพิเศษ
      └─ ไม่ใช่ → ปล่อยผ่านอัตโนมัติ
```

---

### 4.2 ด่าน 1-1: แครอทเปื้อนโคลน

**โจทย์:** คัดแครอทเปื้อนโคลนไปล้างน้ำ  
**กฎที่ผู้เรียนต้องสร้าง:**

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้างน้ำ]
```

**พฤติกรรมอัตโนมัติ:**

```text
แครอทอื่น ๆ → ปล่อยผ่านอัตโนมัติ
```

**วัตถุที่ควรมี:**

| วัตถุ | เข้าเงื่อนไขหรือไม่ | ผลลัพธ์ที่ถูกต้อง |
|---|---|---|
| แครอทเปื้อนโคลน | ใช่ | ส่งเข้าเครื่องล้างน้ำ |
| แครอทสะอาด | ไม่ใช่ | ปล่อยผ่าน |
| แครอทมีตำหนิ | ไม่ใช่ | ปล่อยผ่าน |
| แครอทหัวเล็ก | ไม่ใช่ | ปล่อยผ่าน |
| แครอทรูปร่างแปลก | ไม่ใช่ | ปล่อยผ่าน |

**ข้อผิดพลาดที่ควรเกิดได้:**

| สถานการณ์ | ถือว่าผิดเพราะ |
|---|---|
| แครอทเปื้อนโคลนถูกปล่อยผ่าน | เข้าเงื่อนไขแต่ไม่ได้ทำคำสั่งพิเศษ |
| แครอทสะอาดถูกส่งเข้าเครื่องล้าง | ไม่เข้าเงื่อนไขแต่ถูกทำคำสั่งพิเศษ |
| แครอทมีตำหนิถูกส่งเข้าเครื่องล้าง | ตำหนิไม่ใช่โคลน จึงไม่เข้าเงื่อนไข |
| แครอทหัวเล็กถูกส่งเข้าเครื่องล้าง | ขนาดไม่เกี่ยวกับเงื่อนไขโคลน |

---

### 4.3 ด่าน 1-2: ผักกาดมีหนอน

**โจทย์:** ส่งผักกาดที่มีหนอนไปโต๊ะตรวจศัตรูพืช  
**กฎที่ผู้เรียนต้องสร้าง:**

```text
ถ้า [ผักกาดมีหนอน] → [ส่งไปโต๊ะตรวจศัตรูพืช]
```

**พฤติกรรมอัตโนมัติ:**

```text
ผักกาดอื่น ๆ → ปล่อยผ่านอัตโนมัติ
```

**วัตถุที่ควรมี:**

| วัตถุ | เข้าเงื่อนไขหรือไม่ | ผลลัพธ์ที่ถูกต้อง |
|---|---|---|
| ผักกาดมีหนอน | ใช่ | ส่งไปโต๊ะตรวจศัตรูพืช |
| ผักกาดปกติ | ไม่ใช่ | ปล่อยผ่าน |
| ผักกาดมีรอยดิน | ไม่ใช่ | ปล่อยผ่าน |
| ผักกาดใบแหว่งแต่ไม่มีหนอน | ไม่ใช่ | ปล่อยผ่าน |
| ผักกาดสีซีดแต่ไม่มีหนอน | ไม่ใช่ | ปล่อยผ่าน |

**จุดเน้น:** เด็กต้องแยกให้ได้ว่า “ใบแหว่ง” หรือ “มีรอยดิน” ไม่ใช่เงื่อนไขเดียวกับ “มีหนอน”

---

### 4.4 ด่าน 1-3: มันฝรั่งมีหน่องอก

**โจทย์:** ส่งมันฝรั่งที่มีหน่องอกไปคัดแยกพิเศษ  
**กฎที่ผู้เรียนต้องสร้าง:**

```text
ถ้า [มันฝรั่งมีหน่องอก] → [ส่งไปคัดแยกพิเศษ]
```

**พฤติกรรมอัตโนมัติ:**

```text
มันฝรั่งอื่น ๆ → ปล่อยผ่านอัตโนมัติ
```

**วัตถุที่ควรมี:**

| วัตถุ | เข้าเงื่อนไขหรือไม่ | ผลลัพธ์ที่ถูกต้อง |
|---|---|---|
| มันฝรั่งมีหน่องอก | ใช่ | ส่งไปคัดแยกพิเศษ |
| มันฝรั่งปกติ | ไม่ใช่ | ปล่อยผ่าน |
| มันฝรั่งเปื้อนดิน | ไม่ใช่ | ปล่อยผ่าน |
| มันฝรั่งลูกเล็ก | ไม่ใช่ | ปล่อยผ่าน |
| มันฝรั่งสีคล้ำแต่ไม่มีหน่อ | ไม่ใช่ | ปล่อยผ่าน |

**จุดเน้น:** เพิ่มความยากด้วยตัวหลอกที่ดูคล้าย แต่ไม่เข้าเงื่อนไข

## 5. ผลกระทบต่อ UI ของเกมที่ 1

### 5.1 แผงสร้างโปรแกรม

UI ของเกม If ต้องแสดงเพียง 1 แถว

```text
ถ้า [__________] → [__________]
```

ใต้แถวกฎควรมีข้อความระบบ เช่น

```text
วัตถุที่ไม่เข้าเงื่อนไขจะปล่อยผ่านอัตโนมัติ
```

หรือ

```text
ไม่ต้องสร้าง Else ในด่านนี้ ระบบจะปล่อยวัตถุอื่นผ่านเอง
```

---

### 5.2 ไม่ควรแสดง Else

ห้ามแสดง UI แบบนี้ในเกม If

```text
ถ้า [__________] → [__________]
นอกเหนือจากนี้ → [__________]
```

เพราะอันหลังคือ If / Else แล้ว

---

### 5.3 พื้นที่ปลายทางในฉาก

ในฉากเกม If ควรมีเพียง

1. เครื่องพิเศษ เช่น เครื่องล้างน้ำ โต๊ะตรวจศัตรูพืช หรือจุดคัดแยกพิเศษ
2. สายพานหลักที่ปล่อยวัตถุไม่เข้าเงื่อนไขผ่านไปเอง

ไม่ควรทำให้ตะกร้าปกติดูเป็น “ปลายทางที่ต้องเลือก” เทียบเท่าเครื่องพิเศษ

ตัวอย่างการจัดฉาก

```text
สายพานหลัก ────────→ ปล่อยผ่านอัตโนมัติ

                 ↘
                  เครื่องล้างน้ำ
```

---

### 5.4 ป้ายอธิบายบนฉาก

ควรมีป้ายสั้น ๆ เช่น

```text
จับเฉพาะแครอทเปื้อนโคลนไปล้าง
อย่างอื่นปล่อยผ่าน
```

หรือ

```text
เป้าหมาย: ส่งเฉพาะวัตถุที่เข้าเงื่อนไขไปเครื่องพิเศษ
```

## 6. ผลกระทบต่อ Engine

### 6.1 เพิ่ม logicType แบบ Single-Action If

เพิ่มหรือแก้ config ของเกม If ให้ชัดเจน

```javascript
{
  logicType: 'if',
  mode: 'single_action_if',
  ruleSlots: [
    { type: 'if', condition: null, action: null }
  ],
  defaultBehavior: {
    type: 'pass_through',
    label: 'ปล่อยผ่านอัตโนมัติ'
  }
}
```

---

### 6.2 การประเมินผลในเกม If

สำหรับ `logicType = if` ต้องประเมินแบบนี้

```javascript
function evaluateIfOnly(item, rule, config) {
  const matched = checkCondition(item, rule.condition);

  if (matched) {
    return rule.action;
  }

  return config.defaultBehavior?.type || 'pass_through';
}
```

หรือใน engine กลาง

```javascript
function evaluateRules(item, rules, config) {
  if (config.logicType === 'if') {
    const rule = rules.find(r => r.type === 'if');
    if (!rule) return 'pass_through';

    return checkCondition(item, rule.condition)
      ? rule.action
      : config.defaultBehavior?.type || 'pass_through';
  }

  for (const rule of rules) {
    if (rule.type === 'else') return rule.action;
    if (checkCondition(item, rule.condition)) return rule.action;
  }

  return 'pass_through';
}
```

---

### 6.3 Action Handler ต้องรองรับ pass_through

เพิ่ม action พิเศษใน engine

```javascript
function applyAction(item, actionId, config) {
  if (actionId === 'pass_through') {
    return passThrough(item);
  }

  return routeToDestination(item, actionId, config);
}
```

`pass_through` ต้องเป็น internal action ของระบบ ไม่ใช่ action block ที่เด็กลาก

---

### 6.4 การประเมินถูก/ผิด

ในเกม If ต้องใช้หลักนี้

```javascript
function isCorrectResult(item, actualAction, config) {
  const matched = checkCondition(item, config.ifRule.condition);

  if (matched) {
    return actualAction === config.ifRule.action;
  }

  return actualAction === 'pass_through';
}
```

หรือใช้ข้อมูลรายวัตถุ

```javascript
{
  id: 'clean_carrot',
  matchesCondition: false,
  correctResult: 'pass_through'
}
```

แล้วตรวจว่า actual action ตรงกับ `correctResult` หรือไม่

---

## 7. ผลกระทบต่อระบบคะแนนและดาว

### 7.1 หลักการให้คะแนนในเกม If

เกม If ต้องถือว่า “ปล่อยผ่านอัตโนมัติ” เป็นผลลัพธ์ที่ถูกต้องสำหรับวัตถุไม่เข้าเงื่อนไข

| วัตถุ | การกระทำของระบบ | ผลลัพธ์ |
|---|---|---|
| เข้าเงื่อนไข → ส่งเข้าเครื่องพิเศษ | ถูก |
| เข้าเงื่อนไข → ปล่อยผ่าน | ผิด |
| ไม่เข้าเงื่อนไข → ปล่อยผ่าน | ถูก |
| ไม่เข้าเงื่อนไข → ส่งเข้าเครื่องพิเศษ | ผิด |

---

### 7.2 เกณฑ์ดาวที่เสนอ

ใช้เกณฑ์เดียวกับบทที่ 3 โดยปรับให้เหมาะกับ If

```text
3 ดาว = คัดถูก 90% ขึ้นไป และส่งวัตถุไม่เข้าเงื่อนไขผิดไม่เกิน 1 ชิ้น
2 ดาว = คัดถูก 75% ขึ้นไป
1 ดาว = คัดถูก 60% ขึ้นไป
0 ดาว = ต่ำกว่า 60% หรือกฎไม่ครบ
```

ข้อผิดพลาดที่ควรนับเป็น damage:

1. วัตถุเข้าเงื่อนไขแต่หลุดผ่าน
2. วัตถุไม่เข้าเงื่อนไขแต่ถูกส่งเข้าเครื่องพิเศษ

## 8. ผลกระทบต่อ Preview Bar

### 8.1 Preview Bar ต้องแสดงว่าอะไรเข้าเงื่อนไข

เมื่อคลิกวัตถุในแถบรายการด้านล่าง ระบบควรแสดงรายละเอียด เช่น

```text
แครอทมีตำหนิ

คุณสมบัติ:
- เป็นแครอท
- มีตำหนิ
- ไม่เปื้อนโคลน

ผลลัพธ์ที่ถูกต้อง:
ปล่อยผ่านอัตโนมัติ

เหตุผล:
โจทย์ตรวจเฉพาะแครอทเปื้อนโคลน ตำหนิไม่ใช่เงื่อนไขของด่านนี้
```

---

### 8.2 ก่อนเล่นไม่ควรเฉลยมากเกินไป

ก่อนเล่นควรแสดงแบบไม่เฉลยทั้งหมด เช่น

```text
วัตถุนี้มีลักษณะพิเศษ แต่ต้องดูว่าใช่เงื่อนไขของด่านหรือไม่
```

หลังเล่นหรือในโหมดครูจึงแสดงเฉลยเต็มว่า

```text
ไม่เข้าเงื่อนไข เพราะไม่ได้เปื้อนโคลน
```

---

### 8.3 ตัวหลอกในเกม If

ตัวหลอกควรเป็นวัตถุที่ดูเหมือนน่าส่งเข้าเครื่องพิเศษ แต่จริง ๆ ไม่เข้าเงื่อนไข เช่น

| ด่าน | เงื่อนไขจริง | ตัวหลอก |
|---|---|---|
| แครอทเปื้อนโคลน | เปื้อนโคลน | แครอทมีตำหนิ, แครอทหัวเล็ก |
| ผักกาดมีหนอน | มีหนอน | ผักกาดใบแหว่งแต่ไม่มีหนอน |
| มันฝรั่งมีหน่อ | มีหน่องอก | มันฝรั่งเปื้อนดิน, สีคล้ำแต่ไม่มีหน่อ |

## 9. ผลกระทบต่อระบบสร้างชิ้นงานบทที่ 3

ระบบสร้างชิ้นงานต้องปรับตาม logic type โดยเฉพาะ `If`

---

### 9.1 ถ้าเด็กเลือกสร้างเกม If

Builder ต้องแสดงโครงสร้างนี้

```text
ถ้า [เงื่อนไขพิเศษ] → [คำสั่งพิเศษ]

วัตถุที่ไม่เข้าเงื่อนไข: ปล่อยผ่านอัตโนมัติ
```

เด็กต้องกำหนด

1. ชื่อด่าน
2. เงื่อนไขพิเศษ 1 เงื่อนไข
3. คำสั่งพิเศษ 1 คำสั่ง
4. เครื่องจักรหรือปลายทางพิเศษ 1 จุด
5. วัตถุที่เข้าเงื่อนไขอย่างน้อย 1 ชิ้น
6. วัตถุที่ไม่เข้าเงื่อนไขอย่างน้อย 2 ชิ้น
7. ตัวหลอกอย่างน้อย 1 ชิ้น
8. ข้อความ Feedback เมื่อวัตถุผิดพลาด

ไม่ต้องให้เด็กสร้าง

- Else
- ปลายทางปกติ
- คำสั่งสำหรับวัตถุไม่เข้าเงื่อนไข

---

### 9.2 Validation สำหรับชิ้นงาน If

ก่อนทดสอบหรือส่งงาน ต้องตรวจว่า

| เงื่อนไขตรวจสอบ | ต้องผ่านหรือไม่ |
|---|---|
| มีเงื่อนไขพิเศษ 1 เงื่อนไข | ต้องมี |
| มีคำสั่งพิเศษ 1 คำสั่ง | ต้องมี |
| มีวัตถุเข้าเงื่อนไขอย่างน้อย 1 ชิ้น | ต้องมี |
| มีวัตถุไม่เข้าเงื่อนไขอย่างน้อย 2 ชิ้น | ต้องมี |
| มีตัวหลอกอย่างน้อย 1 ชิ้น | ต้องมี |
| ไม่มี Else block | ต้องไม่มี |
| ไม่มีปลายทางปกติที่ต้องเลือก | ต้องไม่มี |
| defaultBehavior เป็น pass_through | ต้องมี |

ข้อความเตือนตัวอย่าง

```text
เกม If ไม่ต้องมี Else ระบบจะปล่อยวัตถุที่ไม่เข้าเงื่อนไขผ่านอัตโนมัติ
```

```text
กรุณาเพิ่มตัวหลอกอย่างน้อย 1 ชิ้น เช่น วัตถุที่ดูคล้ายแต่ไม่เข้าเงื่อนไข
```

```text
ยังไม่มีวัตถุที่เข้าเงื่อนไขพิเศษ กรุณาเพิ่มวัตถุอย่างน้อย 1 ชิ้น
```

---

### 9.3 UI Builder สำหรับเกม If

หน้าสร้างด่าน If ควรแสดงเป็นขั้นตอนดังนี้

#### ขั้นที่ 1: ตั้งโจทย์

```text
ชื่อด่าน: คัดแครอทเปื้อนโคลน
ภารกิจ: ส่งเฉพาะแครอทเปื้อนโคลนเข้าเครื่องล้าง
```

#### ขั้นที่ 2: เลือกเงื่อนไขพิเศษ

```text
เงื่อนไขพิเศษ:
[แครอทเปื้อนโคลน]
```

#### ขั้นที่ 3: เลือกคำสั่งพิเศษ

```text
คำสั่งพิเศษ:
[ส่งเข้าเครื่องล้างน้ำ]
```

#### ขั้นที่ 4: เลือกวัตถุบนสายพาน

```text
[✓] แครอทเปื้อนโคลน  เข้าเงื่อนไข
[✓] แครอทสะอาด        ไม่เข้าเงื่อนไข
[✓] แครอทมีตำหนิ      ตัวหลอก / ไม่เข้าเงื่อนไข
[✓] แครอทหัวเล็ก      ตัวหลอก / ไม่เข้าเงื่อนไข
```

#### ขั้นที่ 5: ทดลองเล่น

```text
[ทดลองเล่นด่านของฉัน]
```

## 10. ผลกระทบต่อระบบสร้างโจทย์บั๊กของบทที่ 4

เนื่องจากบทที่ 4 จะต่อยอดจากบทที่ 3 ระบบ Debugger ต้องเข้าใจความต่างของเกม If ด้วยเช่นกัน

### 10.1 บั๊กที่เกี่ยวข้องกับเกม If

ในเกม If บั๊กที่เหมาะสม ได้แก่

1. เงื่อนไขผิด
2. คำสั่งผิด
3. เงื่อนไขไม่ชัดเจน
4. ส่งวัตถุไม่เข้าเงื่อนไขเข้าเครื่องพิเศษ
5. ปล่อยวัตถุเข้าเงื่อนไขผ่านไป

### 10.2 ตัวอย่างบั๊กเกม If

กฎถูก

```text
ถ้า [แครอทเปื้อนโคลน] → [ส่งเข้าเครื่องล้าง]
ไม่เข้าเงื่อนไข → ปล่อยผ่าน
```

กฎมีบั๊ก

```text
ถ้า [แครอทมีตำหนิ] → [ส่งเข้าเครื่องล้าง]
ไม่เข้าเงื่อนไข → ปล่อยผ่าน
```

อาการ

```text
แครอทมีตำหนิถูกส่งเข้าเครื่องล้าง แต่แครอทเปื้อนโคลนหลุดผ่าน
```

วิธีแก้

```text
เปลี่ยนเงื่อนไขเป็น “แครอทเปื้อนโคลน”
```

ระบบสร้างโจทย์บั๊กต้องไม่บังคับให้มี Else ในโจทย์ If

## 11. การปรับชื่อและคำอธิบายในเอกสาร/หน้าเกม

```text
เกมที่ 1: โรงคัดผักสวนครัว
```

```text
เลือกเงื่อนไขพิเศษและคำสั่งพิเศษ วัตถุอื่นที่ไม่เข้าเงื่อนไขจะปล่อยผ่านอัตโนมัติ
```

### 11.2 หน้า Instruction

ควรเพิ่มคำอธิบายความต่างระหว่างเกม 1, 2, 3

```text
เกมที่ 1 ใช้ If: ถ้าเจอวัตถุที่เข้าเงื่อนไข ให้ส่งไปเครื่องพิเศษ วัตถุอื่นปล่อยผ่าน
```

```text
เกม If มีปลายทางพิเศษเพียง 1 ทางเท่านั้น ไม่ต้องสร้าง Else
```

```text
เกม If / Else ต้องมี 2 ทาง คือทางเข้าเงื่อนไข และทางนอกเหนือจากนี้
```

ถ้าเลือกเกม If / Else If / Else ให้ขึ้นคำแนะนำ

```text
เกม If / Else If / Else ใช้คัดหลายระดับ โดยระบบจะตรวจเงื่อนไขจากบนลงล่าง
```

## 12. แผนการพัฒนาเป็นระยะ

### ระยะที่ 1: ปรับ Config ของเกมที่ 1

งานที่ต้องทำ

1. ปรับ `stage7.js` ให้ใช้ `logicType: 'if'`
2. เพิ่ม `mode: 'single_action_if'`
3. ลบ `else` slot ออกจากเกมที่ 1
4. เพิ่ม `defaultBehavior: pass_through`
5. ปรับวัตถุทุกชิ้นให้มี `matchesCondition` และ `correctResult`
6. เพิ่มตัวหลอกที่ไม่เข้าเงื่อนไขแต่ดูคล้าย

ผลลัพธ์

- เกมที่ 1 ใช้ If อย่างถูกต้อง
- ไม่มีปลายทาง Else ในเกมที่ 1

---

### ระยะที่ 2: ปรับ Engine

งานที่ต้องทำ

1. เพิ่ม logic ประเมิน If-only
2. เพิ่ม internal action `pass_through`
3. แยกการประเมิน `if`, `if_else`, `if_else_if_else`
4. ปรับ scoring ให้รองรับวัตถุไม่เข้าเงื่อนไข
5. ปรับ feedback สำหรับกรณี pass-through

ผลลัพธ์

- Engine เข้าใจความต่างของ If กับ If / Else

---

### ระยะที่ 3: ปรับ UI เกมที่ 1

งานที่ต้องทำ

1. แสดง rule panel เพียง 1 แถว
2. เพิ่มข้อความ “วัตถุอื่นปล่อยผ่านอัตโนมัติ”
3. ซ่อน action slot ของ Else
4. ปรับฉากให้มีเครื่องพิเศษ 1 จุด
5. ปรับ preview bar ให้แสดงรายละเอียดว่าเข้าเงื่อนไขหรือไม่

ผลลัพธ์

- UI ชัดเจนว่าเกมนี้เป็น If ไม่ใช่ If / Else

---

### ระยะที่ 4: ปรับระบบสร้างชิ้นงาน

งานที่ต้องทำ

1. ถ้าเลือก logic_type = if ให้ใช้ Single-Action Builder
2. ไม่ให้สร้าง Else
3. ไม่ให้สร้างปลายทางปกติ
4. บังคับมี defaultBehavior = pass_through
5. บังคับมีตัวหลอกอย่างน้อย 1 ชิ้น
6. บันทึก work_data ตาม schema ใหม่
7. ทดสอบ preview game จาก work_data

ผลลัพธ์

- นักเรียนสร้างชิ้นงาน If ได้ถูกหลัก

---

### ระยะที่ 5: ปรับบทที่ 4 Debugger ให้รองรับ If-only

งานที่ต้องทำ

1. เพิ่ม bug config สำหรับ If-only
2. ไม่บังคับ Else ในโจทย์บั๊ก If
3. เพิ่มประเภทบั๊ก เช่น ส่งวัตถุไม่เข้าเงื่อนไขเข้าเครื่องพิเศษ
4. เพิ่ม feedback สำหรับกรณี pass-through

ผลลัพธ์

- บทที่ 4 ต่อกับบทที่ 3 ได้ถูกต้อง

---

### ระยะที่ 6: ทดสอบผู้ใช้

งานที่ต้องทำ

1. ให้เด็กทดลองเล่นเกมที่ 1
2. ตรวจว่าเด็กเข้าใจว่า “ไม่เข้าเงื่อนไข = ปล่อยผ่าน”
3. ให้เด็กสร้างชิ้นงาน If

## 16. สรุป

การปรับปรุงนี้เป็นจุดสำคัญของบทที่ 3 เพราะช่วยให้การสอน `If` ถูกต้องและไม่ปะปนกับ `If / Else`

หลักที่ต้องล็อกคือ

> เกมที่ 1 สอน If จึงมีคำสั่งพิเศษเพียง 1 คำสั่ง วัตถุที่ไม่เข้าเงื่อนไขต้องปล่อยผ่านอัตโนมัติ

เมื่อปรับตามแผนนี้ เด็กจะเห็นความแตกต่างชัดเจนว่า

- `If` คือ เจอเงื่อนไขแล้วทำงาน ไม่เจอก็ปล่อยผ่าน
- `If / Else` คือ มีสองทางเลือก
- `If / Else If / Else` คือ มีหลายทางเลือกตามลำดับเงื่อนไข

สิ่งนี้จะทำให้บทที่ 3 แข็งแรงขึ้นทั้งในเชิงเกม เชิงการสอน และระบบสร้างชิ้นงาน
